## Resumo

A tarefa F0.6 pede ligar a fronteira `the-light-app-core` ao alvo web/WASM, com a
condição inegociável de **configurar apenas o consumo do core** (features /
`default-features=false` em `core/Cargo.toml`), sem modificar/forkar/copiar o
`the-light-core`. O **portão decisivo** é o sub-passo 2: compilar a fronteira para
`wasm32-unknown-unknown`.

Verifiquei de forma independente e **confirmo o bloqueio reportado pelo Executor**.
O `the-light-core` (rev `0888ac0`) **não possui seção `[features]`**, e `rusqlite`
(SQLite-C *bundled*) e `reqwest` são dependências **incondicionais** (sem
`optional`, sem `[target.'cfg(...)']`). Como a fronteira arrasta o crate inteiro
`the-light-core`, o cargo compila `rusqlite`, que no alvo wasm resolve para
`sqlite-wasm-rs v0.5.5`; o build falha (exit 101) ao tentar compilar o SQLite-C via
`clang --target=wasm32-unknown-unknown` (o clang do sistema não tem backend wasm).

Por não haver `[features]` nem `optional` no core, **nenhuma configuração de
consumo** em `core/Cargo.toml` (nem `default-features=false`, nem seleção de
features) consegue excluir o `rusqlite`. A única saída seria gating/split no
**próprio core** = PR no `the-light` + ADR = **decisão humana**. Logo, este é o
**gate estratégico de Fase 0 (Apêndice A #1 / VISION §4 fricção #1)** e o desfecho
correto é **`blocked`**, não `failed` — não é bug do nosso código; é barreira
arquitetural externa. O caminho `passed` não foi atingido (parou no portão
decisivo, antes dos sub-passos 3–8).

## Decisões / ADRs

**NENHUMA** decisão de store/transporte no web foi tomada nesta tarefa — e
corretamente: a tarefa proíbe inventar escopo e reserva a decisão para o HALT/Guia.
**Não há ADR-0005** (só seria escrito se uma decisão de arquitetura web fosse
efetivamente tomada e implementada). Candidatos de estratégia a serem decididos
pelo humano/Guia (fora desta tarefa):

1. **PR ao `the-light-core`** adicionando uma feature de exclusão do store (gate de
   `rusqlite`/`reqwest`, ou split do crate em `the-light-core-reference` puro), de
   modo que a fronteira possa consumir só o módulo `reference` no wasm. Exige PR +
   ADR no `the-light`.
2. **`wa-sqlite` + OPFS** com store injetado no web (SQLite compilado para wasm por
   caminho próprio, fora do `rusqlite`-bundled).
3. **Web como "leitura + IA"** com dados pré-indexados (sem SQLite embarcado no
   cliente), tratando o web como superfície reduzida.

## Bloqueios / perguntas

**BLOQUEIO (gate estratégico de Fase 0 — Apêndice A #1 / VISION §4 fricção #1):**
A fronteira `the-light-app-core` **não compila para `wasm32-unknown-unknown`**
porque arrasta o crate inteiro `the-light-core`, cuja dependência **incondicional**
`rusqlite` (SQLite-C *bundled*) puxa `sqlite-wasm-rs v0.5.5`, cujo build via
`clang --target=wasm32-unknown-unknown` falha ("No available targets are compatible
with triple wasm32-unknown-unknown"). Como o core **não** é feature-gateável sem ser
modificado, **não há saída pelo consumo** — a única solução cruza a fronteira do
`the-light`, que é PR + ADR = decisão humana.

**Pergunta de decisão estratégica para o humano/Guia:** qual estratégia de
store/transporte para o alvo web? (a) abrir PR ao core para excluir o store no wasm
(feature/split); (b) `wa-sqlite`+OPFS com store injetado; ou (c) web como
leitura+IA pré-indexada. Cada caminho define como — e se — o web carrega o store.

**Retomar exige decisão humana, não retry.** Este resultado é `blocked` (não
`failed`): o loop não deve gerar `<ID>-fixN` nem reexecutar — um retry recompilaria
o mesmo SQLite-C e falharia identicamente. O Driver deve escrever `loop/HALT` para
o Guia/humano escolher a estratégia (e, se for o caminho do PR ao core, abrir
PR + ADR no `the-light`). Só então o `HALT` é removido e o loop prossegue.
