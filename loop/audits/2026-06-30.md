## A decisão pendente (F0.10 — store web)

O store/source do `the-light-core` são **concretos sobre `rusqlite::Connection`**
(sem trait injetável); no web `rusqlite` está fora do grafo por design, então hoje
`get_passage` web é stub de erro. Para o web ler uma passagem é preciso **escolher**
como ter SQLite no web. Opções (a registrar em ADR-0011):

- **A) `wa-sqlite` + OPFS, `get_passage` web em TS** — destrava já, sem PR ao core;
  `SELECT` de passagem reimplementado em TS (infra); parsing/RAG/citação seguem no
  Rust; texto vem do store local (anti-alucinação preservada). *Recomendada p/ Fase 0.*
- **B) PR ao the-light abstraindo store sobre trait de conexão** — ideal da VISION
  ("store injetado"), porém refator substancial e ponte Rust-sync ↔ JS-async difícil.
- **C) Web como leitura pré-indexada (sem SQLite)** — mais simples, menos fiel ao
  data-model; não escala p/ Bíblia completa + FTS.

**Recomendação do auditor: Opção A** para a Fase 0 (mesma do HALT). Baixo atrito,
mantém anti-alucinação e "uma fonte da verdade" no domínio; B fica como refator
futuro quando o web precisar de **todo** o store em Rust.
