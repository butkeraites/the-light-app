## Resumo

UI de busca **nativa** entregue e provada de ponta a ponta no simulador iOS. Uma
tela (`app/app/search/index.tsx`, rota `search/index` registrada no `_layout.tsx`)
com `TextInput` + debounce 300 ms abre o banco bundled (`ensureReadingDb`) e chama
a **fronteira `search` da F1.5** (`app/web/reading.ts` → binding gerado → JSI →
`the_light_core::search`, FTS5/BM25), renderizando uma `FlatList` de
`ReaderSearchResultItem`. Cada item mostra a **referência** ("John 3:16") e o
**snippet com o termo destacado** — `app/lib/highlight.ts` faz o split dos
marcadores de controle `HL_START`(U+0002)/`HL_END`(U+0003) em runs `<Text>`
(matched/normal via tokens de tema), **consumindo** os controles (nunca exibidos).
O item é `Pressable` → `router.push('/read/[book]/[chapter]')` (rota F1.3), com
`verse` como param opcional (ancoragem = best-effort/follow-up).

Anti-fake confirmado pela leitura do código: **nenhum** `verses_fts`/`MATCH`/
`bm25(`/`highlight(`/`SELECT` FTS reimplementado em TS de produção (grep só achou
**comentários** que descrevem o que NÃO é reimplementado); o texto do hit é
verbatim do store; `first_text` do self-test vem do **retorno real** de `search`
(hit de João 3:16 localizado no resultado), limpo via `stripMarkers`. O glue web
(`reading.web.ts`, `search-selftest.web.ts`) é stub (lança/SKIP — busca web =
F1.14). `verses_fts` é populado no **gerador de dados/fixture**
(`gen_reading_sample_db.rs`, `INSERT...SELECT FROM verses`), não no core nem na
busca. `core/src/lib.rs` e `../the-light` **intactos** (pinado `8f66004`).
Re-execução independente da prova (com Metro órfão de :8081 morto antes): self-test
sai **0** e emite `TLA_SEARCH ... hits=646 first_ref="John 3:16"` + todos os
marcadores anteriores, sem regressão. `tsc`/`expo export`/`cargo test` verdes.

## Decisões / ADRs

- **Sem ADR novo.** Reusa **ADR-0014** (bundling do subset no app nativo) e
  **ADR-0015** (tema/tokens). O realce do termo usa `accent`/`verseText`
  existentes — nenhuma paleta nova.
- **`verses_fts` no gerador de dados:** `core/examples/gen_reading_sample_db.rs`
  popula `verses_fts(text, translation_id, verse_id)` via `INSERT...SELECT FROM
  verses` (`verse_id = verses.id`), com asserção `indexed == inserted`. É
  **fixture/pipeline de dados** (o schema do core não tem trigger; gotcha da F1.5),
  **não** reimplementação de busca — `MATCH`/BM25/`highlight` seguem no core,
  chamados pela fronteira `search`.
- **`limit=1000` no self-test:** uso **legítimo** do param opcional da F1.5. O BM25
  ranqueia Salmos acima de João 3:16 para "God" (top-20 default não o traz); o
  limite generoso garante o versículo conhecido no conjunto retornado, tornando a
  prova determinística sem hardcodear texto nem reordenar/snippetar à mão.
