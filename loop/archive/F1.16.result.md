## Resumo

A F1.16 dá **paridade web de NOTAS/MARCAÇÕES + EXPORT** — a ÚLTIMA paridade web da
Fase 1. As **7 funções** de userdata em `app/web/reading.web.ts`
(`putNote`/`getNote`/`deleteNote`/`listNotes` + `addHighlight`/`removeHighlight`/
`listHighlights`) **deixaram de ser stub** (`WEB_NOTES_MSG` removido) e agora
persistem em **OPFS** (`userdata-opfs.web.ts`) via um glue VFS-agnóstico
(`userdata-fs.web.ts`) que **espelha o formato em disco do core** (`the_light_core::
userdata`, rev `8f66004`): slug `notes/John_3.16.md`, `.md` só-corpo,
`highlights.json` array `{ref,color,tag?}` pretty/2-espaços. A referência é
canonicalizada pelo **wasm** (`parseReference` nas duas direções; nome EN via
`listBooks().nameEn`, não tabela hardcoded). O export reusa `buildNotesExport`
(agregado puro, idêntico ao nativo). Verifiquei de forma independente cada item:
**tudo verde, the-light e caminho nativo intactos → passed.**

Confronto direto contra a FONTE do core (rev `8f66004`,
`crates/the-light-core/src/userdata/{notes.rs,highlights.rs}` + `reference.rs`)
confirma que o TS é um espelho fiel, não uma reimplementação de domínio: `slug` =
`format_reference(_, En).replace(' ',"_").replace(':',".") + ".md"`; read-back por
`parse_reference(stem.replace('_'," "))`; filtro `extension == "md"`; `sort_by_key`
por `(book,chapter,start)`; `HighlightDto` com `#[serde(rename="ref")]` +
`skip_serializing_if = "Option::is_none"` no `tag` + `to_string_pretty`; `add` =
`retain(ref !=) ; push` (substitui por referência, ordem de inserção); `remove` =
retain + contagem; `list` = ordem do array.

## Decisões / ADRs

- **ADR-0022** (registrado neste commit) — userdata WEB reimplementado em TS sobre
  **OPFS** ESPELHANDO o formato em disco do core (`notes/<slug>.md` +
  `highlights.json`). É **infra de armazenamento, não lógica de domínio** —
  precedente **ADR-0011** (store/SQL do core espelhado em TS como infra), estendido
  a userdata (ADR-0019/0020/0021).
- **Referência via wasm** nas duas direções: `parseReference` resolve a referência
  de entrada ANTES do I/O e re-analisa o stem/`ref` na leitura; nome EN do livro vem
  de `listBooks().nameEn` (wasm), nunca tabela hardcoded → uma fonte da verdade do
  cânon. A única "format" em TS é a convenção de nome de arquivo/`ref` (nível de
  espelhar um `SELECT`).
- **Anti-alucinação preservada** (ADR-0017): não se aplica ao corpo da nota / tag
  (dado livre do usuário, igual ao nativo); a referência é canônica (wasm); nenhum
  texto bíblico do banco vaza pelo export.
- **Export = `buildNotesExport`** (agregado puro dos Records) — IDÊNTICO ao nativo;
  nenhuma serialização de domínio reescrita à mão.
- **Offline-first/BYOK:** OPFS local em `data_dir` web próprio
  (`the-light/userdata/`), separado do `reading-sample.sqlite` só-leitura; zero rede
  em runtime; APIs OPFS de arquivo inteiro na main thread, sem SharedArrayBuffer/
  COOP-COEP. **the-light não tocado** (8f66004); core e caminho nativo intactos.
