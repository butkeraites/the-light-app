## Resumo

F5.32 faz o Reader honrar `?verse=N` (empurrado por busca `openHit` e por xref
`openXref`), que antes era descartado silenciosamente — o alvo caía no TOPO do
capítulo. Agora `[chapter].tsx` lê o param opcional, normaliza `string|string[]`
→ 1º, valida finitude/positividade (senão `null` = comportamento atual) e o
repassa como `anchorVerse` ao `ReaderChapterView` compartilhado, que mede o Y de
cada linha via `onLayout`, rola até o alvo (`scrollTo`, com folga) e aplica um
realce TRANSITÓRIO reusando o visual `verseSelected` (some sozinho após ~2,5s),
sem abrir o painel de seleção. Re-executei os 6 gates de forma independente:
todos verdes. Verifiquei a substância de forma adversarial nos 6 pontos.
Veredito: **passed**.

## Achados dos 6 checks (adversarial)

1. **No-param regression (crítico) — OK.** Sem `verse`: `verseParamRaw`=undefined
   → `verseParam`=NaN → `Number.isFinite(NaN)` false → `anchorVerse=null`. No
   componente, `anchorVerse==null` → `setFlashVerse(null)` (React faz bail-out,
   pois já é null → sem re-render), `return`. Nenhuma rolagem, nenhum flash;
   `isAnchored` sempre false → nenhuma linha ganha `verseSelected`. Comportamento
   IDÊNTICO ao de hoje. O `onLayout` adicionado a cada linha é uma escrita em ref
   (`offsetsRef`) que só dispara rolagem se `pendingRef` casar (null na montagem)
   — invisível e sem re-render.
2. **Param honrado — OK.** `anchorVerse=N` → efeito seta `flashVerse` e chama
   `scrollToVerse`. Como `passage` carrega async, o `ReaderChapterView` MONTA
   fresco (offsets vazios) → `scrollToVerse` grava `pendingRef=N` e retorna; o
   `onLayout` da linha-alvo então completa a rolagem. O flash usa `flashVerse`,
   separado de `selectedVerse` → NÃO abre o painel de ação. Auto-limpa após 2,5s
   com guarda (`cur === anchorVerse`) contra apagar um flash mais novo; cleanup
   limpa o timer.
3. **Robustez — OK.** N fora de faixa (999) → finito/>0 mas nenhuma linha casa →
   `pendingRef` nunca dispara → sem rolagem para offset ruim, sem crash; flash
   some em 2,5s. `?verse=abc`/`0`/`-5`/`""`/`"  "` → `null`. Array
   (`verse[0]`) → 1º elemento. Fracionário (3.5) / enorme (1e10) → finito>0 mas
   sem linha correspondente → no-op inofensivo. Normalização sólida.
4. **Anti-alucinação/offline — OK.** Nenhuma mudança de conteúdo/store; texto
   segue do `getChapter`. `grep` não achou `fetch`/rede/segredo nos 2 arquivos.
5. **Cross-target — OK.** Só `ScrollView`/`scrollTo`/`onLayout`/`LayoutChangeEvent`
   /`nativeEvent.layout.y` — todos cross-platform (react-native-web + nativo).
   Sem DOM-only (`document`/`window`/`getElementById`), sem `fetch`, sem
   `scrollToIndex` (usa corretamente `scrollTo` para `ScrollView`). Componente
   compartilhado → web + nativo cobertos.
6. **Não negociáveis — OK.** the-light em `225b8c9`; `.claude/settings.json` NÃO
   está no commit (apenas `M` na árvore de trabalho, não-produto); commit

## Julgamento sobre o teste de render ausente

Aceitável. O projeto não tem renderer React nos guards web (são esbuild+node,
sem jsdom/react-test-renderer), então uma asserção de scroll/highlight exigiria
uma harness nova e potencialmente instável. A lógica é pura, simples e foi
traçada adversarialmente; a propriedade de segurança central (sem-param = no-op)
é comprovável por inspeção (`anchorVerse=null` → efeito curto-circuita). Cobertura
via tsc + build + reuso de testIDs/estilo existentes. Sugestão (não bloqueante):
extrair a normalização `string|string[]→finito>0` para um helper puro e cobri-la
com um unit-test determinístico numa próxima varredura.
