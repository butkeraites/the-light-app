dar **paridade WEB** dos planos de leitura, mantendo ZERO-DRIFT com o nativo.

## Passos sugeridos
1. **Fronteira de GERAÇÃO cfg-free:** as 3 fns da F5.1 (`list_reading_plans`, `reading_plan_day`,
   `reading_plan_day_index`) que hoje são impl nativa `#[cfg(not wasm32)]` + **stub web** viram
   **cfg-free** (chamam `userdata::plans` real, agora disponível sob ai-pure). Os Records
   `ReadingPlanSummary`/`ReadingPlanDay` já são puros. Regenerar bindings (nativo + web) — a
   assinatura NÃO muda → idempotente; o web ganha a impl REAL (não mais stub).
2. **Progresso WEB em OPFS (app-side):** `PlanStore` é embedded → as fns de PERSISTÊNCIA da F5.4
   (`reading_plan_progress`/`start_reading_plan`/`set_reading_plan_completed`/`clear_reading_plan`)
   seguem native+stub. Para o web, criar uma camada TS que persiste o progresso (`active.json`
   equivalente) em OPFS, reusando o VFS de userdata (F1.16) — mesmo formato `{plan_id, start_date,
   completed}` que o core usa (para paridade). Validar o plan_id contra o CATALOG (via
   `list_reading_plans`) antes de gravar (anti-alucinação).
3. **Rota `/plans` no web:** remover o `PlansWebNotice`; a tela usa a geração cfg-free + o progresso
   OPFS. Reusar a mesma UI da F5.7 (nativo) o máximo possível.
4. **Prova headless web:** self-test (molde dos `*-selftest`/`test:web:*`) provando: 3 planos
   gerados no web (nomes do core), dia de hoje com refs, round-trip do progresso em OPFS (iniciar →
   marcar → reabrir persiste), plan_id inválido rejeitado. Sem rede/chave.
