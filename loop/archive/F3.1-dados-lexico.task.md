## Objetivo

Popular as tabelas **`original_tokens`** (tokens de língua original + número de
Strong por palavra), **`lexicon`** (glosas breves TBESH/TBESG por Strong) e
**`scholarly_sources`** (atribuição das fontes) do banco
`assets/data/bible.sqlite`, **rodando o importador canônico `xtask
import-scholarly`** do `the-light` (rev pinado **`c8ecb2f`**, o **mesmo** que
`core/Cargo.toml` consome — ADR-0002) — **sem modificar/forkar `the-light`**, de
forma **reprodutível** (um script) e **idempotente** (reimportar não duplica).

Esta é a **primeira** tarefa da Fase 3 (estudo profundo) e é de **DADOS/pipeline**,
análoga à **F1.1/F1.7**: **sem UI** e **sem fronteira**. A fronteira de léxico
(`lexical_entries` → `ai::lexicon::verified_lexicon`) é a **F3.2**; o estudo
profundo é a **F3.3**; a UI com léxico inline + atribuição visível + propagação ao
subset `reading-sample.sqlite` (bundling) é a **F3.5**. Não antecipar.

O léxico **NÃO exige banco próprio**: usa o **MESMO `bible.sqlite`**. As tabelas
`original_tokens`/`lexicon`/`scholarly_sources` fazem parte do schema v2 do core
(criadas por `Store::open`) e ficam **vazias** até serem populadas por esta
importação. Sem esses dados, `ai::lexicon::verified_lexicon` devolve **vazio** e o
estudo declara "sem base léxica" — **honesto**, nunca inventa (anti-alucinação).

**Licenciamento (correção importante):** o dado do core é **STEP Bible /
STEPBible-Data — CC BY 4.0** (Tyndale House, Cambridge), **não domínio público
puro**. A **numeração de Strong** é PD, mas os **tokens e glosas amalgamados**
(TAHOT/TAGNT/TBESH/TBESG) que o core usa são **CC-BY** → **atribuição
obrigatória**. Só embarcamos **dados livres** (PD/CC0/CC-BY); CC-BY exige
atribuição — esta tarefa **registra** a origem/licença/string em **ADR-0026** (a
exibição visível é responsabilidade da F3.5).

Anti-alucinação/licenciamento: os tokens/glosas vêm **sempre** do importador
canônico sobre a fonte CC-BY registrada no `scholarly` do core — **nunca**
inventados, **nunca** fonte não-livre (a denylist do core recusa `sblgnt`/`bdag`/
`halot`/… por código).

## Contexto / arquivos

**Subcomando REAL descoberto na fonte do xtask (rev `c8ecb2f`):** é
**`import-scholarly`** (subcomando dedicado). Confirmado em
`~/.cargo/git/checkouts/the-light-9eb8809a6d68281a/c8ecb2f/xtask/src/main.rs`:
`Some("import-scholarly") => scholarly_import::run(&args[1..])`. A lógica vive em
`xtask/src/scholarly_import.rs` (wrapper fino) → `the_light_core::scholarly::import`
(o parser TSV STEP, download, license enforcement e inserção vivem **no core**).
Leitura permitida — cópia read-only do rev pinado; o repo canônico
`/Users/butkeraites/Documents/the-light` é **leitura-apenas / não tocar**.

**Flags de `import-scholarly`** (de `scholarly_import.rs::run`, parser próprio):
`--version <ids>` (lista separada por vírgula, ex.: `tahot,tagnt,tbesh,tbesg`;
**default = todos** os SPECS = `scholarly::default_datasets()`), `--db <PATH>`,
`--seed-dir <DIR>` (default `data/seed/scholarly`), `--force`, `--offline`.
Qualquer outra flag → o xtask aborta ("argumento desconhecido para
`import-scholarly`"). **Sem `--version` importa os quatro conjuntos.**

**Fonte (CC-BY, URLs fixadas no core `scholarly.rs`):** STEP Bible /
STEPBible-Data, base
`https://raw.githubusercontent.com/STEPBible/STEPBible-Data/master/...`. Conjuntos
(todos CC-BY, um parser TSV STEP):
- **TAHOT** (Hebrew OT, 4 arquivos Gen-Deu/Jos-Est/Job-Sng/Isa-Mal) → `original_tokens`
  (testamento OT) — ~305.577 tokens.
- **TAGNT** (Greek NT) → `original_tokens` (testamento NT) — ~140.000 tokens.
- **TBESH** (léxico breve hebraico, BDB-derivado) → `lexicon` — ~11.682 entradas.
- **TBESG** (léxico breve grego) → `lexicon` — ~11.035 entradas.
O core grava/lê os arquivos no `<seed-dir>` e baixa por rede **só** se ausentes e
sem `--offline` (rede em build permitida). **Volume grande** (dezenas de MB).

**Guarda de drift no próprio core:** `scholarly.rs` fixa mínimos por conjunto
(`tahot => 300_000`, `tagnt => 100_000`, `tbesh|tbesg => 5_000`) — importar abaixo
disso **aborta** ("fonte incompleta?"). Trate os aceites como **guardas** (≳ N),
não igualdade exata (o snapshot do STEP master pode variar levemente).

**Atribuição (obrigatória, verbatim — `scholarly::ATTRIBUTION`, gravada em
`scholarly_sources.attribution`):**
`Credit it to 'STEP Bible' linked to www.STEPBible.org (data based on work at
Tyndale House, Cambridge; CC BY 4.0)`. O `import-scholarly` a imprime ao final
("STEPBible, CC BY 4.0") e a grava no banco. **License enforcement do core:** só
`public-domain`/`cc0`/`cc-by*` são aceitos (CC-BY entra **com** atribuição).

**Schema (uma fonte da verdade — migrações do core, aplicadas por `Store::open`):**
`original_tokens(testament, book_number, chapter, verse, word_index, surface,
lemma, strongs, strongs_raw, source_id, …)`; `lexicon(strongs, lemma, gloss,
gloss_pt, source_id, …)`; `scholarly_sources(id, name, license, embeddable,
attribution, url, version)`. Estas tabelas já existem (vazias) num `bible.sqlite`
migrado por `Store::open` do rev `c8ecb2f`.

**Rev do banco vs. rev do xtask (decisão → ADR-0026):** o `gen-bible-db.sh` atual
gera `bible.sqlite` com o xtask do rev **`8f66004`**; o app hoje pina **`c8ecb2f`**.
Rodar `import-scholarly` do **`c8ecb2f`** contra o banco existente é seguro
(`Store::open` aplica as migrações v2 idempotentemente, criando as tabelas se
faltarem). **Caminho recomendado:** **alinhar o `gen-bible-db.sh` para o rev
`c8ecb2f`** (verses + xref + scholarly no **mesmo** rev pinado do app) — reduz o
descasamento 8f66004↔c8ecb2f e mantém um pipeline único. Registrar a decisão de
rev em ADR-0026.

**Como rodar SEM tocar `the-light` (mesmo padrão da F1.1/F1.7 / ADR-0013):** rodar
o member `xtask` do **checkout pinado do cargo**
(`$HOME/.cargo/git/checkouts/the-light-9eb8809a6d68281a/c8ecb2f/xtask/Cargo.toml`)
com `CARGO_TARGET_DIR` **FORA** do checkout (`.cache/xtask-target`, ignorado) e
`--locked`. **Não** rodar dentro de `/Users/butkeraites/Documents/the-light`.

**Script (F1.7 = molde):** `scripts/gen-bible-db.sh` já roda `import` (verses+FTS) e
`import-xref` (xrefs) com exatamente esse padrão de isolamento (checkout pinado,

**após**/exigir `gen-bible-db.sh` — rodar `import-scholarly` num `--db` inexistente

**Armazenamento (decisão → ADR-0026):** o `bible.sqlite` já é **gerar-ignorado**

**Documentos:** `DECISIONS.md` (ADR-0002 pin; ADR-0013 gerar-ignorado + xtask sem

## Restrições

- **NÃO modificar nem forkar `the-light`** (`/Users/butkeraites/Documents/the-light`
  é leitura-apenas; consumo é git dep pinada `rev c8ecb2f`, ADR-0002). Rodar o
  `xtask import-scholarly` **sem** escrever no source dele (`CARGO_TARGET_DIR` fora
  do checkout + `--locked`). Se a **única** forma de invocar o importador exigir
  tocar/alterar o `the-light` → `blocked` com o erro EXATO (decisão humana) —
  **não** improvisar uma cópia paralela do parser/léxico.
- **Só dados LIVRES CC-BY** (STEP Bible / STEPBible-Data, exatamente as fontes
  registradas no `scholarly` do core). CC-BY exige **atribuição** — registrar a
  string verbatim em ADR-0026 (exibição visível é F3.5). **Jamais** apontar para
  fonte não-livre (a denylist do core recusa `sblgnt`/`bdag`/`halot`/…); **nenhum**
  dado de léxico inventado/hardcoded (anti-alucinação: vem verbatim da fonte
  canônica via xtask).
- **Offline-first é regra de RUNTIME:** rede só em **dev/build** (download dos TSV
  STEP). O app em runtime **não** faz rede. **Segredos nunca** em git/log.
- **Blocked legítimo (rede/seed):** os datasets STEP **não estão em cache**
  (`data/seed/scholarly` vazio) → o `import-scholarly` **baixa por rede** (dezenas
  de MB). Se o ambiente estiver **offline** e sem seed → rodar com `--offline`
  falha → escrever `done/F3.1.result.md` com `status: blocked`, a **saída literal**
  do comando e a decisão pendente (origem/seed do dataset STEP). **Não** fabricar
  dados para "passar".
- **Armazenamento:** o `bible.sqlite` continua **gerar-ignorado** (ADR-0013); o
  seed-dir é **sempre ignorado**. Não commitar binário grande; registrar o tamanho
  novo em ADR-0026.
- **Escopo:** F3.1 entrega **só os DADOS** (popular `original_tokens`/`lexicon`/
  `scholarly_sources` no `bible.sqlite`). A **fronteira** de léxico
  (`lexical_entries`/`verified_lexicon`) é **F3.2**; o **estudo profundo**
  (`deep_study`/`study`) é **F3.3**; a **UI** com léxico inline + **atribuição STEP
  CC-BY visível** + propagação ao subset `reading-sample.sqlite` (bundling) é
  **F3.5**. Não antecipar.
