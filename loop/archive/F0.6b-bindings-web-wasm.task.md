## Objetivo

Fechar o caminho **WEB end-to-end** da ponte Rust → Expo, usando o **caminho web
do `ubrn`** (não o flavor `jsi`/nativo e **não** um `cargo build` cru):

1. Gerar os bindings **web/wasm** da fronteira `the-light-app-core` pelo caminho
   web do `uniffi-bindgen-react-native` (`ubrn`, versão fixada `0.31.0-3`),
   produzindo o módulo wasm (wasm-bindgen) + os `.ts` web.
2. Criar o **glue** em `app/web/` (carregar/instanciar o módulo wasm de forma
   assíncrona + um wrapper fino sobre `parseReference`).
3. Ligar `app/app/index.tsx` para que, **no web**, a referência digitada seja
   resolvida **pelo Rust (wasm)** — não por eco nem lógica TS.
4. **Provar de forma HEADLESS** (teste node) que `parseReference` via wasm
   resolve `"Jo 3.16"` (PT) e `"John 3:16"` (EN) para o mesmo
   `book=43, chapter=3, verses=Single 16`.

Esta tarefa é o downstream da F0.6a: o core já está feature-gated e a fronteira
consome o `the-light-core` **puro** (sem `rusqlite`/`reqwest`) no alvo wasm. O
que falta é o **caminho web** (geração wasm + glue + tela + prova).

## Contexto / arquivos

- **F0.6a (aceita):** `core/Cargo.toml` re-pinado no rev `8f66004` com matriz de
  features por alvo — web/wasm = núcleo PURO (`model`+`reference`), nativo =
  `embedded`. `cargo build -p the-light-app-core --target wasm32-unknown-unknown`
  já é verde; `cargo tree` wasm não traz `rusqlite`/`reqwest`. **A fricção
  SQLite-no-WASM está resolvida** — não reabra essa discussão aqui.

  `cargo build` **cru** da fronteira p/ `wasm32-unknown-unknown` falha em
  `uniffi_core 0.31.2` (`UniffiCompatibleFuture: … + Send`; wasm é single-thread,
  futures não são `Send`). **Por isso esta tarefa NÃO pode usar build cru** — deve
  usar o **caminho web do `ubrn`**, que é desenhado para web/wasm (gera um
  wasm-crate wrapper + roda `wasm-pack`/`wasm-bindgen`) e deve tratar o setup do
  uniffi p/ wasm. Pode exigir uma **feature/config do `uniffi` p/ wasm** ou
  ajustes de config do próprio `ubrn web`.
- **Caminho web do ubrn (já localizado no pacote, confirmar na execução):** o
  CLI `ubrn 0.31.0-3` tem o subsistema wasm — subcomando provável **`ubrn web
  build`** (ver `node_modules/uniffi-bindgen-react-native/crates/ubrn_cli/src/wasm/`
  e o guia `node_modules/uniffi-bindgen-react-native/docs/src/guides/web/getting-started.md`).
  O fluxo: compila p/ wasm32 → gera um wasm-crate → roda `wasm-pack`/`wasm-bindgen`
  → emite `index.web.ts` + saída `wasm-bindgen/` (`index_bg.wasm`, `index.js`,
  `index.d.ts`). Runtime = `@ubjs/core`; init assíncrono via `uniffiInitAsync()`.
  O `ubrn.config.yaml` ganha um bloco `wasm:`/`web:` (camelCase, schema `WasmConfig`:
  `manifestPath`, `wasmCrateName`, `features`, `defaultFeatures`, `target`
  web/bundler/nodejs, `tsBindings`/alias `ts`, `entrypoint`…). **Confirme as chaves
  EXATAS na fonte/fixtures** (`crates/ubrn_cli/src/wasm/config.rs` e
  `crates/ubrn_cli/fixtures/defaults/ubrn-wasm.*.config.yaml`) antes de escrever.
- **Hoje:** `scripts/gen-bindings.sh` só faz o caminho **jsi/nativo** (host-only,
  `ubrn generate jsi`). NÃO o altere para wasm — crie um script novo dedicado
  ao web (`scripts/gen-bindings-web.sh`). `ubrn.config.yaml` é a fonte da verdade
  dos caminhos e será **estendido** com o bloco wasm/web.
- **`app/web/`** existe como diretório do glue do web (hoje só `.gitkeep`). É AQUI
  que entra o wrapper hand-written que carrega o wasm e expõe `parseReference`.
- **`app/app/index.tsx`** hoje é a tela F0.5 "sem core" (comentário diz que a
  ligação real chega na F0.6). **A guarda de escopo da F0.5 NÃO vale mais aqui** —
  esta tarefa É a ligação. No web, a tela deve exibir a referência resolvida pelo
  Rust (livro/cap/verso), não o eco do input.
- **Ferramentas externas:** `wasm32-unknown-unknown` já instalado (F0.6a).
  `wasm-pack`/`wasm-bindgen-cli` podem **não** estar instalados (não constam na
  tabela de ambiente do DECISIONS) — instalá-los é **dev/build** (rede permitida;
  offline-first é regra de RUNTIME). Se faltarem, o script deve instalá-los de
  forma reprodutível/fixada (ou falhar com mensagem clara).
- **App Expo** vive em `app/` (package.json próprio; `@ubjs/core` precisa estar
  nas deps do app para os `.ts` web — o guia do ubrn exige isso).

   core nem na forma da fronteira para isso.
3. **Criar `scripts/gen-bindings-web.sh`** reproduzível (espelhando o estilo de
   `scripts/gen-bindings.sh`): `set -euo pipefail`, raiz via `git rev-parse`,
   garantir `wasm32` + `wasm-pack`/`wasm-bindgen` (instalar fixado se faltar),
   rodar o caminho web do `ubrn`, limpar/regenerar a saída de forma determinística,
   sair 0. Comentar o ACHADO uniffi-wasm e por que se usa o caminho web.
4. **Glue em `app/web/`** (hand-written, VERSIONADO): um módulo que carrega/instancia
   o wasm de forma assíncrona (`uniffiInitAsync()` ou equivalente do ubrn web) e
   expõe um `parseReference(input)` tipado, delegando ao binding gerado. Não
   reimplemente parsing.
5. **Ligar `app/app/index.tsx`** no web: ao submeter o input, chamar o glue e
   exibir a `Reference` resolvida pelo Rust (book/chapter/verses). Manter o

6. **Prova HEADLESS** (teste node, sem browser/Expo): importar o módulo
   wasm-bindgen + bindings web gerados, inicializar o wasm e chamar
   `parseReference("Jo 3.16")` e `parseReference("John 3:16")`, asseverando
   `book===43`, `chapter===3` e a variante `Single` com valor `16` em ambos.
   Nota: o alvo wasm-bindgen para node headless pode exigir `--target nodejs`/
   `bundler` ou instanciação manual do `.wasm` (o `--target web` espera
   `fetch`/`instantiateStreaming`). Escolha o alvo que torna a prova node viável
   **sem** falsear nada.
7. **`npx expo export --platform web`** a partir de `app/` continua saindo 0
   (pode exigir ensinar o Metro a tratar `.wasm` como asset e/ou

## Restrições

- **NÃO modificar/forkar `the-light`, o `core/` (a fronteira) nem a forma da
  fronteira para "fazer funcionar".** O core já está feature-gated (F0.6a) e a
  fronteira já compila p/ wasm. Se a única forma de avançar exigir mudar o core
  de novo → isso é **PR + ADR no `the-light`** (ação humana) ⇒ pare com `blocked`
  (não é o caso esperado: o achado uniffi-wasm é do `uniffi`/`ubrn`, não do core).
- **NÃO fingir/stubar** o wasm ou os bindings, e **NÃO reimplementar o parsing em
  TS.** A referência DEVE ser resolvida pelo Rust (wasm) — uma fonte da verdade.
  Eco do input ou lógica TS = reprovação.
- **Rede só em dev/build** (instalar `ubrn`/`wasm-pack`/`wasm-bindgen`, resolver
  deps de cargo/npm). **Nada de rede no runtime** do produto; offline-first é
  regra de runtime. **Segredos nunca em git/log.**
- **Bindings/wasm gerados não são versionados** — mantenha-os ignorados (estenda
  `.gitignore` se a saída cair fora de `bindings/`). Versiona-se só
  glue/scripts/config/ADR/ajustes de app.
- **Se o caminho web do `ubrn 0.31.0-3` for inexistente/imaturo/inoperante**, OU
  não resolver o obstáculo uniffi-wasm (`+Send`) **sem hacks proibidos** (mexer no
  core/fronteira, stubar, reimplementar em TS), **reporte `blocked`** com o **erro
  / saída EXATA** do comando que falhou. A decisão a tomar nesse caso é
  **estratégica sobre o caminho web do ubrn** (ex.: aguardar release que trate
  wasm/`uniffi`; usar `wasm-bindgen` manual sobre a fronteira; ou outra estratégia
  de bindings web) — **NÃO improvise** uma saída.
- **ADR-0006** deve ser escrito em `DECISIONS.md` se uma decisão de caminho web
  for tomada (subcomando/flavor do ubrn web escolhido, config wasm, alvo
  wasm-bindgen para a prova headless, tratamento do `+Send`).
