## Resumo

A F0.9 prova a **camada de store no nativo**: a fronteira UniFFI expõe
`get_passage(db_path, reference, translation) -> Result<Passage, CoreError>` que
**delega** ao `the-light-core` (`the_light_core::store::Store::open` +
`source::EmbeddedSource::passage`, com a trait `BibleSource` em escopo) para ler
uma passagem do SQLite local — **sem** reimplementar SQL nem parsing. Os Records
UniFFI `Passage`/`Verse` espelham o `model` do core (com `From<…>`). O
`assets/data/sample.sqlite` (≈128 KB) está **versionado**, contém o subset KJV de
**domínio público** (1 `translations` kjv + 1 `books` John/43 + João 3:16
verbatim) e é **regerável de forma determinística** por `scripts/gen-sample-db.sh`
→ `core/examples/gen_sample_db.rs` (schema = migrações do core via `Store::open`).

Verifiquei de forma independente: o teste Rust de host
`store_tests::get_passage_reads_john_3_16_verbatim_from_sample` abre o sample e
assere o **texto KJV verbatim** + `book=43, chapter=3, verse=16` (8 passed).
**Anti-fake confirmado** (leitura de `core/src/lib.rs`): o corpo de produção de
`get_passage` LÊ do store (`Store::open` → `EmbeddedSource::passage`); a constante
KJV só aparece no **example** (que insere o dado no banco) e no **módulo de teste**
(guarda de asserção) — **nunca** no caminho de leitura de produção. Sem regressão
web (`cargo tree --target wasm32` sem `rusqlite`/`reqwest`), `the-light` intacto
(commit toca só `the-light-app`; `rev 8f66004` inalterado), `fmt`/`clippy` limpos
e working tree limpo. **Decisão: passed.**

## Decisões / ADRs

- **ADR-0010 (aceito):** Camada de store no nativo.
  - **API do store consumida:** `Store::open(path)` (cria/migra schema das
    migrações do core — uma fonte da verdade) + `EmbeddedSource::new(&store)` +
    `BibleSource::passage(&Reference, &TranslationId)`. `model::Passage/Verse/
    TranslationId` puros. Schema (`translations`/`books`/`verses`) vem das
    migrações, não de SQL à mão.
  - **`sample.sqlite` VERSIONADO** (`assets/data/`, ≈128 KB, KJV domínio público):
    escolhido por determinismo do teste de host (sem passo de geração prévio) e
    por ser asset pequeno 100% livre; **regerável** por `gen-sample-db.sh`
    (reprodução byte-idêntica confirmada nesta revisão).
  - **Origem/licença:** João 3:16 da King James Version, **domínio público**
    (único texto bíblico no repo; nada protegido/não-livre).
  - **Gating de CORPO por alvo (não da exportação):** a `#[uniffi::export]
    get_passage` existe em todos os alvos; só o corpo que toca
    `store`/`source`/`rusqlite` é `cfg(not(target_arch="wasm32"))`. No web, stub
    de `CoreError`. Gatear a exportação inteira quebraria o link wasm do `ubrn
    build web` (símbolo ausente) — gating de corpo mantém o grafo wasm puro E a
    surface da fronteira uniforme. Verificado: `cargo tree wasm` sem `rusqlite`.

## Bloqueios / perguntas

Nenhum bloqueio para o aceite. Itens de escopo posterior (não bloqueiam F0.9,
registrados no ADR-0010):

- **Store WEB** (`wa-sqlite` + OPFS) é a **F0.10** — no alvo web `get_passage` é
  stub de erro até lá.
- **Embarque (bundling)** do `sample.sqlite` no app nativo (iOS/Android) e o run
  nativo de `getPassage` via Turbo Module são **fase posterior / verificação
  adicional** — F0.9 prova a leitura via store por **teste Rust de host**, que é
  a prova obrigatória e está verde.
