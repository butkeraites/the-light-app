## Objetivo

Entregar a **UI nativa (iOS/Android) de estudo profundo** ancorada numa passagem do
Reader: seletores de **modo × lente × profundidade** que chamam `deep_study` (F3.3) e
exibem, com **anti-alucinação visível**, a `passage_text` (do STORE) **separada** da
`interpretation` (IA) + `sections` + `citations` + `warnings`, mais o **léxico Strong
inline** (`lexical_entries`, F3.2) com a **atribuição STEP CC-BY VISÍVEL** (ADR-0026,
obrigatória). Estender o glue nativo `app/web/reading.ts` (`deepStudy`/`lexicalEntries`
→ JSI → Rust), regenerar os bindings nativos e provar tudo por um **self-test
determinístico headless** com o provedor `"mock"` (sem chave, sem rede). Web = stub
(estudo web = F3.12). **Não** toca a fronteira Rust/`the-light` (F3.2/F3.3 já entregaram
`deep_study`/`lexical_entries`; nenhuma função nova de fronteira).

## Contexto / arquivos

**Fronteira JÁ entregue (F3.2/F3.3, aceitas) — assinaturas REAIS confirmadas em
`core/src/lib.rs` (NÃO modificar este arquivo):**

- **`deep_study`** (`core/src/lib.rs:1311`) — ordem/tipos REAIS (atenção: `lang` vem
  ANTES de `provider_name`, e a passagem é `book/chapter/verse` numéricos, NÃO uma
  string "John 3:16"):
  ```
  deep_study(db_path: String, translation: String, book: u8, chapter: u16,
             verse: Option<u16>, mode: StudyMode, lens: StudyLens, depth: StudyDepth,
             lang: String, provider_name: String, key: Option<String>,
             model: Option<String>) -> Result<StudyResultOut, CoreError>
  ```
  No binding JS (camelCase): `deepStudy(dbPath, translation, book, chapter, verse, mode,
  lens, depth, lang, providerName, key, model)`.
- **`lexical_entries`** (`core/src/lib.rs:948`) — **SEM** `translation` (léxico é
  independente de tradução, chaveado por book/chapter[/verse]):
  ```
  lexical_entries(db_path: String, book: u8, chapter: u16, verse: Option<u16>,
                  lang: String, limit: Option<u32>) -> Result<VerifiedLexiconOut, CoreError>
  ```
  Binding JS: `lexicalEntries(dbPath, book, chapter, verse, lang, limit)`.
- **Enums UniFFI (variantes REAIS):** `StudyMode{Academic,Devotional,Introductory,Sermon}`
  · `StudyLens{Baptist,Presbyterian,Lutheran,Pentecostal,Catholic,Orthodox}` ·
  `StudyDepth{Overview,Exegetical,WordStudy}`.
- **Records REAIS:** `StudyResultOut{ reference, reference_label, mode, lens, depth,
  language, passage_text, interpretation, sections: StudySection[], warnings: string[],
  citations: StudyCitation[], provider, model }` · `VerifiedLexiconOut{ entries:
  LexEntry[], sources: string[] }` · `LexEntry{ strongs, lemma?, translit?, gloss?,
  occurrences, testament }` · `StudySection{ heading, body }` · `StudyCitation{ kind,
  key, author?, title?, locus?, publisher?, year?, url?, license?, attribution?,
  accessed?, quote? }`.
- **Atribuição STEP CC-BY:** vem em `VerifiedLexiconOut.sources` (cada item é o
  `scholarly_sources.attribution` verbatim, ADR-0026): **`Credit it to 'STEP Bible'
  linked to www.STEPBible.org (data based on work at Tyndale House, Cambridge; CC BY
  4.0)`**. A UI DEVE exibir esse(s) `sources` sempre que o léxico/estudo aparecer.

**DECISÃO REGISTRADA — léxico no subset bundled (propagar × provar só no host):**
o subset `reading-sample.sqlite` **NÃO** tem as tabelas de léxico (`original_tokens`/
`lexicon`/`scholarly_sources`; a F3.1 populou o `bible.sqlite`, não o subset — o gerador
não as copia). **Decisão adotada = (a) PROPAGAR o léxico ao subset** (molde F1.9 que
propagou xref / F1.5 que propagou `verses_fts`): estender `gen_reading_sample_db.rs`
para copiar, do `bible.sqlite`-fonte para o subset, (i) `original_tokens` dos versículos
dos livros incluídos (Gn/Sl/Jo), (ii) as linhas de `lexicon` referenciadas por esses
tokens (ou todas — poucas dezenas de milhares; decidir por tamanho no ADR), e (iii)
**todas** as `scholarly_sources` (4 linhas — atribuição STEP CC-BY). `Store::open` já cria
o schema v2 dessas tabelas no subset. Assim o self-test no device asserta `lexicon>=1`
para João 3 e `attribution_ok=true`. Registrar tamanho/estratégia em **ADR-0027** (molde
ADR-0014). *(Fallback só se a propagação se mostrar inviável sem hack: aceitar `lexicon=0`
no device e provar o léxico via bônus `if bible.sqlite exists` no host — mas isto NÃO é o
alvo; a propagação é de baixo risco, já feita para xref/FTS.)*

## Restrições (regras rígidas relevantes)

- **NÃO tocar `the-light` nem a fronteira Rust.** F3.2/F3.3 já entregaram `deep_study`/
  `lexical_entries`; F3.5 é **só** app/UI/glue/self-test + dados do subset. **Nenhuma**
  função nova de fronteira. Se a UI parecer exigir mudar `core/src/lib.rs`/`the-light`:
  **`blocked` + HALT** com o erro EXATO (o núcleo só muda via PR + ADR — decisão humana);
  não stubar/forkar/copiar o core.
- **SEM chave/rede na prova.** Só o provedor `"mock"` (sem `key`, sem `model`, sem SSE);
  a chave real + rede são a F3.10 (gate). Nenhuma chave é logada/exibida em claro.
- **Atribuição STEP CC-BY VISÍVEL é OBRIGATÓRIA** (ADR-0026): a string
  `Credit it to 'STEP Bible' linked to www.STEPBible.org …; CC BY 4.0` (de
  `VerifiedLexiconOut.sources`) aparece sempre que o léxico/estudo é exibido.
- **Anti-alucinação VISÍVEL:** `passage_text`/léxico/citações vêm do **store** e são
  exibidos com rótulo distinto, SEPARADOS da `interpretation` (LLM); texto bíblico e
  glosas **nunca** apresentados como saída do modelo; nenhum texto/glosa/citação
  hardcoded — tudo do RETORNO real da fronteira.
- **Só NATIVO** nesta tarefa (iOS/Android); estudo/léxico **web = F3.12** (`reading.web.ts`
  = stub). O `expo export web` deve seguir 0.
- **Offline-first / BYOK:** nada essencial exige rede/conta; a prova roda offline (mock).
- **Qualidade por tarefa:** `tsc --noEmit` limpo + eslint; export web 0; self-test verde
  sem regressão. Diante de conflito com qualquer regra acima: **HALT**, não improviso.
