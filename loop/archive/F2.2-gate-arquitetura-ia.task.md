## Objetivo

**Gate estratégico (decisão humana/arquitetural).** Por ser `gate: true`, o loop
**NÃO** executa esta tarefa autonomamente: o Driver escreve `loop/HALT` e o
**humano (Renan)/Guia** decide antes de liberar a construção de Gemini/chaves/UI
(F2.3–F2.7).

Esta tarefa **não tem código de produto a implementar**. É um **BRIEF DE DECISÃO**:
com a fronteira `ask` ancorada já provada com **MOCK** (F2.1, aceita — `357abca`),
definir **como a Fase 2 entrega a camada de IA BYOK** mantendo **offline-first**,
**BYOK**, **anti-alucinação** (texto sempre do store local) e **`the-light` intacto**
(`8f66004`; mudança no core só via PR + ADR). São **4 decisões** (D1–D4). O desfecho
do ciclo é **HALT p/ sign-off humano**; com a decisão registrada (novo **ADR-0023**),
o Driver re-escopa/semeia **F2.3** (Gemini, per D1), **F2.4** (chave nativa) e
**F2.7** (web, per D2). **F2.3/F2.4/F2.5/F2.6/F2.7/F2.8 dependem desta decisão.**

## 1. Contexto (o que está provado e o que o core oferece)

### 1.1 O que a F2.1 provou (a base sobre a qual as opções incidem)
`core/src/lib.rs` expõe na fronteira UniFFI `ask_anchored(db_path, translation,
reference, question, provider_name, key: Option<String>, model: Option<String>,
lang) -> Result<AiAnswer, CoreError>`, que **delega** à camada de IA do core e
devolve o Record `AiAnswer { reference, cited_text, interpretation, provider,
model }`. A anti-alucinação está **materializada no contrato**: `cited_text` é a
passagem **numerada, verbatim do store** (via `ai::numbered_passage`), e
`interpretation` é a saída do **modelo** (via `ai::ask`). Verificado: **37/37**
testes de host verdes com `provider_name="mock"` (`key=None`, sem rede); a prova
anti-fake mostra que 3:16 vs 3:17 muda o `cited_text` (segue o store) mas **não** a
`interpretation` (vem do modelo). O corpo que toca `ai`/store é
`cfg(not(target_arch = "wasm32"))` + **stub web** (`"ai não disponível no alvo web
(F2.7)"`) → o grafo wasm permanece **puro** (sem `rusqlite`/`reqwest`).

### 1.2 O que o `the-light-core` JÁ oferece (símbolos REAIS, checkout só-leitura `8f66004`)
Fonte: `~/.cargo/git/checkouts/the-light-9eb8809a6d68281a/8f66004/crates/the-light-core/src/ai/`.

  `name()`, `model()`, `complete(system, user) -> Result<String>`, `chat(system,

  fixa, **sem rede**). **Consequência para D1:** dá para escrever

  **Consequência-chave para D1:** um `GeminiProvider` construído **localmente** pode
  ser passado como `&dyn LlmProvider` para a orquestração RAG/anti-alucinação do

- **Fábrica `build_provider(name, key: Option<String>, model: Option<String>) ->

  (**sem gemini**); `default_model(provider)` — `ai/providers.rs:26`
  (`_ => ""`, **sem** braço gemini); `estimate_cost_usd(model, in, out)` —
  `ai/providers.rs:69` (**sem** preço de gemini). Estes três são os pontos onde
  Gemini precisaria de entradas sob D1=A (PR ao core).
- **Transporte = `reqwest::blocking`** — `ai/providers.rs:83`

- **Ancoragem/anti-alucinação — funções puras** (`ai/study.rs`): `numbered_verses`

  related)` (:127), `ask(...)` (:591). `ai/citation.rs` (`Citation`/
  `CitationCollector`) idem. Importam só `crate::model`, `serde`, `std` e
  `super::{citation,lexicon,research,prompts}` — **nenhum** `rusqlite`/`reqwest`
  direto.

### 1.3 A fricção central: IA no web (VISION §4 fricção #2)
- **Todo o módulo `ai` é `#[cfg(feature = "embedded")]`** — core `lib.rs:19-20`
  (junto de `config`/`export`/`scholarly`/`search`/`source`/`store`/`userdata`/
  `util`/`xref`). No alvo **web/wasm o `ai` NÃO existe** (matriz de features do
  ADR-0005) → `build_provider`/`ask`/`numbered_passage` **não** estão disponíveis no
  web, exatamente como store/busca/xref/userdata estavam antes da paridade web da
  Fase 1.
- `reqwest::blocking` **não compila** em `wasm32-unknown-unknown` (VISION §4 #2). O
  transporte precisa ser plugável: nativo = `reqwest`; web = provider que delega
  `fetch` ao JS (VISION §4: "a ancoragem anti-alucinação continua 100% no Rust").
- **Nuance importante (contra D2=C "expor só as partes puras via wasm"):** embora
  `study.rs`/`citation.rs` sejam **logicamente puros**, ambos **importam**
  `super::research::WebSource` (`ai/study.rs:11`, `ai/citation.rs:18`) e `research.rs`
  usa `reqwest` → as funções puras estão **estruturalmente acopladas** ao grafo
  `embedded`/`reqwest`. Torná-las compiláveis em wasm exigiria **desacoplar/mover**
  código no `the-light` = **PR + ADR** (precedente ADR-0005). Ou seja: "rodar a
  ancoragem pura no web" **não** é grátis hoje.

### 1.4 Nuance de runtime (amarrada a D4)
Os provedores do core são **não-streaming** (`complete` devolve `String` completa).
No nativo, uma chamada `reqwest::blocking` dentro do JSI **bloqueia a thread** até a
resposta — aceitável para o MVP (uma resposta), mas é o motivo de D4 (streaming
melhoraria a UX, ao custo de callbacks/observers sobre UniFFI + provável core change).

## 2. As 4 decisões (com Opções · Prós/Contras · Recomendação do Planner)

> O Planner recomenda, **não decide**. Cada decisão traz o trade-off honesto.

### D1 — Onde vive o provedor **Gemini**?
O core tem anthropic/openai/ollama, **não** gemini (`ai/mod.rs:30`,
`ai/providers.rs:26,69`). O endpoint alvo é `generativelanguage.googleapis.com`
(forma de API distinta: `contents`/`parts`, chave via `?key=`).

- **(A) PR ao `the-light-core`** — adicionar `GeminiProvider: LlmProvider` +
  entradas gemini em `PROVIDERS`/`default_model`/`estimate_cost_usd`.
  - **Prós:** **uma fonte da verdade** (todos os provedores no mesmo lugar);
    **CLI/TUI e futuros apps ganham** Gemini de graça; segue o precedente sancionado
    **ADR-0005** (PR não-quebrante + ADR).
  - **Contras:** **toca o `the-light`** → **exige autorização humana** e é **ação
    outward** (o loop não faz sozinho → spec de PR em `loop/proposals/` + HALT p/
    merge, como o feature-gating). Ciclo mais longo.
- **(B) `impl LlmProvider` LOCAL na fronteira** (`core/src/` do app) — o trait é
  público (`ai/mod.rs:327`) e `ai::ask` aceita `&dyn LlmProvider` (`ai/study.rs:591`),
  então a fronteira constrói o `GeminiProvider` e chama `ai::ask(&gemini, …)`; a
  fronteira roteia `provider_name == "gemini"` e provê `default_model`/custo gemini.
  - **Prós:** **autônomo** (o loop avança sem tocar o `the-light`); **rápido**;
    **preserva a anti-alucinação** (RAG/prompt/citação continuam no core, intactos);
    espelha o padrão já usado na Fase 1 (web reimplementa infra na fronteira,
    ADR-0011/0019–0022) — mantendo o **domínio** no core.
  - **Contras:** **duplica** o padrão dos provedores do core (Gemini fica só no nosso
    app; CLI/TUI não ganham); se um dia o core adotar Gemini, há convergência a fazer.
- **(C) Adiar Gemini** — usar só anthropic/openai/ollama que o core já tem.
  - **Prós:** entrega IA BYOK **já** (Claude/GPT), sem novo provedor; menor superfície.
  - **Contras:** **contraria o escopo explícito da Fase 2** (o título é "Claude · GPT
    · **Gemini**"); só empurra a mesma decisão para frente.

**Recomendação do Planner:** **(B) impl local na fronteira** — é o caminho **mais
autônomo e sem tocar o `the-light`**, e é **tecnicamente viável hoje** (trait público
+ `ask` recebe `&dyn`), preservando a anti-alucinação no core. **Ressalva honesta:**
se o sign-off valoriza mais a **fonte única** (CLI/TUI + apps futuros) do que a
autonomia do loop, **(A) PR sancionado** é a escolha arquiteturalmente "mais correta"
— e tem precedente (ADR-0005). Não recomendo **(C)** (adia o escopo declarado).

### D2 — Caminho de **IA no WEB**
O módulo `ai` é `embedded`-only (core `lib.rs:19-20`); `reqwest::blocking` não compila
em wasm (VISION §4 #2). Hoje o web tem só o **stub** de `ask_anchored`.

- **(A) Reimplementar o caminho de IA em TS no web** — espelhar `numbered_passage`/
  `ask_context`/prompt/citação em TS + chamada REST ao provedor via `fetch`; o texto
  do versículo vem do **store web** (`wa-sqlite`/OPFS, F1.13–F1.16).
  - **Prós:** **paridade web** de IA; precedente ADR-0011/0019–0022 (infra em TS no web).
  - **Contras:** **duplica a lógica de prompt/citação** → **risco de drift do
    anti-alucinação** entre nativo (Rust) e web (TS) — o mais sensível do produto;
    exige testes de paridade rígidos p/ mitigar. Mais superfície a manter.
- **(B) IA nativa-only por ora** — web permanece sem IA (leitura/busca/xref/notas da
  Fase 1 já são **paridade total** no web); IA fica "premium nativo". Adia a **F2.7**.
  - **Prós:** **menor risco** (a lógica sensível de ancoragem fica só no Rust, uma
    fonte); consistente com o padrão da Fase 1 (nativo-primeiro, web depois do gate);
    fecha a Fase 2 sem prometer no web algo ainda não validado com chave real.
  - **Contras:** web sem estudo assistido nesta fase (adiado p/ Fase 3 ou gate futuro).
- **(C) Expor as partes PURAS via wasm (PR ao core)** — tornar `numbered_passage`/
  `ask_context`/citação **wasm-safe** no `the-light` + transporte que delega `fetch`.
  - **Prós:** **uma fonte da verdade** da ancoragem também no web (zero drift).
  - **Contras:** **exige PR ao `the-light`** (as puras estão acopladas a
    `research::WebSource`/`reqwest` — §1.3) → ação humana; maior custo/risco agora;
    o transporte `fetch`-delegado sobre wasm ainda é infra nova.

**Recomendação do Planner:** **(B) nativa-only por ora (adiar F2.7)** — a Fase 1 já
dá **paridade web completa** de leitura/estudo offline, então adiar a IA no web
**não** deixa o web "quebrado"; e mantém a **anti-alucinação numa única
implementação (Rust)**, evitando o drift TS↔Rust — que é o maior risco de produto.
Reavaliar IA no web como gate próprio quando a IA nativa estiver validada (F2.6). Se
o sign-off exigir paridade web já, **(A)** é possível (com testes de paridade); **(C)**
só se aceitar um PR ao core.

### D3 — Armazenamento da **chave no web** (BYOK)
Nativo já decidido (F2.4): `expo-secure-store` (Keychain iOS / Keystore Android). A
chave **nunca** em git/log; trafega da UI direto para `build_provider(name, key,
model)` (a fronteira **não** persiste). Web precisa de política explícita.

- **session-only / in-memory** — re-inserir a cada sessão; nunca persiste.
  **Prós:** mais **seguro** (nada em disco/`localStorage`, imune a exfiltração
  persistente). **Contras:** UX de re-inserir a chave por sessão.
- **`localStorage` com aviso** — persistente. **Prós:** conveniência. **Contras:**
  **acessível a XSS** e a qualquer script da origem — impróprio para segredo.
- **sem chave no web** — coerente se **D2=B** (sem IA no web, não há chave a guardar).

**Recomendação do Planner:** se **D2=B**, então **sem chave no web** por ora (não há
IA web); e fixar, como **política de registro** para quando a IA web existir,
**session-only/in-memory** (nunca `localStorage` para segredo). Se **D2=A/C**,
adotar **session-only/in-memory + aviso** já na F2.7.

### D4 — **Streaming** vs resposta única (fronteira UniFFI)
Os provedores do core são não-streaming (`complete` → `String` completa,
`ai/providers.rs`). O `AiAnswer` da F2.1 já é uma resposta única.

- **(A) não-streaming primeiro** — uma resposta. **Prós:** **simples** sobre UniFFI/
  JSI; casa com o core atual; o contrato `AiAnswer` já existe. **Contras:** sem
  tokens incrementais (espera a resposta completa; a chamada nativa bloqueia — §1.4).
- **(B) streaming** — tokens incrementais. **Prós:** melhor UX em respostas longas.
  **Contras:** **callbacks/observers sobre UniFFI são bem mais complexos** e
  **exigiriam core change** (o `complete` não expõe stream) → PR + adiamento.

**Recomendação do Planner:** **(A) não-streaming primeiro** (MVP) — alinhado ao core
e ao `AiAnswer` já entregue; **(B) streaming** fica como **evolução** (Fase 3+), se e
quando justificar o core change.

## 3. Recomendação integrada do Planner (a combinação sugerida)

**D1 = (B) impl local na fronteira · D2 = (B) nativa-only, adiar F2.7 · D3 = sem
chave no web agora / política session-only registrada · D4 = (A) não-streaming.**

**Por quê, em conjunto:** maximiza a **autonomia do loop** (nada toca o `the-light`),
mantém a **anti-alucinação numa única implementação (Rust)** — sem drift TS↔Rust — e
entrega o escopo declarado (Claude · GPT · **Gemini**) no nativo com **prova
determinística por MOCK**, deixando IA-no-web e streaming como evoluções com gate
próprio. É o mesmo ritmo que fechou a Fase 1 (nativo-primeiro → gate → web).

**Ressalva honesta ao sign-off:** se a prioridade for **fonte única de provedores**
(CLI/TUI ganham Gemini) ou **paridade web imediata** de IA, então **D1=A** e/ou
**D2=A/C** são legítimos — ao custo de PR(s) ao `the-light` (ação humana) e/ou de
manter a ancoragem em TS com testes de paridade. O Planner **não** decide isso.

## 5. Consequência

- **Enquanto não houver decisão, o loop fica em HALT** (o Driver escreve `loop/HALT`
  com motivo "aguardando sign-off do gate F2.2").
- Decidido, registra-se o **ADR-0023** (próximo livre; o maior atual é ADR-0022) com
  D1–D4 + a evidência dos símbolos reais do `ai` (trait público, `ask` recebe `&dyn`,
  `PROVIDERS`/`default_model`/`estimate_cost_usd` sem gemini, `ai` `embedded`-only,
  fricção reqwest-no-web).
- O Driver então **re-escopa/semeia**: **F2.3** (Gemini per D1 — impl local *ou* spec
  de PR + HALT), **F2.4** (chave nativa via `expo-secure-store`), **F2.7** (web per D2
  — implementar *ou* "adiada e registrada"). **F2.5** (UI nativa) depende de F2.3+F2.4;
  **F2.6** (validação real, bloqueante-chave) de F2.5; **F2.8** (Marco 2) fecha a fase.
- Se **D1=A** ou **D2=C**, a spec de PR em `loop/proposals/` é aberta **antes** de
  F2.3/F2.7.

## Como liberar

Humano/Guia revisa o estado (`STATUS.md`; `DECISIONS.md` ADR-0005 = precedente de PR

## Restrições (não negociáveis — valem em QUALQUER opção)

- **Offline-first:** a IA é **opt-in e aditiva**; a **Fase 1 segue 100% offline** (sem
  IA, sem rede). A **única** rede em runtime é a chamada de IA, e só com a chave do
  usuário.
- **BYOK:** a chave é **do usuário**, **nunca** em git/log; trafega direto para
  `build_provider(name, key, model)` (a fronteira não persiste); nativo = secure-store;
  web = política de D3. O `KeyStore`/`secrets.toml` do core **não** é usado.
- **Anti-alucinação é LEI:** o texto do versículo vem **sempre do store local**
  (verbatim, domínio público); o LLM **só interpreta** — vale em **qualquer** opção,
  inclusive no web (se D2=A/C, a ancoragem em TS deve espelhar o Rust com testes de
  paridade).
- **`the-light` intocado** (`8f66004`): mudança no core (D1=A, D2=C) **só via PR +
  ADR** (ação humana; spec em `loop/proposals/`). O loop não toca o `the-light`.
- **Prova por MOCK, nunca o LLM real no CI/loop:** a chamada real (chave + rede) é a
  **F2.6** (bloqueante). Diante de conflito com qualquer regra acima: **HALT**.
- **Sem código de produto nesta tarefa.** Desfecho = **HALT p/ sign-off humano**.
