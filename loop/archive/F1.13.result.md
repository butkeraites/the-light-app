## Resumo

Paridade web de LEITURA entregue e verificada de forma independente. O
`app/web/reading.web.ts` deixou de ser stub para a leitura: `listBooks` vem do
RUST (wasm), e `listTranslations`/`getChapter`/`chapterCount` leem do subset
`reading-sample.sqlite` (~4,4 MB; o MESMO que o nativo empacota, ADR-0014) via
`wa-sqlite` (OPFS no browser / MemoryVFS na prova), ESPELHANDO os SELECTs de
`EmbeddedSource` do core — sem reimplementar domínio. A prova headless node passou
(exit 0) com texto VERBATIM do store (João 3:16 KJV + Almeida 1911), chapterCount
21, João 3 KJV = 36 versículos, listTranslations = [kjv, alm1911] e listBooks = 66
(via wasm). `tsc --noEmit` e `expo export --platform web` saem 0 com o subset
(4.567.040 B) + `wa-sqlite.wasm` no bundle. O caminho nativo, o core e o
`the-light` (8f66004) ficaram intocados. `search`/`crossRefs`/userdata seguem
stubs (F1.14–F1.16), conforme escopo.

## Decisões / ADRs

- **ADR-0019** (aceito, F1.13): store WEB de LEITURA A1 — espelho TS dos SELECTs de
  capítulo (`WholeChapter`)/`chapter_count`/`translations`/`has_translation` sobre o
  subset `reading-sample.sqlite` via `wa-sqlite` (OPFS no browser + MemoryVFS na
  prova; build SYNC, sem SharedArrayBuffer/COOP-COEP). `listBooks` vem do RUST
  (wasm) — uma única fonte da verdade do cânon; gate `useWasmReady()` no
  `_layout.tsx` (no-op no nativo) garante o wasm pronto antes de `listBooks()`
  síncrono. `db.web.ts::ensureReadingDb` devolve o sentinela `'web:reading-sample'`.
  Reusa ADR-0018 (Opção A/A1), ADR-0011/0012 (wa-sqlite+OPFS+MemoryVFS), ADR-0014
  (subset bundled), ADR-0010/0005 (gating por alvo / anti-alucinação).
