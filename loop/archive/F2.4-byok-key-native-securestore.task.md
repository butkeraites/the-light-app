## Objetivo

Entregar o serviço **app-side** de chave BYOK **nativo** — `app/lib/keystore.ts` —
sobre **`expo-secure-store`** (Keychain no iOS / Keystore no Android), para
**salvar / ler / apagar** a **API key** do usuário **por provedor**
(`anthropic`/`openai`/`gemini`/`ollama`), guardando-a **apenas** no cofre seguro do
dispositivo. A chave **nunca** é logada/impressa, **nunca** vai ao git, e só é lida
**sob demanda** para ser passada à fronteira (`ask_anchored(..., key)`) — quem a
**usa de fato é a F2.6** (chamada real). No **web** há apenas um **stub**
(`app/lib/keystore.web.ts`) sem chave persistida na Fase 2 (a política de chave web
é decidida na F2.7).

Esta é a **próxima tarefa não-bloqueante** após o gate F2.2 (ADR-0023, D3): é
**app-side puro** (não toca o `the-light`/`core`) e **testável SEM chave real** (com
um placeholder e um backend fake injetável), então o executor a roda no próximo
ciclo. O primeiro handoff bloqueante da Fase 2 é o **PR ao `the-light`** (F2.3:
Gemini + IA-wasm + streaming); a chave real é a **F2.6** (bloqueante).

**Decisão de escopo (justificada):** F2.4 entrega **serviço + teste**, **sem** UI de
device. A UI mínima de configuração de chave (colar 1×, deep-link p/ a página de key
do provedor) fica na **F2.5**, onde o painel de `ask` já traz o seletor de
provedor/modelo — acoplar a entrada da chave ao painel de IA é mais coeso, evita uma
tela órfã, e mantém a F2.4 como **infra determinística** provável de forma headless
(sem device, sem chave real). O `keystore.ts` da F2.4 é exatamente o que a UI da
F2.5 consome.

## Contexto / arquivos

- **Padrão de split nativo/web já estabelecido no app** (Metro escolhe `.web.ts` no
  web): `app/lib/db.ts` (nativo) + `app/lib/db.web.ts` (sentinela/stub). **Espelhe
  esse molde** com `app/lib/keystore.ts` (nativo) + `app/lib/keystore.web.ts` (stub).
- **`app/lib/`** já tem `db.ts`, `db.web.ts`, `theme.ts`, `userdata.ts`,
  `userdata.web.ts`, `notesExport.ts`, etc. — coloque o serviço de chave aqui.
- **Dependência a adicionar:** `expo-secure-store`, via **`npx expo install
  expo-secure-store`** (de `app/`), que resolve a versão alinhada ao **Expo SDK 56**
  (como `expo-asset`/`expo-file-system` estão `~56.0.x`). Rede só em dev/build.
- **API do `expo-secure-store`** (async): `SecureStore.setItemAsync(key, value)`,
  `getItemAsync(key) -> string | null`, `deleteItemAsync(key)`. No **web** o módulo
  não tem implementação nativa (Keychain/Keystore não existem no browser) → por isso
  o web é **stub** (não importe `expo-secure-store` no caminho web).
- **BYOK / fronteira:** a chave é **argumento** de `build_provider` (core; ver
  `providers.rs:39`), passada pela fronteira `ask_anchored(..., key: Option<String>)`
  (F2.1, `core/src/lib.rs`). A fronteira **não** persiste a chave; o
  `KeyStore`/`secrets.toml` do core **não** é usado. O `keystore.ts` é a **única**
  camada que toca o cofre do dispositivo.
- **Testes no app:** não há Jest configurado; o padrão de prova headless é **node
  `.mjs`** em `app/web/__tests__/` com backends fake/mock (ex.:
  `reading.web.test.mjs` usa wa-sqlite num MemoryVFS). Como `expo-secure-store` é
  módulo nativo (não roda em node puro), **injete o backend de storage** para provar
  a LÓGICA do serviço de forma determinística e headless (ver Passos).

## Passos sugeridos

1. **Adicionar a dependência:** de `app/`, `npx expo install expo-secure-store`
   (fixa a versão SDK-56 em `package.json`/lock).

2. **Serviço nativo `app/lib/keystore.ts`** — desenhe com um **backend injetável**
   para ser testável sem device:
   - Uma interface mínima `SecureBackend { getItemAsync(k), setItemAsync(k,v),
     deleteItemAsync(k) }` e uma **factory** que, por padrão, usa
     `expo-secure-store`, mas aceita um backend alternativo (fake) — ex.:
     `createKeystore(backend = defaultSecureBackend)`.
   - Constante de namespace de chave por provedor, ex.
     `keyIdFor(provider) => \`tla.apikey.${provider}\`` (nunca o valor no nome).
   - Provedores suportados: `['anthropic','openai','gemini','ollama']` (constante
     `SUPPORTED_PROVIDERS`; valide `provider` contra ela).
   - Funções (async): `setKey(provider, key)`, `getKey(provider): Promise<string |
     null>`, `deleteKey(provider): Promise<void>`, `listProviders():
     Promise<string[]>` (retorna **os nomes** dos provedores que **têm** chave —
     **nunca** os valores).
   - **Nunca** logar/imprimir a chave (sem `console.log`/`console.warn`/`console.*`
     com o valor da chave, nem em mensagens de erro). Trim de whitespace ao salvar;
     rejeitar chave vazia.

3. **Stub web `app/lib/keystore.web.ts`** (molde `db.web.ts`): implementa a mesma
   assinatura, **sem** importar `expo-secure-store`. Na Fase 2 o web **não** persiste
   chave: `getKey` → `null`, `listProviders` → `[]`, `setKey`/`deleteKey` no-op ou
   erro claro `'BYOK key não suportada no web na Fase 2 (F2.7)'` (decida e documente;
   não arraste `expo-secure-store` ao bundle web). Anote que a política de chave web
   é a **F2.7**.

4. **Teste headless determinístico** `app/web/__tests__/keystore.test.mjs` (node,
   **sem device, sem chave real**), usando um **backend fake em memória** (um `Map`)
   injetado no `createKeystore`:
   - Grava uma **key PLACEHOLDER** (ex. `'PLACEHOLDER_test_key_do_not_use'`) para
     `gemini` via `setKey`; **lê de volta** com `getKey` (== placeholder);
     `listProviders` inclui `'gemini'` e **não** contém o valor; `deleteKey` remove
     (`getKey` → `null`, `listProviders` sem `gemini`).
   - Round-trip por **múltiplos provedores** independentes (a key de um não vaza no
     outro).
   - **Invariante de log:** faça `console.log/…` serem espionados durante o teste e
     asserte que a **string da key NUNCA** aparece na saída de console; e um
     **grep** no fonte de `keystore.ts`/`keystore.web.ts` garante que não há
     `console.*` com a variável da chave.
   - Adicione um script `test:keystore` em `app/package.json` (`node
     app/web/__tests__/keystore.test.mjs`).

5. **Auditoria de não-versionamento:** garanta que a key placeholder **não** é
   escrita em nenhum arquivo versionado (o teste usa apenas o Map em memória); um
   `git grep` do placeholder no working tree após o teste deve dar **vazio** (fora do
   próprio arquivo de teste, onde é a string de asserção, não uma key real).

6. **Qualidade:** `tsc --noEmit` limpo em `app/`; `expo export --platform web` sai 0
   (o stub web compila sem `expo-secure-store` no bundle). Commit `feat(F2.4): ...`.

## Restrições (regras rígidas relevantes)

- **BYOK / chave nunca em git/log (LEI):** a chave vive **só** no secure-store do
  device (Keychain/Keystore); nunca é logada, impressa, serializada em arquivo, nem
  committada. O teste usa **placeholder** + backend fake em memória — **nenhuma**
  chave real. `listProviders` expõe **nomes**, nunca valores.
- **Secure-store só no nativo:** não importe `expo-secure-store` no caminho web (o
  Metro escolhe `keystore.web.ts`); o web **não** persiste chave na Fase 2 (política
  = F2.7). Não use `localStorage`/`IndexedDB` para a chave.
- **`the-light`/`core` intocados:** F2.4 é **app-side puro** — nenhuma mudança em
  `core/**` nem no repo `the-light`. A fronteira recebe a chave como **argumento**
  (F2.1); a chave real é **usada** só na F2.6.
- **Offline-first:** o secure-store é I/O local; **nenhuma** rede nesta tarefa. A
  Fase 1 segue 100% offline; a IA (com chave + rede) é opt-in e validada na F2.6.
- **Qualidade por tarefa:** `tsc --noEmit` limpo; teste headless verde; `expo export
  --platform web` 0. Verde antes do commit (prefixo `feat(F2.4): ...`). Diante de
  conflito com qualquer regra acima: **HALT**, não improviso.
