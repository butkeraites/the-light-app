## Objetivo

Dar **paridade web de LEITURA** ao app: navegar **livro → capítulo → texto**,
**trocar de versão** (KJV ⇄ Almeida 1911) e ver **duas versões lado a lado** —
tudo no **browser**, lendo do **store local** (zero rede em runtime), conforme a
decisão do gate F1.12 (**ADR-0018 — Opção A / A1**): `wa-sqlite` + OPFS
**espelhando em TS os mesmos SELECTs** que a fronteira nativa (F1.2) executa via
o `the-light-core`, sobre o **subset `reading-sample.sqlite` (~4,4 MB)** — o
**MESMO** subset que o nativo empacota hoje (ADR-0014), usando o **MemoryVFS já
provado na F0.10**. O **corpus completo (~59 MB)** com VFS-live em Worker é
**backlog transversal pós-paridade** — **fora desta task**.

Concretamente: tornar `app/web/reading.web.ts` (hoje **stubs** que lançam) uma
implementação **real** das funções de **leitura** (`listTranslations`,
`getChapter`, `chapterCount`) espelhando os SELECTs do core, e `listBooks`
delegando ao **Rust (wasm)** já gerado — de modo que as **mesmas telas React**
de `app/app/read/**` (compartilhadas com o nativo) passem a **funcionar no web**
pelo glue. Busca/xref/notas web **permanecem stubs** (F1.14/F1.15/F1.16).

A entrega visual é a leitura no browser; a **PROVA determinística** é um
**teste headless node** (molde F0.10 — OPFS é browser-only) que carrega os
**bytes do `reading-sample.sqlite`** num **MemoryVFS** e chama as **funções web
reais**, assertando **texto VERBATIM do store** + contagens — e documentando que
esses números/textos são **os MESMOS** que o nativo prova via `TLA_READ`/
`TLA_PARALLEL` (F1.3/F1.4).

**Os SELECTs/semântica REAIS do core que o web deve ESPELHAR** (confirmados na
fonte `~/.cargo/git/checkouts/the-light-9eb8809a6d68281a/8f66004/crates/the-light-core/src/source/embedded.rs`,
leitura-only; a fronteira F1.2 em `core/src/lib.rs` apenas **delega** a estes):

- **`get_chapter`** → `EmbeddedSource::passage` (variante **`WholeChapter`**, pois
  a fronteira usa `Reference::whole_chapter(book, chapter)`):
  ```sql
  SELECT verse, text FROM verses
   WHERE translation_id = ?1 AND book_number = ?2 AND chapter = ?3
   ORDER BY verse
  ```
  (binds: `translation_id`, `book`, `chapter`). O core **primeiro** checa
  `has_translation` (`SELECT 1 FROM translations WHERE id = ?1`) e, se ausente,
  retorna `UnknownTranslation` — **espelhe** essa checagem (translation inexistente
  → erro/`Passage` consistente com o nativo; o subset tem `kjv` e `alm1911`).
  Compõe `Passage { reference: WholeChapter(book,chapter), verses: [ Verse {
  reference: Single(book,chapter,verse), text, translation } ] }` (cada `Verse`
  com referência **`Single`**, o `Passage` com referência **`WholeChapter`**).
- **`chapter_count`** → `EmbeddedSource::chapter_count`:
  ```sql
  SELECT max(chapter) FROM verses WHERE translation_id = ?1 AND book_number = ?2
  ```
  `NULL` → **0** (livro/tradução ausente). DB-backed (≠ o `chapter_count` canônico
  de `Book`).
- **`list_translations`** → `EmbeddedSource::translations`:
  ```sql
  SELECT id, abbrev, name, language, license, embeddable
    FROM translations ORDER BY language, id
  ```
  Mapeia cada linha → `Translation { id, abbrev, name, language, license,
  embeddable: <int>!=0 }`. No subset, `language` já é `"en"`/`"pt"` e `license`
  `"public-domain"` (casam com `Lang::code()`/`License::as_str()` do Record da
  fronteira — **sem drift**). Ordem: en (`kjv`) antes de pt (`alm1911`).
- **`list_books`** é **PURO** no nativo (`reference::BOOKS`, 66 livros) e **JÁ
  está exportado nos bindings WEB gerados** (`app/web/generated/the_light_app_core.ts`
  exporta `listBooks(): Array<Book>` via wasm — checksum 43013). → No web,
  **delegar ao wasm** (mesmo padrão de `parseReference` em `reference.web.ts`),
  **NÃO** reimplementar a lista de 66 à mão nem ler a tabela `books` (a fronteira
  nem usa a tabela `books`).
