## Resumo

F5.14 torna o override de modo de tema (claro/escuro) PERSISTENTE entre reinícios,
reusando o MESMO KV de prefs OFFLINE da F5.2 (`app/lib/prefs.ts` — arquivo JSON nativo /
`localStorage` web), sob a chave namespaceada `tla.pref.theme.mode`. Fecha a lacuna
"persistência entre reinícios é futura" do ADR-0015, SEM criar um 2º mecanismo de
persistência e SEM dependência nova.

A lógica pura de persistência (tipo `ThemeMode`, `THEME_MODES`, `THEME_PREF_KEY`, guard
`isThemeMode`) vive em `app/lib/themePrefs.ts` (novo, dependency-free, sem `react-native`),
o que mantém a prova headless bundlável em node. O `ThemeProvider` em `app/lib/theme.ts`
hidrata no boot (`getPref` + `isThemeMode`) e persiste on-change (`setPref`/`removePref`),
re-exportando `ThemeMode` para não quebrar os ~20 consumidores. Verifiquei independentemente
o diff e RE-RODEI todos os gates — todos verdes.

## Critério de aceite

- [x] **Alternar o tema e reabrir o app mantém o modo escolhido (persistido via prefs offline).**
  — `theme.ts`: `setMode` grava via `setPref('theme.mode', next)` (ou `removePref` p/ `null`);
  o `useEffect` de boot lê `getPref('theme.mode')` e aplica se `isThemeMode(saved)`. Fire-and-forget,
  falha tolerada (offline-first). Provado pelo headless: `setPref('dark')` SOBREVIVE a uma NOVA
  instância de `Prefs` sobre o MESMO storage (simula reabrir) e re-hidrata como `ThemeMode` válido.

- [x] **Sem override salvo, o app segue o esquema do sistema (comportamento F1.4 preservado).**
  — `mode = override ?? (system === 'dark' ? 'dark' : 'light')` mantém `useColorScheme()` como base
  (ADR-0015). Estado inicial `override=null`; `removePref` volta a seguir o sistema; valor
  desconhecido/corrompido é IGNORADO pelo guard `isThemeMode`. Test assevera `removePref → null`
  e `isThemeMode('sepia') === false` (segue o sistema).

- [x] **Round-trip de 'theme.mode' provado por headless (backend injetável em memória); nenhuma
  dependência nova.** — `web/__tests__/theme.test.mjs` (+ `theme-headless-entry.ts`) bundla

  simétrico e higiene de log. Diff do `package.json` adiciona SÓ o script `test:web:theme` — nenhuma
  linha de dependência (confirmado); `prefs.ts`/`prefs.web.ts` NÃO foram tocados (reuso puro).

- [x] **tsc 0, eslint 0, expo export web exit 0; core intacto.** — `tsc --noEmit` exit 0; `expo export

## Decisões / ADRs

- **ADR-0043** (novo, aceito, 2026-07-03): persistir o modo de tema no KV de prefs OFFLINE da F5.2
  sob `tla.pref.theme.mode`, com lógica pura em `themePrefs.ts` e hidratação/persistência no
  `ThemeProvider` (molde `I18nProvider`). Número CONFIRMADO livre — é o `## ADR-` mais alto do arquivo,
  logo após ADR-0042; só 1 ocorrência de `## ADR-0043`.
- **ADR-0015 (amendment)**: nota adicionada registrando que a lacuna "persistência entre reinícios é
  futura" está FECHADA, remetendo ao ADR-0043. Amendment CONFERIDO e preciso (aponta o reuso da infra
  F5.2, sem `AsyncStorage`/`expo-secure-store` nem dep nova).
- **Boot-flash (trade-off aceitável, não defeito):** a hidratação é assíncrona (KV nativo/web é async),
  então o 1º paint pode seguir o esquema do sistema por um instante até a leitura resolver — idêntico ao
  `I18nProvider` (F5.2) e explicitamente documentado no ADR-0043 e no amendment ao ADR-0015. Uma leitura
  síncrona acoplaria `theme.ts` ao `localStorage` e quebraria a paridade nativa; a escolha documentada é
  razoável.
