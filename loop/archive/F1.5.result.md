## Resumo

F1.5 expõe na fronteira UniFFI a busca full-text (FTS5)
`search(db_path, query, translation, book: Option<u8>, limit: Option<u32>)
-> Result<Vec<SearchHit>, CoreError>`, **delegando** a
`the_light_core::source::BibleSource::search` (via `EmbeddedSource`, montando
`the_light_core::search::SearchOptions` a partir de `translation`/`book`/`limit`).
Nenhum SQL/FTS é reimplementado na fronteira: o índice acento-insensível
(`remove_diacritics 2`), o ranking BM25 e o destaque vivem no core. Novo Record
`SearchHit { reference, translation, text, highlighted, score }` definido em
**todos** os alvos, construído **somente** via `From<model::SearchHit>` (referencia
só `model`, puro). Mesmo molde da F0.9/F1.2/ADR-0010: corpo DB sob
`#[cfg(not(target_arch = "wasm32"))]` + stub web sob `#[cfg(target_arch="wasm32")]`
retornando `CoreError` (sem arrastar `search`/`source`/`store`/`rusqlite` ao grafo
wasm). Teste de host `search_tests` (6 testes; fixture offline determinístico que
**popula `verses_fts` explicitamente** — sem trigger no schema do core). Total: 20
testes. Verificação independente (blocos 1–5) toda verde; `the-light` intacto
(8f66004); forma da fronteira preservada (só adições). **Decisão: passed.**

## Decisões / ADRs

- **Sem ADR novo.** Reusa **ADR-0005** (matriz de features por alvo: `embedded`
  só no nativo → `rusqlite`/`search`/`source`/`store` fora do grafo wasm) e
  **ADR-0010** (store nativo + **gating de corpo** com stub web + anti-alucinação:
  texto vem sempre do store). Mesmo molde de F0.9/F1.2.
- **Limpeza mecânica** em `core/examples/gen_reading_sample_db.rs`
  (`iter_nth_zero` + rustfmt) para destravar os gates `fmt`/`clippy` do crate —
  pré-existente, sem mudança de lógica (revisado no diff). Aceitável.
- `Cargo.toml` **não** precisou mudar: a matriz por alvo já liga `embedded` só no
  não-wasm; `search`/`source`/`store` do core existem só no nativo.
