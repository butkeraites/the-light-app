## Objetivo

Dar **paridade web de NOTAS/MARCAÇÕES (userdata) + EXPORT**, fechando a Fase 1 no
browser. Hoje as **7 funções** de userdata em `app/web/reading.web.ts`
(`putNote`/`getNote`/`deleteNote`/`listNotes` + `addHighlight`/`removeHighlight`/
`listHighlights`) são **stubs** que lançam `WEB_NOTES_MSG`, e
`app/lib/userdata.web.ts::ensureUserDataDir()` **lança** em runtime.

Esta tarefa **destuba** as 7 funções, persistindo os dados do usuário em **OPFS**
(o MESMO backend já usado pela leitura/busca/xref web) num **`data_dir` web**
análogo ao nativo — `notes/<slug>.md` (um arquivo por referência) + um único
`highlights.json` — **ESPELHANDO byte-a-byte o formato em disco do core**
(`the_light_core::userdata`, rev pinado `8f66004`), para o **export ser idêntico e
portável** entre nativo e web. O **EXPORT** reusa `app/lib/notesExport.ts`
(agregado puro dos Records — já existe), produzindo o MESMO agregado que o nativo.

As MESMAS telas React compartilhadas (`app/app/read/[book]/[chapter].tsx` +
`app/components/ReaderVersePanel.tsx`) passam a funcionar no browser **só por este
glue** + o sentinela de `userdata.web.ts`; a persistência **sobrevive a reload**
(OPFS). O caminho NATIVO de userdata (F1.10/F1.11) **não** é tocado.

**É a ÚLTIMA paridade web da Fase 1.** Após F1.16 aceita, **F1.17 (Marco 1,
`gate: true`) fica elegível → o loop PARA (HALT) para sign-off humano** (leitura
offline completa multiplataforma).

## Decisão de arquitetura que esta tarefa FIXA (registrar em ADR-0022)

O módulo `userdata` do `the-light-core` é **`#[cfg(feature = "embedded")]`
(nativo-only)** → **NÃO entra no grafo wasm** (matriz por alvo, ADR-0005/0010). Logo
o web **NÃO pode delegar** a ele — exatamente como o store de leitura/busca/xref
(ADR-0011/0019/0020/0021), em que a query/SQL do core foi **espelhada em TS como
INFRAESTRUTURA**, não como lógica de domínio.

Portanto a F1.16 **reimplementa o I/O de userdata em TS no web**, espelhando o
**formato em disco** do core (mesmo slug de arquivo, mesmo `.md`, mesmo schema do
`highlights.json`). **Isto é infra de armazenamento, NÃO lógica de domínio:**
- O **corpo** da nota é **dado livre do usuário** → anti-alucinação **não se aplica**
  ao corpo (igual ao nativo, ADR-0017).
- A **referência é canonicalizada pelo WASM** (`parseReference`), **não inventada em
  TS** — em AMBAS as direções: ao **gravar**, o slug/`ref` deriva da `Reference`
  resolvida pelo wasm + o **nome EN do livro** vindo do wasm `listBooks()`; ao
  **ler**, o nome do arquivo / a string `ref` é re-analisada por `parseReference`
  (wasm). A única coisa "de formato" em TS é a **convenção de nomenclatura de
  arquivo** (a montagem da string `{nome} {cap}:{verso}` + a troca `' '→'_'`,
  `':'→'.'`), que é **format de armazenamento** — o mesmo nível de "infra" de
  espelhar um `SELECT`.
- Persistência web = **OPFS** (mesmo backend das F1.13/F1.14/F1.15, build SYNC, sem
  SharedArrayBuffer/COOP-COEP), num `data_dir` web **SEPARADO** do conteúdo público
  só-leitura (`reading-sample.sqlite`) e do `sample.sqlite` (F0.10).

Objetivo central = **paridade de FORMATO** (export idêntico/portável: o `.md`/JSON
que o web grava o **core lê** e vice-versa). É o precedente ADR-0011 aplicado a
userdata.

> **Guarda de viabilidade:** se a investigação na implementação revelar que espelhar
> o formato é **inviável sem tocar o core/fronteira** (ex.: precisaria expor uma
> nova função `format_reference` na fronteira `core/src/lib.rs`, ou o formato fosse
> serialização proprietária acoplada), então **`blocked` + HALT** com o achado
> EXATO (núcleo/fronteira só via PR + ADR). **O esperado é viável** — o formato é
> aberto (`.md` + JSON) e a referência/cânon já vêm do wasm.

## Contexto / arquivos

### O FORMATO REAL do userdata do core (confirmado na FONTE, rev `8f66004`)

Fonte só-leitura: `~/.cargo/git/checkouts/the-light-9eb8809a6d68281a/8f66004/crates/the-light-core/src/userdata/{notes.rs,highlights.rs}`
e `…/src/reference.rs::format_reference`.

**NOTAS — `notes/<slug>.md` (um arquivo por referência):**
- **Slug (nome do arquivo):** `format_reference(reference, Lang::En)` com
  `.replace(' ', "_").replace(':', ".")` + sufixo `.md`. Exemplos:
  - `John 3:16` (Single) → **`John_3.16.md`**
  - `Genesis 1:1-3` (Range) → **`Genesis_1.1-3.md`** (o `-` é preservado)
  - `Psalms 23` (WholeChapter) → **`Psalms_23.md`**
- **`format_reference(_, Lang::En)`** (a montar em TS, espelhando `reference.rs`):
  - `WholeChapter` → `"{name_en} {chapter}"`
  - `Single(v)`   → `"{name_en} {chapter}:{v}"`
  - `Range{a,b}`  → `"{name_en} {chapter}:{a}-{b}"`
  - `name_en` = nome inglês do livro (**de `listBooks().nameEn`**, casado por
    `number === reference.book`).
- **Conteúdo do arquivo = o BODY puro** (Markdown) — **SEM front-matter, SEM
  título** (`put` grava `body.as_bytes()`; `get` devolve o arquivo inteiro como
  `body`).
- **`list_notes` (ORDENADO):** lê os `.md` de `notes/`, **ignora** não-`.md` e nomes
  não-parseáveis, e **ordena por `(book, chapter, verses.start)`** (referência
  canônica). Diretório ausente → lista vazia (sem erro).
- **`delete_note`:** remove o arquivo → `true` se existia, `false` se não
  (idempotente). **`get_note`:** ausente → `undefined`.
- **Read-back da referência (uma fonte da verdade):** o core faz
  `parse_reference(stem.replace('_', " "))` — ou seja, **NÃO** desfaz o `.`→`:`; o
  `parseReference` aceita `.` como separador cap.verso (mesmo motivo de "Jo 3.16"
  funcionar). **No web, ler-de-volta = `parseReference(stem.replace(/_/g, ' '))`
  via WASM.**

**HIGHLIGHTS — um único `highlights.json`:**
- **Schema:** um **array JSON** de objetos
  `{ "ref": "<ref EN>", "color": "<str>", "tag"?: "<str>" }`.
  - `ref` = `format_reference(reference, Lang::En)` — **a string legível SEM a troca
    `_`/`.`** (ex.: `"John 3:16"`, `"Genesis 1:1-3"`, `"Psalms 23"`). (Distinto do
    slug de nota, que troca `' '→'_'`, `':'→'.'`.)
  - `tag` é **OMITIDO** quando ausente (`skip_serializing_if = Option::is_none`).
  - Serializado com `serde_json::to_string_pretty` → **2 espaços de indentação**,
    ordem de chaves `ref`, `color`, `tag`. (No web: `JSON.stringify(dtos, null, 2)`
    com objeto `{ ref, color, ...(tag != null ? { tag } : {}) }`.)
- **`add_highlight`:** **substitui** a entrada de MESMA referência (remove a antiga,
  faz push) → a ordem é **de INSERÇÃO** (≠ notas, que ordenam por referência).
- **`list_highlights`:** devolve na **ordem do array** (inserção) — **NÃO** ordenado.
- **`remove_highlight`:** remove todas as de uma referência; devolve a contagem.
- **`load`:** arquivo ausente → vazio; entradas com `ref` inválida são ignoradas.
- **Read-back da referência:** `parse_reference(dto.ref)` → no web,
  `parseReference(dto.ref)` via WASM.

## Restrições (regras rígidas relevantes)

- **NÃO tocar `the-light`/fronteira Rust nem o caminho NATIVO.** A F1.10/F1.11 já
  entregaram as 7 funções e a UI/persistência nativas; o web **reimplementa o I/O
  como INFRA espelhando o formato** (precedente ADR-0011/0019/0020/0021). Se algo
  exigir mudar o core OU a fronteira `core/src/lib.rs` (ex.: expor `format_reference`)
  → **`blocked` + HALT** com o erro EXATO (núcleo/fronteira só via PR + ADR).
- **Userdata web SEPARADO do conteúdo público só-leitura** (`reading-sample.sqlite`)
  e do `sample.sqlite` (F0.10): diretório OPFS próprio, gravável.
- **Referência canonicalizada pelo WASM** (`parseReference`), **não inventada em
  TS**, nas duas direções (gravar e ler). O nome EN do livro vem de `listBooks()`
  (wasm). A única "format" em TS é a convenção de nome de arquivo/`ref` (infra).
- **Export = agregado dos Records** (`buildNotesExport`, já existe) — **NÃO**
  reescrever a serialização de domínio (nada de gerar `.md`/`highlights.json` à mão
  para o export). O export web produz o **MESMO** agregado que o nativo.
- **Anti-alucinação:** **não se aplica ao corpo** da nota/`tag` (dado do usuário); a
  **referência** é canônica (wasm). Nenhum texto bíblico do banco vaza pelo export.
