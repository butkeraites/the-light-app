## Resumo

O **caminho web do `ubrn` EXISTE e funciona até a geração**: `ubrn build web`
gera os bindings web + o wasm-crate wrapper. O muro **não** é o caminho web nem o
obstáculo `+Send` do uniffi-wasm (este é resolvível pela feature
`wasm-unstable-single-threaded`, presente tanto no crate `uniffi 0.31.2` quanto no
runtime web do ubrn). O muro é um **conflito de versão de `uniffi_core`** ao
compilar o wasm-crate para `wasm32`:

- nossa fronteira `core/Cargo.toml` declara `uniffi = "0.31.2"`, e o crate
  `uniffi 0.31.2` **fixa exatamente** `uniffi_core = "=0.31.2"`;
- o runtime web do ubrn `uniffi-runtime-javascript` **fixa exatamente**
  `uniffi_core = "=0.31.0"`.

Dois pins exatos (`=0.31.2` e `=0.31.0`) sobre o **mesmo** crate `uniffi_core` no
mesmo grafo de build são irreconciliáveis → `failed to select a version for
uniffi_core`. Verificado de forma independente lendo os manifests reais (sem
re-rodar o caminho web de ~14 min) — a resolução do cargo é determinística.

A correção é **mecânica e de baixo risco** e está **em escopo do loop**: alinhar a
`uniffi` da NOSSA fronteira a `=0.31.0` (a versão que a toolchain ubrn 0.31.0-3
fixa). Por isso o desfecho é **`failed`** (corrigível pelo loop), **não**
`blocked` (não falta decisão estratégica humana, credencial nem ferramenta, e
**não** se toca o `the-light`).

### Raciocínio do conflito (determinístico — não precisa compilar)

No grafo de build do wasm-crate gerado pelo `ubrn build web` convivem:

- via nossa fronteira → `uniffi 0.31.2` → **exige** `uniffi_core =0.31.2`;
- via `uniffi-runtime-javascript` (runtime web do ubrn) → **exige** `uniffi_core =0.31.0`.

Cargo precisa de **um único** `uniffi_core` que satisfaça `=0.31.2` **E** `=0.31.0`
ao mesmo tempo. Impossível → `failed to select a version for uniffi_core`. Ambos
estão em 0.31.x, mas por serem **pins exatos divergentes**, não há união. O muro é
o pin de versão, não o `+Send` (linha 18 acima mostra que a feature wasm já cuida
do single-threaded).

## Decisões / ADRs

**Correção proposta (em escopo do loop, sem tocar `the-light`):** alinhar a
`uniffi` da NOSSA fronteira de `0.31.2` para `=0.31.0` — a versão que a toolchain
ubrn 0.31.0-3 fixa em todo o subsistema (bindgen e runtime web). Quando a fronteira
usar `uniffi 0.31.0`, este crate fixará `uniffi_core =0.31.0`, coincidindo com o
pin do runtime web → resolução única → o caminho web pode compilar p/ wasm32.

Por que é mecânico e de baixo risco (e portanto `failed`, não `blocked`):

- É mudança no **`the-light-app`** (`core/Cargo.toml`), **NÃO** no `the-light`.
  Não há PR/ADR no core externo, não há ação humana obrigatória.
- É um **downgrade de patch dentro de 0.31.x** (mesmo *contract version* do
  UniFFI). O ADR-0003 já prevê que "subir/trocar a versão do `uniffi` é trocar a
  linha + `Cargo.lock` num commit/ADR — auditável".
- O ADR-0004 já registrou empiricamente que o **bindgen 0.31.0 do ubrn** lê a
  metadata da cdylib **0.31.2 sem erro de contrato/checksum** e gera os tipos
  corretos. Alinhar o runtime/fronteira a 0.31.0 só **aperta** essa consistência —
  não introduz um salto de minor nem mudança de API/forma da fronteira.

**Roteamento:** criar tarefa precursora **F0.6c** ("Alinhar `uniffi` da fronteira
a `=0.31.0` para destravar o caminho web do ubrn") que: (1) troca
`uniffi = "0.31.2"` → `uniffi = "=0.31.0"` no `core/Cargo.toml`; (2) regenera
`core/Cargo.lock`; (3) re-roda as verificações verdes de **F0.3**
(`parse_reference`: fmt/clippy -D warnings/test) e **F0.4** (`gen-bindings.sh` jsi
sai 0 e popula `bindings/`) **sob 0.31.0**; (4) escreve **ADR-0006** registrando o
re-pin e o motivo (conflito `uniffi_core` `=0.31.2` vs `=0.31.0` no caminho web do
ubrn). Depois → **retry F0.6b** (agora com a fronteira em 0.31.0). ADR-0006 fica a
cargo da F0.6c.

## Bloqueios / perguntas

**Nenhum bloqueio humano.** Não falta decisão estratégica, credencial, segredo nem
ferramenta; não se modifica o `the-light`. A correção é o "ajustar à realidade do
ecossistema + ADR" sancionado pelo IMPLEMENTATION_PLAN, inteiramente dentro do
`the-light-app`.

**Risco a registrar (baixo, mitigável na F0.6c):** o re-pin 0.31.2 → 0.31.0 exige
**revalidar F0.3 e F0.4 sob 0.31.0**. Mitigação: as três condições acima
(mesmo contract version; ADR-0004 já provou geração com bindgen 0.31.0; mudança de
patch sem alterar a forma da fronteira) tornam a quebra improvável. A F0.6c deve
**provar** com fmt/clippy/test (F0.3) e `gen-bindings.sh` verde (F0.4) — se algo
quebrar de fato sob 0.31.0, aí sim reavaliar como bloqueio.
