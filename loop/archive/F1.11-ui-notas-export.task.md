## Objetivo

Entregar a **UI de notas e marcações (highlights) NATIVA** (iOS/Android) ancorada
no Reader, com **persistência no device** e **export** dos dados do usuário —
**lendo/gravando SEMPRE pela fronteira `userdata` da F1.10** (Turbo Module → JSI →
`the_light_core::userdata::{NoteStore,HighlightStore}`). Ao **selecionar um
versículo** na tela `read/[book]/[chapter]` (reusando o gesto/seleção introduzido
na F1.9 p/ o painel de xref), o usuário pode:

- **criar / editar / remover uma NOTA** (corpo em texto/Markdown — texto livre do
  usuário) na referência selecionada (`put_note` / `get_note` / `delete_note`); e
- **marcar / desmarcar um HIGHLIGHT** (escolha de **cor**; `tag` opcional) na
  referência (`add_highlight` / `remove_highlight`); e
- ver um **indicador visual** nos versículos que têm nota e/ou highlight (ex.: cor
  de fundo do highlight; ícone/realce de nota), derivado de `list_notes` /
  `list_highlights`.

As notas/highlights **persistem** num **`data_dir` gravável e ESTÁVEL** sob
`FileSystem.documentDirectory` (ex.: `${documentDirectory}userdata/`), **distinto**
do `reading-sample.sqlite`/conteúdo público (que é só-leitura). Sobrevivem a
restart do app — a **prova relê de um handle independente** (cada chamada da
fronteira abre um novo `NoteStore`/`HighlightStore` a partir do disco). Há um
**EXPORT** portável dos dados do usuário (formato mais simples que **não
reimplemente** serialização do core).

**Nada de I/O de arquivo, serialização (`.md`/`highlights.json`), slug de
referência ou ordenação de userdata é reimplementado em TS** — a UI só chama a
fronteira e apresenta os Records `Note`/`Highlight` retornados (uma fonte da
verdade). **Anti-alucinação:** o **corpo** da nota é dado livre do usuário (não se
aplica); a **referência** é canônica via a fronteira (`parse_reference` no core).

Escopo: **só nativo** nesta tarefa. A **paridade web de notas/highlights é a
F1.16** (depois do gate estratégico F1.12) — não construir o caminho web aqui
(apenas manter `tsc`/Metro web verdes: o glue web de notas continua um stub que
lança em runtime).

A entrega visual é a UI (editor de nota + controles de highlight + indicadores +
export); a **PROVA determinística** é um **self-test de notas no device** (molde
F1.9/F1.3) que exercita a fronteira REAL e emite um marcador estável composto do
**retorno real** das funções.

## Restrições

- **NÃO modificar/forkar `the-light`** nem a fronteira `core/src/lib.rs`/
  `core/Cargo.toml`. A F1.10 já entregou as **7 funções** de notas/highlights;
  **nenhuma função nova de fronteira é necessária**. Notas/highlights são
  **delegados** a `the_light_core::userdata::{NoteStore,HighlightStore}` via essas
  funções. Se a UI exigisse mudar o core/fronteira → **`blocked`/HALT** (PR + ADR é
  ação humana), com o erro EXATO.
- **NÃO reimplementar I/O/serialização/ordenação de userdata no app** — a UI **só**
  chama a fronteira e apresenta os Records `Note`/`Highlight`. Proibido em TS de
  produção: escrever/parsear `.md` de nota à mão, escrever/parsear `highlights.json`
  à mão, computar slug de referência, ordenar a lista de notas à mão, ou instanciar
  `NoteStore`/`HighlightStore` em TS. (O EXPORT pode **apresentar** os Records de
  `list_notes`/`list_highlights` ou compartilhar os arquivos que o **core** já
  gerou no `data_dir` — isso não é reimplementar serialização.)
- **`data_dir` de userdata gravável e SEPARADO do conteúdo público:** as
  notas/highlights vivem em `${documentDirectory}userdata/` (gravável), **nunca** no
  `bible.sqlite`/`reading-sample.sqlite` (só-leitura, conteúdo de domínio público).
  Passar o caminho errado quebra a separação — manter os dois serviços distintos
  (`ensureReadingDb()` vs `ensureUserDataDir()`).
- **Anti-alucinação:** o **corpo** da nota é **texto livre do usuário** (a
  anti-alucinação **não** se aplica ao corpo); a **referência** é canônica via a
  fronteira (`parse_reference` no core). No self-test, `note_ref`/`note_len`/
  `highlights`/`persisted` provêm do **retorno** de `list_notes`/`get_note`/
  `list_highlights`; a asserção do script usa padrões esperados (ref "John 3:16",
  `highlights≥1`, `persisted=true`), não a fonte.
- **Export não vaza nada além dos dados do próprio usuário:** só as notas/highlights
  do `data_dir`/Records do usuário entram no export — nenhum conteúdo bíblico do
  banco, nenhum dado de outro escopo, nenhum segredo.
- **Offline-first:** **zero rede em runtime** (userdata é I/O local no device, via a
  fronteira → core; export é local/Share sheet); rede só em dev/build (gen-*-db,
  gen-bindings, Metro, Gradle, pods, npm). **Sem segredos** em git/log.
- **Só nativo:** a paridade web de notas/highlights é a **F1.16** (pós-gate F1.12).
  Manter o glue web de userdata como **stub** (lança em runtime) e o **build web
  verde** (sem quebra de import/build). Não construir o caminho web aqui.
- **Dependências com parcimônia:** `expo-asset`/`expo-file-system` já estão em
  `app/package.json`. Adicione **`expo-sharing` SÓ** se for usar o Share sheet do
  export; não inventar dependência pesada. Se adicionada, registrar no ADR-0017.
- **Gerados/bundled/userdata-runtime ignorados:** `bible.sqlite`,
  `reading-sample.sqlite`, asset/userdata copiados em runtime, `app/ios`,
  `app/android`, `app/web/native-generated`, jniLibs/`.so` permanecem no
  `.gitignore`; working tree limpo após o commit. Apenas os arquivos **versionados**
  pertinentes entram: glue (`reading.ts`/`reading.web.ts`), serviço
  (`userdata.ts`/`.web.ts`), componentes (`ReaderNoteEditor`/
  `ReaderHighlightControls` e/ou painel estendido), `ReaderChapterView.tsx`,
  `[chapter].tsx`, self-test (`notes-selftest.ts`/`.web.ts`, `selftest.ts`),
  `run-ios-selftest.sh`, `package.json` (se +expo-sharing), `DECISIONS.md`
  (ADR-0017).
- **ÚLTIMA TAREFA ANTES DO GATE F1.12:** após a aceitação da F1.11, o Driver semeia
  a **F1.12** (`gate: true`) → o loop **PARA (HALT)** p/ **sign-off humano**
  (decisão estratégica do store web do corpus completo). Não antecipar a F1.12 nem
  o caminho web aqui.
- **BLOQUEIO (`blocked` + HALT) com erro EXATO** se: a UI de notas exigir mudar o
  core/fronteira; OU a gravação em `documentDirectory/userdata/` no Expo nativo for
  inviável sem hack proibido (ex.: a fronteira não conseguir abrir/gravar o
  `data_dir`); OU o export exigir reimplementar serialização do core. Reportar a
  saída exata e propor a decisão/ADR; não improvisar.
