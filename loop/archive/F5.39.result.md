## Resumo

Fix da causa raiz REAL da leitura web quebrada: o glue Emscripten do wa-sqlite
avaliava `new URL("wa-sqlite.wasm", import.meta.url)` e, sob o bundler DEV do
Metro, `import.meta.url` é a string `"null"` → `Failed to construct 'URL':
Invalid base URL` ao abrir o SQLite. A leitura web SEMPRE esteve quebrada no
browser; só passava nos testes headless (Node, onde `import.meta.url` é `file://`
válido). O fix passa `locateFile: (path) => path` ao factory nos DOIS stores de
produção, desviando do branch inválido `new URL(...)` (os bytes do wasm já vêm
por `wasmBinary`).

**Verificado no BROWSER REAL (Chrome do sistema via puppeteer-core):** Mateus 1
(`/read/40/1`) renderiza o texto KJV real. Sem `Invalid base URL` de
`openReadingDbWeb`/`SQLiteESMFactory`. Todos os gates headless verdes. VERDICT:
PASSED.

## Verificação da causa (glue ternary)

`app/web/vendor/wa-sqlite-fts5/wa-sqlite.mjs:89`:
```
Ca ||= f.locateFile ? (...) : (new URL("wa-sqlite.wasm", import.meta.url)).href
```
Com `f.locateFile` truthy, o ternário toma o primeiro ramo e NUNCA avalia
`new URL(import.meta.url)`. Com `locateFile` ausente (arquivos de teste), toma o
ramo do `new URL` — que lança no Metro DEV. O fix é correto e mínimo.
