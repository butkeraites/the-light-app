## Resumo

F6.4 adiciona um GUARD de staleness por versão/hash ao `ensureReadingDb`
(`app/lib/db.ts`) para que quem ATUALIZA o app adote o DB de leitura novo em vez de
manter uma cópia velha ("Mateus 404 p/ quem atualiza"). A identidade do asset é
`${DB_VERSION}:${Asset.hash}` (MD5 do conteúdo, muda deterministicamente quando o DB
muda) versus um sidecar `reading-sample.sqlite.version`; em mismatch (ou sidecar
ausente = instalação pré-F6.4) re-copia, gravando o sidecar SÓ APÓS a cópia OK. Um
self-test de upgrade (`reading-selftest.ts`) pré-semeia um DB stale + sidecar velho e
prova `TLA_DBUP adopted=true` a partir do retorno REAL de `get_chapter(Matthew/40)`.
Verifiquei os 6 pontos por inspeção adversária de código + re-rodei os gates headless.
Verdito: PASSED.

## Critério de aceite (como foi satisfeito)

- [x] **Re-copia em mismatch; copia em instalação limpa; userdata nunca apagado.**
  Em `ensureReadingDb`, a condição de cópia é `!info.exists || localVersion !== expectedVersion`.
  Fresh install (`!info.exists`) → copia + sidecar. Sidecar ausente (pré-F6.4) →
  `readLocalVersion` retorna `null` → mismatch → re-copia. Sidecar diferente → re-copia.
  Match → reusa, sem cópia. O laço de limpeza toca APENAS
  `reading-sample.sqlite`(+`-wal/-shm/-journal`) e o sidecar `.version`, todos direto no
  `documentDirectory`; o userdata vive em subdir SEPARADO `${documentDirectory}userdata/`
  (`app/lib/userdata.ts`) — nunca intersectado.
- [x] **Self-test pré-semeia stale → `TLA_DBUP adopted=true` (Mateus consultável).**
  `runReadingDbUpgradeSelfTest` reseta a memoização (`__resetReadingDbCacheForTest`),
  escreve bytes SQLite inválidos em `destUri` + sidecar `'tla-stale-version-0'`, chama
  `ensureReadingDb` (força a re-cópia) e deriva `adopted = matt1.verses.length > 0` do
  retorno REAL de `getChapter(kjv,40,1)`. Não é hardcoded.
- [x] **tsc 0; nenhum web/headless regride.** Ver saída abaixo.
- [x] **Prova on-device:** `TLA_DBUP adopted=true matt1_verses=25 matt1_v1_len=81`
  reportado pelo executor no simulador iOS. Triangulei com dados web independentes
  (abaixo) — consistente e não-falsificável.

## Achados da inspeção (6 checks)

1. **Gate correto & crash-safe:** o sidecar é DELETADO antes da cópia e REESCRITO só
   após `copyAsync` OK. Um crash entre delete e cópia (ou mid-copy) deixa o sidecar
   ausente → próximo boot vê `localVersion=null` → mismatch → re-copia (idempotente). O
   laço também remove `-wal/-shm/-journal` velhos, evitando corrupção do DB novo por WAL
   defasado. Identidade = `Asset.hash` (MD5 do conteúdo) prefixado por `DB_VERSION` —
   muda deterministicamente com o conteúdo. **Crash-safe confirmado.**
2. **Userdata NUNCA version-wiped:** o guard invalida só a cópia read-only de leitura
   (paths `reading-sample.sqlite*` no docDir raiz). O userdata é o subdir dedicado
   `userdata/` (`userdata.ts`) — fora do conjunto deletado. Nativo usa DB COMBINADO
   (leitura+léxico), sem cache de léxico separado. **Confirmado no nível de path.**
3. **Self-test não pode falsificar `adopted=true`:** reseta memoização primeiro,
   pré-semeia stale genuíno + sidecar velho, chama `ensureReadingDb`, e deriva `adopted`
   do retorno REAL de `get_chapter(Matthew/40)`. **Genuíno / anti-alucinação.**
4. **Web inalterado:** `db.web.ts` e `reading-selftest.web.ts` NÃO tocados pelo commit
   (Metro resolve `.web.ts` no web; a mudança `.ts` é nativa). Headless + smoke verdes.
5. **Falha pré-existente `TLA_SEARCH` NÃO é do F6.4:** o executor não tocou
   `app/web/search-selftest.ts` (último commit a tocá-lo: `ab75071`/F1.6, muito antes).
   O `TLA_SEARCH ERROR john_3_16_not_found hits=1000` é a suposição stale `PROOF_LIMIT=1000`
   da F5.36 (Bíblia completa → >1000 hits de "God", João 3:16 fora do top-1000). Follow-up
   SEPARADO (folded em F6.5), não regressão de F6.4.
6. **Não-negociáveis:** the-light em `225b8c9` (intocado); `.claude/settings.json` NÃO
   commitado (só mod de working-tree); sem segredo no diff; commit `bf3d663` toca apenas
   `app/lib/db.ts`, `app/web/reading-selftest.ts`, `scripts/run-ios-selftest.sh` e
   `scripts/gen-reading-sample-db.sh` (este só um bloco de comentário/nota).

## Decisões / ADRs

Nenhum ADR novo. F6.4 estende o padrão de bundling do ADR-0014 com invalidação por
versão. Não re-rodei o `run-ios-selftest.sh`: (a) o build iOS é ~15-30 min; (b) a
lógica do gate + a genuinidade do self-test são plenamente verificáveis por inspeção e
o marcador `TLA_DBUP` foi capturado e triangulado contra dados web independentes; (c) o
script completo hoje falharia na asserção pré-existente e NÃO-relacionada `TLA_SEARCH`
(item 5), não na parte F6.4. A prova device-específica de F6.4 (`TLA_DBUP adopted=true
matt1_verses=25`) está satisfeita.
