## Decisões / ADRs

- **ADR-0007 (aceito, F0.6b)** — Caminho WEB/WASM do `ubrn`, conferido contra a
  implementação real:
  1. **Subcomando `ubrn build web --config ubrn.config.yaml`** (não `jsi`, não
     `cargo build` cru) + bloco top-level **`web:`** (schema `WasmConfig`):
     `manifestPath: rust_modules/wasm/Cargo.toml`, `manifestPatchFile`,
     `target: web`, `ts: app/web/generated`, `entrypoint`. `package.json` raiz com
     `repository.url` (exigido pelo caminho web). Verificado em `ubrn.config.yaml`
     e no log do passo 1.
  2. **`--no-wasm-pack`**: o script só gera com o `ubrn` e roda ele mesmo os passos
     wasm32 + `wasm-bindgen`, para controlar a ordem e a versão do CLI.
  3. **`wasm-bindgen --target web` em lock-step com o crate**: o script LÊ a versão
     de `wasm-bindgen` do `Cargo.lock` gerado (`0.2.126`) e instala/garante o
     binário `wasm-bindgen-cli` NESSA versão (`--locked`). Conferido: passo [3/4]
     "wasm-bindgen do lock: 0.2.126".
  4. **manifest-patch do wasm-crate** (`wasm-crate.patch.toml`, via
     `manifestPatchFile`/`serde_toml_merge`) — conserta o template do `ubrn` **sem
     tocar a fronteira**: `[profile.release] opt-level = "s"` (a string `"3"` do
     template é inválida no cargo 1.96) e `[workspace] resolver = "2"` (faz o cargo
     respeitar o gate `cfg(not(wasm32))` da matriz de features do ADR-0005 e podar
     `embedded`/sqlite no wasm32). `wasm-bindgen` **não** é pinado no patch (entra
     em lock-step pelo `js-sys` via a cadeia `embedded`, presente só na resolução).
  5. **Metro `.wasm` asset**: `app/metro.config.js` registra `.wasm` em
     `resolver.assetExts`; o `expo export` empacota o `.wasm` como asset local
     (offline-first, single-thread, sem COOP/COEP).
- **ADR-0005 / ADR-0006** confirmados em campo: o resolver v2 do wrapper poda
  `rusqlite`/sqlite no wasm32 (matriz de features valendo) e `uniffi` resolve em
  `0.31.0` (casando com o runtime web do `ubrn`), eliminando o conflito de pins da
  attempt 1.
