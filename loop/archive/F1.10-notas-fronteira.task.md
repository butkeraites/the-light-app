## Objetivo

Expor na **fronteira UniFFI** (`core/src/lib.rs`) o **CRUD de notas** e de
**marcações/highlights** por referência, **delegando** ao `the-light-core` (rev
pinado `8f66004`): dado um **diretório de dados gravável do app** (`data_dir`) e uma
**referência canônica**, criar/ler/listar/remover **notas** (Markdown, uma por
referência) e **highlights** (cor + etiqueta opcional), persistidos em **formato
aberto e exportável** (arquivos), **separados** do conteúdo bíblico só-leitura.

**SITUAÇÃO (A) — CONFIRMADA NA FONTE DO CORE (decisivo p/ escopo e risco):** o
`the-light-core` **JÁ TEM** a camada `userdata` file-based — a F1.10 **delega a ela**
(não reimplementa store). Apuração na fonte pinada
(`~/.cargo/git/checkouts/the-light-9eb8809a6d68281a/8f66004/crates/the-light-core/src/userdata/`,
**consumo só-leitura, NÃO modificar**):
- `userdata/mod.rs`: `pub mod notes; pub mod highlights;` (+ `plans`/`sessions`,
  fora de escopo); reexporta `Note`/`NoteStore`/`Highlight`/`HighlightStore`. Erro
  `UserDataError { Io(std::io::Error), Json(serde_json::Error), NoDataDir }` (deriva
  `thiserror::Error` → `Display`). `data_dir()` usa `LIGHT_DATA_DIR`/`directories`
  (XDG) — a **fronteira NÃO usa** isto: recebe o `data_dir` do app explicitamente.
- `userdata/notes.rs` (Markdown, **um arquivo `.md` por referência**, nome =
  referência canônica EN com `_`/`.`, ex.: `John_3.16.md`):
  - `pub struct Note { pub reference: Reference, pub body: String }`
  - `NoteStore::new(dir: impl Into<PathBuf>) -> Self`
  - `get(&self, reference: &Reference) -> Result<Option<Note>>` (ausente → `Ok(None)`)
  - `put(&self, reference: &Reference, body: &str) -> Result<()>` (**escrita atômica**;
    cria/substitui)
  - `delete(&self, reference: &Reference) -> Result<bool>` (**idempotente**: `false`
    se não havia)
  - `list(&self) -> Result<Vec<Note>>` (**ordenada** por referência canônica; dir
    ausente → `Vec` vazio; ignora `.md` não-reconhecível)
  - `covering(&self, book: u8, chapter: u16, verse: u16) -> Result<Vec<Note>>`
- `userdata/highlights.rs` (**um único `highlights.json`**, array legível
  `{ "ref": "John 3:16", "color": "yellow", "tag": "salvação" }`):
  - `pub struct Highlight { pub reference: Reference, pub color: String, pub tag: Option<String> }`
  - `HighlightStore::load(path: impl Into<PathBuf>) -> Result<Self>` (arquivo ausente
    → coleção vazia; entradas com referência inválida ignoradas)
  - `save(&self) -> Result<()>` (**escrita atômica** do arquivo inteiro)
  - `list(&self) -> &[Highlight]`
  - `add(&mut self, highlight: Highlight)` (**substitui** a de mesma referência, não
    duplica)
  - `remove(&mut self, reference: &Reference) -> usize` (quantas saíram)
  - `covering(&self, book, chapter, verse) -> Vec<&Highlight>`

**Por que (A) é o caminho certo (e por que NÃO é tocar o `the-light`):** a camada já
existe, é **file-based** (exatamente o que o backlog pede: "userdata baseado em
arquivos JSON num diretório de dados, não no SQLite") e é consumida pela git dep
pinada — a fronteira só **adapta tipos/erros** (regra "uma fonte da verdade"). **Não
há** caminho (B) (implementar store na fronteira) nem (C) (mudar o core): seria
reimplementar o que o core já entrega. **Anti-alucinação NÃO se aplica ao corpo da
nota/highlight** (é dado do usuário), mas a **referência** usa o `Reference`
canônico (parseado pelo core); nada de rede; os dados vivem num **diretório
gravável separado** do `bible.sqlite` (conteúdo público só-leitura).

Mesmo molde da **F1.8** (NÚCLEO: fronteira Rust + teste de host determinístico):
export em **todos** os alvos + **corpo nativo** `#[cfg(not(target_arch = "wasm32"))]`
+ **stub web**; e — como o `xref::CrossRef` da F1.8 — os tipos-fonte
`userdata::notes::Note`/`userdata::highlights::Highlight` vivem em módulo
**`embedded`-only**, logo os respectivos `From` são **gated** `cfg(not(wasm32))`.

Esta é uma tarefa de **NÚCLEO** (fronteira + teste de host), análoga à F1.2/F1.5/F1.8.
A **UI nativa de notas/highlights, a persistência no device e o EXPORT são a F1.11**
(fora de escopo aqui). A **paridade web** de notas (OPFS/IndexedDB) é a **F1.16**
pós-gate **F1.12**.

## Contexto / arquivos

**Fronteira (este repo) — onde implementar:**
- `core/src/lib.rs` — adicionar:
  - Records `Note { reference: Reference, body: String }` e
    `Highlight { reference: Reference, color: String, tag: Option<String> }` (em
    **todos** os alvos; só tipos puros), com `From<…>` **gated** `cfg(not(wasm32))`.
  - 7 funções `#[uniffi::export]` (export em todos os alvos; corpo gated + stub web):
    notas (`put_note`/`get_note`/`delete_note`/`list_notes`) e highlights
    (`add_highlight`/`remove_highlight`/`list_highlights`).
  - Novo `#[cfg(all(test, not(target_arch = "wasm32")))] mod userdata_tests` (molde de
    `read_tests`/`search_tests`/`xref_tests`, mas com **diretório** temporário, não
    arquivo de DB).
- `core/Cargo.toml` — **NÃO precisa mudar**: a matriz por alvo já liga `embedded`
  só no não-wasm; `userdata` (e suas deps `serde_json`/`directories`/`tempfile`)
  existem só no nativo, já presentes desde a F1.2. Nada novo entra no grafo wasm.

**Gating de módulos do core (confirmado em `the-light-core/src/lib.rs`):** `model` e
`reference` são **puros** (todos os alvos); `userdata` (como `xref`/`search`/`store`)
é **`#[cfg(feature = "embedded")]`** (só nativo). Portanto:
- Os Records `Note`/`Highlight` referenciam só `Reference`/`String`/`Option<String>`
  (puros) → definidos em **todos** os alvos (como `CrossRef`).
- A construção de `NoteStore`/`HighlightStore`, as chamadas e os `From` a partir de
  `userdata::notes::Note`/`userdata::highlights::Highlight` referenciam o módulo
  `userdata` (`embedded`-only) → **só** dentro de `#[cfg(not(target_arch = "wasm32"))]`.
  O **stub web** não pode tocar nenhum desses símbolos (consome os args com `let _ = (...)`).

**Diretório de dados (decisão de layout — IMPORTANTE):** a fronteira recebe **UM**
`data_dir: String` (caminho **gravável** do dispositivo; iOS/Android sandbox de
documentos — quem o fornece é a F1.11) e **deriva os subcaminhos do MESMO layout que o
core usa por padrão** (`data_dir()` → `notes/` + `highlights.json`), para que o formato
em disco seja **idêntico** ao de `open_default`/`load_default` (essencial p/ o EXPORT
da F1.11 e a paridade web F1.16):
- Notas: `NoteStore::new(std::path::Path::new(&data_dir).join("notes"))`.
- Highlights: `HighlightStore::load(std::path::Path::new(&data_dir).join("highlights.json"))`.
A fronteira **NUNCA** chama `data_dir()`/`open_default`/`load_default` (que dependeriam
de XDG/`directories`) → o caso de erro `UserDataError::NoDataDir` **não ocorre** (o
diretório é sempre explícito). Documentar isso.

**Referência canônica (delegação, sem `From` reverso):** as funções recebem a
referência como `String` e a **parseiam pelo core** dentro do corpo gated, como
`get_passage` faz: `the_light_core::reference::parse_reference(&reference)` → um
`the_light_core::model::Reference` (o tipo que `NoteStore`/`HighlightStore` esperam).
Isso **delega a canonicalização ao core** (PT e EN caem na MESMA nota/arquivo) e
**evita** precisar de um `From<Reference> for model::Reference` na fronteira.
Referência que **não parseia** → `CoreError` (antes de qualquer I/O).

**Funções a expor (assinaturas recomendadas — molde de `search`/`cross_refs`):**

Notas:
- `put_note(data_dir: String, reference: String, body: String) -> Result<(), CoreError>`
  - gated: parse ref → `NoteStore::new(data_dir/"notes").put(&ref, &body)` (atômico;
    cria/substitui) → mapeia `UserDataError` p/ `CoreError`.
- `get_note(data_dir: String, reference: String) -> Result<Option<Note>, CoreError>`
  - gated: `NoteStore::new(…).get(&ref)?.map(Note::from)` (ausente → `Ok(None)`).
- `delete_note(data_dir: String, reference: String) -> Result<bool, CoreError>`
  - gated: `NoteStore::new(…).delete(&ref)` (**idempotente**: `false` se não havia).
- `list_notes(data_dir: String) -> Result<Vec<Note>, CoreError>`
  - gated: `NoteStore::new(…).list()` → `Vec<Note>` (ordenada; dir ausente → vazia).

Highlights (modelo **load → mutate → save** a cada chamada; cada chamada abre um
store novo do disco → a persistência é provada relendo de **outro handle**):
- `add_highlight(data_dir: String, reference: String, color: String, tag: Option<String>) -> Result<(), CoreError>`
  - gated: `let mut s = HighlightStore::load(data_dir/"highlights.json")?;
    s.add(Highlight { reference: ref, color, tag }); s.save()?` (**substitui** a de
    mesma referência).
- `remove_highlight(data_dir: String, reference: String) -> Result<u32, CoreError>`
  - gated: load → `let n = s.remove(&ref); s.save()?; Ok(n as u32)` (**idempotente**;
    `usize`→`u32` porque UniFFI não tem `usize`).
- `list_highlights(data_dir: String) -> Result<Vec<Highlight>, CoreError>`
  - gated: `HighlightStore::load(…)?.list().iter().cloned().map(Highlight::from).collect()`
    (arquivo ausente → vazia).

**Stub web (todas as 7):** `let _ = (data_dir, reference, …);` →
`Err(CoreError::Generic { message: "store local indisponível no alvo web (F0.10:
wa-sqlite+OPFS)".to_string() })` — **mesma mensagem-padrão** das outras funções
DB-backed (a paridade web de notas é a F1.16). Para `get_note` (retorna
`Option<Note>`) e `list_*` (retornam `Vec`), o stub também retorna `Err(CoreError…)`.

**Novos Records UniFFI (definir em TODOS os alvos; só tipos puros):**
- `Note { reference: Reference, body: String }` — `From<the_light_core::userdata::notes::Note>`
  **gated** `cfg(not(wasm32))` (tipo-fonte `embedded`-only). Doc-comment: `reference`
  via o `From` de `model::Reference` já existente; `body` é Markdown **do usuário**
  (anti-alucinação não se aplica ao corpo; aplica-se à referência, que é canônica).
- `Highlight { reference: Reference, color: String, tag: Option<String> }` —
  `From<the_light_core::userdata::highlights::Highlight>` **gated** `cfg(not(wasm32))`.
  Doc-comment: `color` é nome livre (ex.: `"yellow"`); `tag` é etiqueta opcional.

**Mapeamento de erro:** `UserDataError` (Io/Json/NoDataDir) →
`CoreError::Generic { message: e.to_string() }` (igual às demais). `NoDataDir` não
ocorre (diretório explícito). Documentar.

**Banco/arquivos de teste (determinismo) — molde F1.5/F1.8, mas DIRETÓRIO temporário:**
- O teste de host **PRIMÁRIO** é **offline e determinístico**: cria um **diretório
  temporário único** (não um arquivo de DB), exercita as 7 funções e **relê de outro
  handle/chamada** (a próxima chamada abre um store novo do disco → prova persistência
  em **disco**, não em memória). Reusar o padrão `TmpDb`/`unique_tmp_db`/`Drop`, mas
  para **diretório** (criar com `std::fs::create_dir_all`; no `Drop`,
  `std::fs::remove_dir_all`). **Não** depende de `bible.sqlite` nem de rede.
- **Sem fixture de conteúdo:** notas/highlights são chaveadas por **referência**,
  independentes de qualquer `bible.sqlite` — o teste **não** abre nem precisa de banco
  bíblico. (Isto reforça a separação de dados: a função nem recebe `db_path`.)
- Textos do teste são **do usuário** (não bíblicos), ex.: `"# Amor de Deus\n\nVersículo
  central."` — anti-alucinação não se aplica ao corpo.

**Política de "referência inexistente" (decidida — documentar no doc-comment e no
result):** as funções de nota/highlight **não recebem `db_path`** e **não verificam**
se o versículo existe em algum `bible.sqlite` — são dados do usuário **chaveados por
referência canônica**. Logo: (1) uma `String` que **não parseia** como referência →
`CoreError` (falha no `parse_reference` do core, antes do I/O); (2) uma referência
**sintaticamente válida** (livro/capítulo/versículo canônicos) é **sempre aceita**,
exista ou não texto para ela — é responsabilidade da UI (F1.11) só oferecer notas em
referências reais. Sem `get_note` prévio, `get_note` de referência válida sem nota →
`Ok(None)` (não erro).

**ADRs/decisões pertinentes:** ADR-0005 (matriz de features por alvo; `embedded` só no
nativo → `userdata` é nativo-only), ADR-0010 (gating de **corpo** por alvo + stub web).
VISION §3 (princípios: "os dados do usuário são do usuário", formato aberto e
exportável; offline-first) e §6 ("dados do usuário: notas, marcações… persistidos no
dispositivo com formato exportável"). Backlog: `loop/backlog/PHASE-1.md` (**F1.10**:
"userdata baseado em arquivos JSON… a fronteira recebe um diretório de dados"; e
**F1.11** = UI/persistência/export, **fora daqui**). Estado do loop: `loop/STATUS.md`.
