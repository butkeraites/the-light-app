## Objetivo

Dar **paridade web de BUSCA** (FTS5) sobre o subset `reading-sample.sqlite` (~4,4 MB,
o MESMO que o nativo empacota — ADR-0014), continuando o ADR-0018 (Opção A / **A1**):
**espelhar em TS o SELECT de busca FTS5 do core** (`the-light-core::search::search`)
sobre o `wa-sqlite`/OPFS+MemoryVFS já provado na F1.13 (ADR-0019), **destubando**
`search` em `app/web/reading.web.ts`. As MESMAS telas de busca da F1.6
(`app/app/search/index.tsx` + `app/components/ReaderSearchResultItem.tsx` +
`app/lib/highlight.ts`) passam a funcionar no browser só por esse glue; referência
clicável → `read/[book]/[chapter]` (leitura web já funciona, F1.13).

**RISCO CRÍTICO — verificado pelo planner e CENTRAL nesta tarefa:** o build do
`wa-sqlite@1.0.0` (SYNC e async) que a F1.13 usa **NÃO tem a extensão FTS5 compilada**.
Prova obtida pelo planner (a confirmar como passo 1):
```
$ # probe: criar uma fts5 vtable no wa-sqlite/dist/wa-sqlite.wasm (build SYNC) em node
FTS5_FAIL no such module: fts5
```
Sem FTS5, `verses_fts MATCH` / `bm25(...)` / `highlight(...)` **não rodam** no web.
O pacote npm só traz os `dist/` pré-compilados (sem Makefile/amalgamation); `emcc`
ausente no ambiente, `docker` presente. Portanto **habilitar FTS5 no wa-sqlite SYNC**
(asset local, offline-first, sem `SharedArrayBuffer`/COOP-COEP) é parte do escopo —
ver "Passos" (árvore de decisão) e a condição de `blocked`.

## Contexto / arquivos

**O que REUSAR da F1.13 (NÃO duplicar — não recarregar o subset):**
- `app/web/sqlite-reading-opfs.web.ts` — backend de runtime: `openReadingDbWeb()`
  abre o subset em OPFS (browser) via MemoryVFS hidratado, build SYNC. A busca web
  **abre o store por aqui** (o MESMO subset), exatamente como `getChapter` faz.
- `app/web/sqlite-reading.web.ts` — queries VFS-agnósticas (SELECTs espelhados +
  `hasTranslation`). A query de busca FTS5 deve viver como par destas (estender este
  arquivo OU um novo `app/web/sqlite-search.web.ts` no MESMO padrão VFS-agnóstico).
- `app/web/reading.web.ts` — onde `search(...)` é HOJE um stub (`throw WEB_SEARCH_MSG`).
  Destubar AQUI, mantendo a assinatura compartilhada com o nativo.
- `app/web/__tests__/reading.web.test.mjs` + `reading-headless-entry.ts` — **molde** da
  prova headless node (MemoryVFS sobre os bytes do subset; OPFS é browser-only).
- `app/web/__tests__/reading.web.test.mjs` deve continuar **verde** (sem regressão da
  leitura) com qualquer wasm novo.

**A FRONTEIRA NATIVA que o web ESPELHA (NÃO reimplementar — só espelhar o SQL):**
- `core/src/lib.rs::search(db_path, query, translation, book: Option<u8>, limit:
  Option<u32>) -> Result<Vec<SearchHit>, CoreError>` delega a `BibleSource::search`.
- `EmbeddedSource::search` (the-light-core `source/embedded.rs`): **checa
  `has_translation` PRIMEIRO** → se ausente, `SourceError::UnknownTranslation` (a
  fronteira mapeia p/ `CoreError`); só então chama `search::search`.
- **O SQL REAL** (the-light-core `search.rs`, rev pinado `8f66004` — confirmado na
  fonte pelo planner):
  ```sql
  SELECT v.book_number, v.chapter, v.verse, v.text,
         highlight(verses_fts, 0, ?, ?) AS hl, bm25(verses_fts) AS score
  FROM verses_fts JOIN verses v ON v.id = verses_fts.verse_id
  WHERE verses_fts MATCH ? AND verses_fts.translation_id = ?
  -- (opcional, só quando `book` informado:)  AND v.book_number = ?
  ORDER BY score LIMIT ?
  ```
  Ordem dos params: `HL_START`(U+0002), `HL_END`(U+0003), `match_query`,
  `translation_id`, `[book]`, `limit`. `ORDER BY score` (bm25; **menor = mais
  relevante**). `limit` é **clampado a `[1, i64::MAX]`** (evita LIMIT 0 e o wrap de
  usize→i64); `limit` ausente → `DEFAULT_LIMIT = 20`.
- **`build_match_query`** (search.rs) — sanitização anti-injeção FTS5, a espelhar em
  TS: divide a query por espaços, envolve **cada** palavra em `"aspas"` (escapando
  `"` interna → `""`) e junta por espaço (AND implícito). Sem termos (vazio/só-espaços)
  → `None` → retorna `[]` **sem erro**. É infra (segurança + semântica AND), **não**
  ranqueamento — mirror obrigatório p/ paridade e p/ não permitir injeção.
- Record alvo: `SearchHit { reference (Single book/chapter/verse), translation,
  text (verbatim, coluna `v.text` — LIMPO, sem marcadores), highlighted (coluna `hl`,
  com U+0002/U+0003 ao redor do termo), score (f64 bm25) }` — `core/src/lib.rs` L262.
  Os marcadores: `HL_START = "\u{2}"`, `HL_END = "\u{3}"` (`search.rs` L12/14;
  espelhados em `app/lib/highlight.ts`).

**Fatos do subset (confirmados pelo planner, `assets/data/reading-sample.sqlite`):**
- `verses_fts` é **FTS5** `unicode61 remove_diacritics 2` (acento-insensível vem
  daqui), 9746 linhas; `verses` 9746; traduções `kjv` (en) e **`alm1911`** (pt — NÃO
  "alm"). Colunas de `verses_fts`: `text`, `translation_id UNINDEXED`,
  `verse_id UNINDEXED`. Join: `verses_fts.verse_id = verses.id`.
- `search("God","kjv")` → **646 hits**; João 3:16 (book 43/cap 3/v 16) ESTÁ entre eles,
  com `v.text` = `For God so loved the world, that he gave his only begotten Son, …`.
  **ATENÇÃO À PARIDADE REAL:** o 1º hit por bm25 é **Salmos 42:2** (`19|42|2`,
  score -4.286), **NÃO** João 3:16 (que tem 1 ocorrência de "God"). O self-test nativo
  (`app/web/search-selftest.ts`) **não** asserta João 3:16 como 1º — ele chama
  `search(db,"God","kjv",undefined,1000)` (limite generoso, `PROOF_LIMIT`) e
  **localiza** o hit de João 3:16 NO conjunto retornado (`.find()`), emitindo-o como
  `first_ref`. A prova web DEVE espelhar ISSO (não assertar "1º hit = João 3:16").
- `search("ceus","alm1911")` → top hit Salmos 148:4 (`19|148|4`, texto com `céus`);
  `search("deus","alm1911")` → 646 hits — comprova acento-insensível do `verses_fts`.
- Tradução inexistente: a fronteira nativa **lança** (UnknownTranslation→CoreError) por
  causa do `has_translation` ANTES do SQL — o web deve fazer o mesmo (≠ "vazio").

**UI compartilhada (F1.6) — deve passar a funcionar no web só com o destub:**
- `app/app/search/index.tsx` importa `search`/`listBooks` de `../../web/reading`
  (Metro resolve `reading.web.ts` no web). Já trata loading/erro/vazio; chama
  `search(dbPath, term, 'kjv')` (sem book/limit → default 20). Referência clicável →
  `router.push('/read/[book]/[chapter]')` (rota F1.13).
- `app/components/ReaderSearchResultItem.tsx` + `app/lib/highlight.ts`:
  `splitHighlighted` consome os marcadores → estilo; **nunca** exibe os controles crus.
- `app/web/search-selftest.web.ts` é hoje um SKIP (`TLA_SEARCH SKIP web`). Destubá-lo
  p/ rodar a busca web real no bundle é **opcional** (a prova determinística é o
  headless node); se destubar, manter o marcador `TLA_SEARCH` e o texto LIMPO
  (`stripMarkers`) como no nativo.

## Passos sugeridos

1. **VERIFICAR FTS5 (passo 1, obrigatório, antes de qualquer SQL de busca).** Rode um
   probe honesto contra o wa-sqlite EM USO (build SYNC): crie uma `CREATE VIRTUAL TABLE
   … USING fts5(…)` + `bm25`/`highlight`/`MATCH` em node (MemoryVFS). Resultado
   esperado pelo planner = **`no such module: fts5`** (FTS5 AUSENTE). Registre a saída
   exata no resultado.

2. **HABILITAR FTS5 no wa-sqlite SYNC (caminho preferido — sem hack proibido).** Como
   FTS5 está ausente, obtenha um wa-sqlite **SYNC com FTS5** (que cobre `MATCH`/`bm25`/
   `highlight` e o tokenizer `unicode61 remove_diacritics 2`), mantendo offline-first
   (artefato = **asset local**; **sem** `SharedArrayBuffer`/COOP-COEP):
   - **Preferir** build reproduzível do wa-sqlite a partir da fonte com
     `-DSQLITE_ENABLE_FTS5` (Emscripten; `emcc` ausente → via `docker`/emsdk — rede de
     dev/build é permitida), num script versionado (ex.: `scripts/build-wa-sqlite-fts5.sh`),
     **mantendo o build SYNC** (só adiciona o módulo FTS5). Vendore o par
     `.mjs`+`.wasm` resultante como asset local (ex.: `app/assets/wasm/…` ou substituindo
     o import único do wa-sqlite) e aponte **reading E busca + as provas headless** p/
     ESSE artefato (um único wasm p/ os dois — a leitura NÃO pode regredir).
   - Registre a decisão em **ADR-0020** (origem/flags do build, reprodutibilidade,
     por que SYNC, offline-first preservado, gerar-vs-versionar o artefato).
   - **Se** o build de um wa-sqlite SYNC com FTS5 for **genuinamente inviável** neste
     ambiente sem violar regra (ex.: toolchain de build indisponível/offline e sem
     alternativa; OU FTS5 no SYNC só seria possível com `SharedArrayBuffer`/COOP-COEP;
     OU a única saída seria trocar de biblioteca SQLite-wasm — decisão arquitetural):
     → **`blocked` + HALT** com o **erro EXATO** e a decisão estratégica a tomar
     (opções a apresentar, NÃO a executar sozinho: investir no pipeline de build FTS5;
     adotar `@sqlite.org/sqlite-wasm` que traz FTS5 mas exige Worker+COOP-COEP p/ OPFS
     ao vivo → conflita com export web estático offline-first; ou **busca degradada via
     LIKE no web** → **diverge** do FTS5/bm25/highlight do nativo = **drift** e quebra
     a paridade, **só com sign-off + ADR**). **NUNCA** caia para `LIKE` em silêncio.

3. **Espelhar `build_match_query` em TS** (infra anti-injeção): `split` por espaços →
   cada termo `"<termo com aspas duplicadas>"` → `join(' ')`; vazio → `[]` (sem throw).

4. **Query de busca FTS5 VFS-agnóstica** (par de `sqlite-reading.web.ts`): uma função
   `querySearch(handle, matchQuery, translation, book?, limit)` que roda **exatamente**
   o SELECT do core (acima), com os params na ordem do core (HL_START, HL_END,
   match_query, translation, [book], limit), `bind` de char(2)/char(3) como os
   marcadores, e devolve linhas `{ book, chapter, verse, text, highlighted, score }`.
   Comente citando a fonte (`the-light-core/src/search.rs`), como a F1.13 faz.

5. **Destubar `search` em `reading.web.ts`** (assinatura intacta): abrir o store via
   `openReadingDbWeb()` (REUSO da F1.13 — sem recarregar o subset); **checar
   `hasTranslation` PRIMEIRO** → se falso, `throw new Error('versão desconhecida:
   ${translation}')` (espelha UnknownTranslation→CoreError, como `getChapter` já faz);
   `build_match_query(query)` → se vazio, retornar `[]`; senão `querySearch(...)` com
   `limit` default 20 (clamp ≥ 1) e compor cada `SearchHit` (reference `Single` via
   `VerseRange.Single`, translation, text verbatim, highlighted com marcadores, score).
   `finally { handle.close() }`.

6. **UI/web:** confirmar que `app/app/search` + `ReaderSearchResultItem` renderizam no
   web só com o destub (sem mudar a UI nativa); markers→estilo via `highlight.ts`;
   referência clicável navega ao Reader (F1.13). Adicionar o script
   `app/package.json::test:web:search` (par de `test:web:reading`).

7. **Prova HEADLESS node** (molde `reading.web.test.mjs`): novo
   `app/web/__tests__/search.web.test.mjs` + `search-headless-entry.ts` (reexporta as
   funções de PRODUÇÃO: `buildMatchQuery`, `querySearch`/`searchOnHandle`,
   `hasTranslation`) — MemoryVFS sobre os bytes do subset, exercitando o MESMO código.
   Asserções **do RETORNO REAL** (não hardcoded além das constantes-asserção):
   - `search("God","kjv", undefined, 1000)` → o conjunto **contém** João 3:16
     (book 43/cap 3/v 16) com `text` = KJV verbatim e `highlighted` contendo U+0002/
     U+0003 ao redor de "God" (mirror do `TLA_SEARCH` nativo — **localizar** João 3:16,
     NÃO assertar como 1º). Opcional: `result.length === 646` (a contagem real).
   - **acento-insensível**: `search("ceus","alm1911")` (ou `"deus"`) → ≥1 hit cujo
     `text` contém a forma **acentuada** (`céus`/`Deus`), provando `remove_diacritics`.
   - query vazia/só-espaços (`search("   ","kjv")`) → `[]` sem throw.
   - `limit` respeitado: `search("God","kjv", undefined, 3)` → 3 resultados.
   - tradução inexistente (`search("God","nope")`) → **lança** (`versão desconhecida:
     nope`).
   - marcadores **não vazam**: `text` limpo (sem U+0002/U+0003); `highlighted` os tem.
   - **PARIDADE explícita** documentada: o João 3:16-no-conjunto é o MESMO que o nativo
     prova em `TLA_SEARCH` (F1.6, `search(...,'kjv',undefined,1000)` + `.find()`).

## Restrições (regras rígidas relevantes)

- **NÃO tocar `the-light`/fronteira Rust.** A busca já existe na F1.5 (`core/src/lib.rs::
  search` → `the-light-core::search`); o web **só ESPELHA** o SQL FTS5 no glue TS.
- **NÃO reimplementar ranqueamento/semântica.** `bm25`/`highlight`/`ORDER BY score` vêm
  do **FTS5 do SQLite**, espelhando o core. **Proibido** inventar score/ordenação em TS.
  **Proibido** cair para `LIKE` (drift vs FTS5 do nativo) — só com **sign-off + ADR**.
- **FTS5 obrigatório no wa-sqlite** (offline-first: asset local; **sem**
  `SharedArrayBuffer`/COOP-COEP). Se inviável sem hack proibido → **`blocked` + HALT**
  com o **erro EXATO** e a decisão estratégica (NÃO improvisar).
- **Subset A1 só** (`reading-sample.sqlite` ~4,4 MB); corpus completo (~59 MB) =
  backlog transversal pós-paridade.
- **Anti-alucinação:** texto/snippets SEMPRE do store local (verbatim); marcadores HL
  (U+0002/U+0003) **não vazam** como texto cru (viram estilo na UI).
- **Offline-first:** zero rede em runtime (assets locais; rede só em dev/build).
- **Paridade real com o nativo:** o João 3:16 que o nativo localiza em `TLA_SEARCH`
  (limite 1000) deve estar no conjunto retornado pela busca web (mesmo SQL, mesmo dado).
- **REUSAR** o store web da F1.13 (`openReadingDbWeb`) — não criar um 2º carregamento
  do subset.
- **Qualidade:** `tsc --noEmit`/`eslint` verdes; `expo export --platform web` 0.
- Em conflito com qualquer regra: **HALT**, não improviso (PROTOCOL.md).
