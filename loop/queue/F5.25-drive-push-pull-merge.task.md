## Objetivo

Terceira etapa da trilha de sync (ADR-0036): usar o token linkado na F5.24 para
**empurrar (push) e puxar (pull)** o snapshot da F5.23 para/da pasta **app-data privada**
do Google Drive do usuário — e, no pull, **mesclar** com o estado local usando o
`mergeSnapshots` determinístico da F5.23. Com isto, os dados do usuário (notas + marcações
+ progresso de plano) transitam entre dispositivos via o Drive DELE, sem servidor do app.

Offline-first permanece a **base** (o app é 100% funcional sem conta/rede). Isto é
estritamente **opt-in** e **aditivo**; o merge nunca apaga dado local (união + `max(completed)`).

## Contexto / peças que esta tarefa costura

- **F5.23** `app/lib/userdataSnapshot.ts` — `buildSnapshot`/`parseSnapshot`/`mergeSnapshots`/
  `importSnapshotIntoStore` (motor puro; NÃO reimplementar; reusar tal-e-qual).
- **F5.24** `app/lib/driveAuth.ts` — provê o `accessToken` (via `currentToken()`), escopo
  `drive.appdata`. NÃO reimplementar OAuth; receber o token/uma função `getToken()` injetada.
- **API Drive (app-data):** upload multipart em `POST /upload/drive/v3/files?uploadType=multipart`
  com `parents:["appDataFolder"]`; listar com `GET /drive/v3/files?spaces=appDataFolder`;
  baixar com `GET /drive/v3/files/{id}?alt=media`; atualizar com `PATCH /upload/.../files/{id}`.
  Um único arquivo canônico (ex.: `the-light-app.snapshot.json`) na pasta app-data.

## Escopo desta tarefa

1. **`app/lib/driveSync.ts` (motor PURO, injeção de dependências).** Injeta `fetch`,
   `getToken()` (da F5.24), e um `SnapshotStore` (o mesmo backend injetável da F5.23).
   Expor:
   - `pushSnapshot()` — `buildSnapshot(store)` → serializa → **upload/replace** do arquivo
     canônico na pasta app-data (cria se não existe; senão PATCH por id). Retorna `{fileId, bytes}`.
   - `pullSnapshot()` — localiza o arquivo canônico (list por nome em `appDataFolder`),
     baixa (`alt=media`), `parseSnapshot` (valida app/version/refs reais via core **antes**
     de tocar o store), **`mergeSnapshots(local, remoto)`**, e aplica via
     `importSnapshotIntoStore` (só o diff). Retorna `{applied:{notes,highlights,planProgress}, merged}`.
   - `syncNow()` — pull-then-push (convergência): puxa+mescla, depois empurra o merge —
     idempotente (rodar 2× seguidas não muda nada na 2ª).
   - `findSnapshotFile()` helper (list por `name` + `spaces=appDataFolder`).
   - Toda request manda `Authorization: Bearer <token>` do `getToken()`; **token nunca logado**.
2. **Anti-alucinação/privacidade (reforço):** o que sobe é EXATAMENTE o snapshot da F5.23
   (notas+marcações+progresso; referências canônicas; **sem texto bíblico, sem sessões de IA,
   sem banco, sem chaves**). O pull valida referências reais (core) antes de escrever.
3. **Merge nunca destrói dado local:** reusar `mergeSnapshots` (união por referência;
   progresso `max(completed)`); provar que um pull de um remoto com MENOS dados não apaga
   local, e que notas divergentes convergem para a união nos dois lados após `syncNow`.
4. **Prova headless** (`app/web/__tests__/driveSync.web.test.mjs`, mock `fetch`+`SnapshotStore`
   in-memory que emula a pasta app-data como um dict `{name→bytes}`):
   - push cria o arquivo (list depois acha 1); segundo push faz replace (ainda 1 arquivo, id estável).
   - pull de app-data vazio = no-op (nada a mesclar; store intacto).
   - Dois "dispositivos" (dois stores + uma "nuvem" compartilhada): A tem nota X, B tem nota Y;
     A.syncNow então B.syncNow ⇒ **ambos convergem** para {X,Y} (união); progresso pega o `max`.
   - Idempotência: `syncNow` 2× seguidas no mesmo device ⇒ 0/0/false aplicados na 2ª.
   - **Token não vaza** no output (asserção `notoken=ok`, espiando console).
   - Marcador `DRIVE_SYNC push=ok pull=ok converge=ok idempotent=ok notoken=ok`.
   - Script `test:web:drivesync`.
5. **MOCK apenas.** NENHUMA chamada real ao Google/Drive. Validação real = **F5.27** (gate humano).
6. **ADR-0053** — decisão do protocolo de sync (arquivo canônico único, pull-merge-push,
   convergência/idempotência, o que sobe, reforço anti-alucinação/privacidade).
