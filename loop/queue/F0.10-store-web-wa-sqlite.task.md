## Objetivo

Provar a **camada de store NO WEB** (Opção A do ADR-0011): expor um
`getPassage` web em **TypeScript** que lê **uma passagem** (`"John 3:16"`) de um
banco **`wa-sqlite`** (SQLite em wasm) persistido em **OPFS**, com o **mesmo
schema/sample** do nativo (`assets/data/sample.sqlite`, F0.9), e retorna o
**texto KJV verbatim** do versículo **lido do store local**.

Pipeline obrigatório (ADR-0011):
1. **A referência é resolvida PELO RUST (wasm):** `getPassage` web chama o
   `parseReference` já existente (`app/web/reference.web.ts` → bindings wasm da
   F0.6b) para obter `book`/`chapter`/`verse`. **NÃO** reimplementar parsing em TS.
2. **A query de passagem é infraestrutura em TS:** com `book`/`chapter`/`verse`,
   rodar em `wa-sqlite` o **mesmo SELECT** de `EmbeddedSource::passage`
   (`SELECT verse, text FROM verses WHERE translation_id=? AND book_number=? AND
   chapter=? AND verse=?`) e compor a `Passage` (Reference do Rust + texto do store).
3. **Anti-alucinação:** o texto do versículo vem **sempre do `wa-sqlite`/OPFS
   local** (verbatim KJV, domínio público) — **nunca** hardcoded no código de
   produção nem gerado por LLM.

A prova **obrigatória e determinística** é um **teste headless em node** (sem
browser): instancia o `parseReference` via wasm (estilo F0.6b, bytes do
`index_bg.wasm`) **+** abre um `wa-sqlite` em **VFS de memória** carregado com os
**bytes de `assets/data/sample.sqlite`**, roda a **mesma função de query** do glue
de produção e assere `text == <KJV verbatim>` com `book=43, chapter=3, verse=16`.
O VFS **OPFS** é o de **runtime no browser** (selecionado sob `typeof navigator`);
a prova node cobre a **query + leitura do sample** com VFS de memória sobre os
**mesmos bytes** (OPFS é browser-only — ver Contexto).

File System) **não existe em node**. Portanto:
- O **VFS OPFS** do `wa-sqlite` é o de **runtime no browser** — selecionado sob um
  guard `typeof navigator !== 'undefined'` / `typeof OPFS`-disponível no glue web.
- A **prova determinística** roda em **node** com um **VFS de memória** do
  `wa-sqlite` (sem OPFS), carregado com os **mesmos bytes** de
  `assets/data/sample.sqlite`, exercitando **a mesma função de query** usada em
  produção. Documentar claramente no teste o que ele cobre (a **query + leitura do
  sample**) e que o **OPFS é o VFS de runtime no browser**. **EVITAR** browser
  headless (puppeteer) se o teste node sobre a mesma função de query for suficiente
  e honesto.

## Restrições

- **NÃO modificar nem forkar `the-light`** (`/Users/butkeraites/Documents/the-light`
  é leitura-apenas; consumo é git dep pinada `rev 8f66004`, ADR-0002). A Opção A
  (ADR-0011) é justamente **não tocar o core**: o web espelha o SELECT em TS.
  Qualquer necessidade de mudar o core/fronteira ⇒ **`blocked`** com o erro EXATO
  (PR + ADR = decisão humana), **não** improviso.
- **NÃO reimplementar PARSING em TS.** A resolução de referência vem do **Rust
  (wasm)** via `parseReference` (uma fonte da verdade). Só a **query de passagem**
  (o SELECT espelhado de `EmbeddedSource::passage`) é TS — é **infraestrutura**, não
  lógica de domínio.
- **Anti-alucinação:** o texto do versículo vem **sempre do store local**
  (`wa-sqlite`/OPFS), verbatim. **Nunca** hardcodar texto bíblico no código de
  produção; a constante KJV pode existir **apenas no teste** (asserção), como na F0.9.
- **Licenciamento:** somente **KJV / domínio público** (o `sample.sqlite` já é
  domínio público). NÃO embarcar texto protegido/não-livre.
- **Offline-first (runtime):** nenhuma rede externa em runtime — o `sample.sqlite` e
  o `.wasm` do `wa-sqlite` são **assets locais/empacotados**; a carga em OPFS é fetch
  do asset da própria origem. Rede só em dev/build (npm/cargo/wasm-bindgen). **Segredos
  nunca** em git/log.
- **Gerados IGNORADOS:** wasm do `wa-sqlite`, artefatos OPFS, `app/web/generated/`,
  `rust_modules/`, `dist/`, `node_modules/`. Versionar **apenas**: o glue web do
  store (`app/web/*.ts` hand-written), o teste headless, `app/package.json`(+lock,
  dep `wa-sqlite`), config Metro se preciso, e a nota de ADR.
- **Sem regressão:** `parseReference` web (F0.6b), nativo iOS/Android (F0.7/F0.8) e
  store nativo (F0.9) **permanecem verdes/intactos**. A tela compartilhada não pode
  quebrar o caminho nativo.
- **Prova honesta (OPFS browser-only):** a prova node usa VFS de memória do
  `wa-sqlite` sobre os **mesmos bytes** do `sample.sqlite` e a **mesma função de
  query** do produto; o OPFS é o VFS de runtime do browser (selecionado sob `typeof
  navigator`). Documentar essa fronteira no teste. EVITAR puppeteer se o node bastar.
- **Bloqueio legítimo (`blocked`) com erro EXATO** se: o `wa-sqlite`/OPFS no Expo/
  Metro web for **inviável sem hack proibido** (ex.: exigir COOP/COEP +
  `SharedArrayBuffer` que o Expo export não suporta sem servidor especial; ou o
  Metro não empacotar o `.wasm` do `wa-sqlite` sem patch proibido; ou `wa-sqlite`
  em node não conseguir carregar honestamente os bytes do sample). Nesse caso,
  `done/F0.10.result.md` com `status: blocked`, a **saída literal** do comando que
  falhou e a decisão pendente → ADR-0011/ADR-0012.
