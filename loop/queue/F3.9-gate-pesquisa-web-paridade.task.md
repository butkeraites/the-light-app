## Objetivo

**Gate estratégico (decisão humana/arquitetural).** Por ser `gate: true`, o loop
**NÃO** executa esta tarefa autonomamente: o Driver escreve `loop/HALT` e o
**humano (Renan)/Guia** decide antes de liberar a validação real (F3.10), um
eventual PR ao core (F3.11) e a paridade web (F3.12) que fecham a Fase 3.

Esta tarefa **não tem código de produto a implementar**. É um **BRIEF DE DECISÃO**:
com **todas as capacidades NATIVAS da Fase 3 (F3.1–F3.8) já aceitas** (léxico Strong
verificado, `deep_study`, conversa/refinamento, UI de estudo/conversa/comparação
multi-IA e export acadêmico — tudo com **MOCK**, anti-alucinação provada), definir
**(D1)** se e como habilitar a **pesquisa web assistida opt-in** (`ai::research`) e
**(D2)** qual a **estratégia de paridade web** do estudo profundo/léxico/conversa —
mantendo **offline-first**, **BYOK**, **anti-alucinação** (texto/glosas/citações
sempre do store local + fontes web citadas verbatim e verificadas) e **`the-light`
intacto** (`c8ecb2f`; mudança no core só via PR + ADR). **(D3)** confirma o escopo
da validação real (F3.10) como gate de chave real. O desfecho do ciclo é **HALT p/
sign-off humano**; com a decisão registrada (novo **ADR-0028**), o Driver
re-escopa/semeia **F3.10**, **F3.11** (condicional), **F3.12** e **F3.13**.

**F3.10, F3.11, F3.12 e F3.13 dependem desta decisão.**

### 1.2 A pesquisa web (`ai::research`) — símbolos REAIS (fonte `c8ecb2f`, só-leitura)
Fonte: `~/.cargo/git/checkouts/the-light-9eb8809a6d68281a/c8ecb2f/crates/the-light-core/src/ai/research.rs`
+ gating em `ai/mod.rs`.

- **Todo o módulo `ai::research` é `#[cfg(feature="embedded")]`** (`ai/mod.rs:19-20`;
  re-exports `ai/mod.rs:39-40` também `embedded`). Usa **`reqwest::blocking`**
  (`research.rs:48`, timeout 15s, User-Agent próprio) → **nativo-only; NÃO compila
  em wasm/`ai-pure`** (mesma fricção do transporte de IA, VISION §4 #2).
- **`RESEARCH_BACKENDS = &["wikipedia", "tavily", "mock"]`** (`research.rs:23`).
- **`WebSource`** (`research.rs:27`): `{ title, url, snippet, site, fetched_at:
  DateTime<Utc> }` — struct serde, **mas** re-exportada só sob `embedded` (o
  `chrono`/`fetched_at` é `embedded`-gated).
- **`trait ResearchProvider`** (`research.rs:41`): `name()`, `search(query, limit) ->
  Result<Vec<WebSource>>`.
- **Backends reais:**
  - **`WikipediaProvider::new(lang)`** (`research.rs:138`) — **KEYLESS**, rede real
    (API pública `/w/api.php?action=query&list=search`; `strip_html` do realce).
    Fonte secundária real e atribuível (título + URL canônica + trecho verbatim).
  - **`TavilyProvider::new(key)`** (`research.rs:200`) — **BYOK** (chave
    `research.tavily`; POST `api.tavily.com/search`). Busca otimizada p/ LLM.
  - **`MockResearchProvider::canned()`** (`research.rs:101`) — **SEM rede**, 2 fontes
    fixas (timestamp determinístico) → prova determinística no CI.
  - **`build_research_provider(backend, key: Option<String>, lang: &str) ->
    Result<Box<dyn ResearchProvider>>`** (`research.rs:268`): `"mock"`→mock;
    `"wikipedia"|"wiki"`→Wikipedia (**chave ignorada**); `"tavily"`→**exige chave**
    (`Err(AiError::NoKey("research.tavily"))` se `None`); `other`→`UnknownProvider`.

### 1.3 Como `study()` consome `web_sources` (anti-alucinação embutida no core)
- `StudyRequest.web_sources: Vec<WebSource>` (`study.rs:53`). **Hoje a fronteira F3.3
  passa `vec![]`.** Habilitar a pesquisa = a fronteira **buscar** as fontes
  (`build_research_provider(...).search(query, limit)`) e **passá-las** ao `study()`.
- Dentro de `study()` (`study.rs:294`, `embedded`-only): se `web_sources` não vazio,
  injeta no prompt um bloco **"FONTES SECUNDÁRIAS (recuperadas da web — cite por
  `[W:n]`; use SOMENTE estas, NÃO invente URLs…)"** (`study.rs:200-217`); após a
  resposta, **`cited_web_indices`** valida cada `[W:n]` — fora do intervalo →
  **warning** (`study.rs:308-317`, anti-fabricação); e
  **`CitationCollector::from_web_results(&req.web_sources)`** (`study.rs:322`) monta
  as citações **das URLs/fontes, NUNCA do modelo**. → A anti-alucinação da pesquisa
  web já é **lei no core**: o LLM só cita `[W:n]`; o trecho vem verbatim da fonte.
- **Importante:** a **busca em si** (chamar `build_research_provider` + `.search()`) é
  do CHAMADOR e é `embedded`/`reqwest` → **rede opt-in, possivelmente com chave**.

### 1.4 A fricção study-embedded (dimensiona D2 — paridade web)
- **`study`/`StudyRequest`/`StudyResult` são `#[cfg(feature="embedded")]`**
  (`ai/mod.rs:41-42`, `study.rs:13/28/79/293`) e o `user_prompt` (montagem do estudo)
  é **`fn` privado `embedded`** (`study.rs:181`). `verified_lexicon` é `embedded`
  (rusqlite). `research` é `embedded` (reqwest). → **No web/wasm, NADA disso existe.**
- **Contraste (o que já é web-ready):** `ask`/`ask_context`/`numbered_passage`/
  `citation::rewrite_anchors` foram tornados **`ai-pure`** por PR (F2.7/**ADR-0024**),
  e o web os usa via **prepare→fetch→finalize** (F2.7b/**ADR-0025**). A **conversa**
  (`ask_session`/`refine_scope`/`parse_refinement`) **já é `ai-pure`** (F3.4) → **roda
  no web sem PR**. Só o **estudo profundo + léxico + [pesquisa]** falta no web.
- **Consequência para a paridade web do estudo:** diferente do `ask` (Fase 2), **não
  há entrada pública pura** para reproduzir a montagem exata do estudo no web — o
  `user_prompt`/`study` são privados/`embedded`. Logo o caminho app-side (espelhar em
  TS) **reconstruiria** a estrutura do estudo (bloco léxico, xref, `[W:n]`, verificação
  `[V:Strong]`) → **risco alto de drift** do anti-alucinação do estudo. O caminho de
  **zero drift** é um **PR ao core** que torne a superfície de `study` **`ai-pure`**
  (molde exato do ADR-0024) — que é a tarefa condicional **F3.11**.

## 2. As decisões (com Opções · Prós/Contras · Recomendação do Planner)

> O Planner recomenda, **não decide**. Cada decisão traz o trade-off honesto.

### D1 — Pesquisa web assistida (opt-in): habilitar? qual backend? qual política de chave?
`ai::research` faz busca web (rede, `reqwest`, `embedded`) → alimenta `study()` com
`web_sources` citados `[W:n]` (o core valida `cited_web_indices` e monta citações das
URLs, §1.3). É **OPCIONAL** — nada essencial do estudo exige rede.

- **(A) Habilitar via backend KEYLESS (Wikipedia)** — a fronteira nativa expõe uma
  flag `research: bool` (+ `research_backend="wikipedia"`) que, quando ligada, chama
  `build_research_provider("wikipedia", None, lang)` + `.search(query, limit)` e passa
  as `WebSource` ao `deep_study`.
  - **Prós:** rede **opt-in** sem chave (menor fricção BYOK); fontes reais **citáveis**
    e atribuíveis; anti-alucinação já garantida pelo core (`[W:n]` verificado, trecho
    verbatim); Wikipedia é fonte secundária neutra e gratuita.
  - **Contras:** rede em runtime (mesmo opt-in) → precisa de **UI de privacidade**
    (mostrar o que sai da máquina, VISION: cada consulta explícita); qualidade de
    busca inferior à otimizada-p/-LLM; Wikipedia tem viés/limites como fonte teológica.
- **(B) Habilitar via backend BYOK (Tavily)** — melhor busca, mas exige a chave
  `research.tavily` do usuário (molde BYOK F2.4/keystore/`SUPPORTED_PROVIDERS`).
  - **Prós:** busca otimizada p/ LLM (melhor recall/precisão); mesmo padrão BYOK já
    provado (secure-store nativo / session-only web, ADR-0025).
  - **Contras:** **mais uma chave** p/ o usuário obter/gerir; custo p/ o usuário;
    aumenta a superfície de segredo (nunca em git/log); rede opt-in obrigatória.
- **(C) Adiar a pesquisa web** — o estudo segue com passagem + léxico + xref (tudo do
  store) **sem web**; a pesquisa vira uma F3.x futura (com gate próprio).
  - **Prós:** **máxima pureza offline-first**; menor superfície; a plataforma de
    estudo já é rica sem web; sem UI de privacidade/rede nova nesta fase.
  - **Contras:** abre mão de um diferencial (fontes secundárias citáveis); só empurra
    a mesma decisão adiante.
- **(D) Combinação: Wikipedia default (keyless, opt-in) + Tavily opt-in (BYOK)** — o
  usuário liga a pesquisa (Wikipedia sem chave) e, se quiser, cola a chave Tavily.
  - **Prós:** melhor dos dois — entra sem fricção (keyless) e escala p/ quem quer
    qualidade (BYOK); ambos já suportados por `build_research_provider`.
  - **Contras:** **maior superfície de UI/estado** nesta fase (seletor de backend +
    política de chave + privacidade); mais a validar na F3.10.

**Recomendação do Planner:** **(A) Wikipedia keyless, opt-in** como habilitação
mínima e honesta — entra sem nova chave, com anti-alucinação já garantida pelo core,
e mantém offline-first (desligada por padrão). Deixar **Tavily (B) como opt-in
registrado** para quem quiser qualidade (molde BYOK já pronto), sem torná-lo
requisito — ou seja, **(D) na prática, começando por A**. Não recomendo **(C)** só se
o sign-off quiser encerrar a Fase 3 sem rede de pesquisa; é legítimo (mais puro), ao
custo de adiar o diferencial. **Requisito transversal a qualquer "sim":** UI de
privacidade (consulta explícita, mostrar o que sai) e opt-in real (padrão desligado).

### D2 — Paridade WEB do estudo/léxico/conversa (F3.12): qual estratégia?
`study()`/`StudyRequest`/`StudyResult`/`verified_lexicon` são **`embedded`-only** e o
`user_prompt` é **privado** (§1.4); no web não rodam. A **conversa já é pura** (F3.4).

- **(A) PR ao core p/ tornar as partes puras do estudo/léxico `ai-pure`** (molde
  exato F2.7/**ADR-0024**) — desacoplar `study`/`StudyRequest`/`StudyResult`/
  `to_academic_markdown` de `research::WebSource`/`reqwest` e a recuperação de léxico
  de `rusqlite`, de modo que a **montagem/citação do estudo** compile em `wasm32`;
  + orquestração TS no web (prepare/fetch/finalize, molde F2.7b/**ADR-0025**), com o
  léxico recuperado do **store web** e os tipos/verificação vindos do Rust.
  - **Prós:** **fonte única, ZERO drift** do anti-alucinação do estudo (o mais
    sensível do produto); espelha o sucesso da Fase 2 (ADR-0024/0025); o web consome
    a MESMA impl de montagem/citação do nativo.
  - **Contras:** **refactor mais pesado** que o `ask` (o estudo combina store+léxico+
    xref+`[W:n]`+verificação `[V:Strong]`+seções); **toca o `the-light`** → **PR +
    ADR + push/merge humano + re-pin** (a tarefa condicional **F3.11**, bloqueante).
- **(B) Estudo web nativo-only por ora** (adiar F3.12 do estudo profundo) — a leitura/
  busca/xref/notas web (Fase 1) e o `ask` ancorado web (F2.7b) **já existem**; a
  **conversa** (`ask_session`) também é web-ready; o **estudo profundo** fica
  "premium nativo" nesta fase.
  - **Prós:** **menor risco/custo**; a lógica sensível do estudo fica numa só impl
    (Rust nativo); consistente com o ritmo nativo-primeiro; fecha a Fase 3 sem
    prometer no web algo ainda não validado com chave real; **dispensa a F3.11**.
  - **Contras:** web sem estudo profundo/léxico/export nesta fase (adiado p/ gate
    futuro); a comparação multi-IA e o export acadêmico web também esperam.
- **(C) Orquestração TS parcial** — subir ao web **só o que já é `ai-pure`**
  (conversa `ask_session`/`refine_scope`/`parse_refinement`, e o `ask` da F2.7b); o
  **estudo profundo completo** (léxico + citações + export) fica adiado.
  - **Prós:** entrega **conversa ancorada no web** já (baixo custo, zero PR); mantém a
    anti-alucinação numa só impl; passo incremental honesto.
  - **Contras:** paridade web **parcial** (sem estudo profundo/léxico/export); pode
    confundir expectativa ("estudo no web" só conversa, não o painel completo).

**Recomendação do Planner:** se a prioridade for **encerrar a Fase 3 com baixo risco**,
**(B) adiar o estudo web** é o caminho mais seguro (o web já tem leitura/busca/xref/
notas/`ask`; a conversa é bônus barato via C). Se a prioridade for **paridade real de
estudo profundo no web sem drift**, então **(A) PR `ai-pure`** é a via correta
(molde já validado 2× na Fase 2), ao custo do handoff humano da **F3.11**. **Não
recomendo espelhar o estudo em TS app-side** (a via "b" citada no backlog): sem
entrada pública pura, reconstruir a montagem do estudo em TS **duplica o
anti-alucinação com alto risco de drift** — precisamente o que ADR-0024/0025
evitaram. Sugestão pragmática: **(C) subir a conversa web agora + (A) PR para o estudo
profundo web** quando o sign-off quiser paridade total.

### D3 — Validação real multi-IA (F3.10): quando/como (confirmar como gate de chave real)
A comparação de **conteúdo real** de Claude/GPT/Gemini e o estudo profundo real
**precisam de chave** (segredo) + rede → o loop **não** roda isso sozinho (molde F2.6).

- **Confirmar:** **F3.10 é o gate bloqueante de chave real** — validação humana, no
  device, com a chave do usuário (do secure-store F2.4), rede opt-in. Cobre:
  **(i)** `deep_study(..., provider real)` — estudo ancorado + léxico Strong do store
  + [pesquisa web se D1 aprovar] + citações verificáveis; **(ii)** `ask_session`
  (conversa) real; **(iii)** a **comparação multi-IA real** (o conteúdo lado a lado
  que a F3.7 provou só em MOCK). O texto citado e as glosas continuam do **store**; só
  a interpretação vem do modelo; Strong/`[W:n]` inventado é sinalizado; a chave não
  vaza em log. O MOCK (F3.3/F3.5/F3.7/F3.8) já provou o determinístico.
- **Molde:** harness estilo `ask_real.rs`/F2.6 (validação manual documentada), ≥1
  provedor real em ≥1 alvo nativo.

**Recomendação do Planner:** **confirmar F3.10 como o gate de chave real**, cobrindo
(i)+(ii)+(iii) acima, com `gate: true` (HALT p/ o humano). É a contraparte real do que
a Fase 3 provou em MOCK; o escopo de pesquisa web dentro dela depende de D1.

## 3. Recomendação integrada do Planner (a combinação sugerida)

**D1 = (A) Wikipedia keyless, opt-in** (Tavily BYOK como opt-in registrado; padrão
desligado; UI de privacidade obrigatória) · **D2 = (C)+(A)** — subir a **conversa
web** agora (já `ai-pure`, zero PR) e, para o **estudo profundo web sem drift**,
adotar o **PR `ai-pure`** (F3.11, molde ADR-0024) **se** o sign-off quiser paridade
total; caso contrário **(B) adiar o estudo web** e fechar a Fase 3 nativa · **D3 =
confirmar F3.10** como o gate de validação real (chave do usuário), cobrindo estudo +
conversa + comparação reais.

**Por quê, em conjunto:** mantém **offline-first** (pesquisa e estudo web = opt-in
aditivo; a Fase 1 e o estudo nativo seguem offline), mantém a **anti-alucinação numa
única implementação (Rust)** — evitando o drift TS↔Rust do estudo — e respeita o
precedente já sancionado 2× na Fase 2 (D1 keyless-primeiro; paridade web só via
`ai-pure`/PR, nunca espelhando o anti-alucinação em TS). É o mesmo ritmo que fechou as
Fases 1 e 2 (nativo-primeiro → gate → web).

**Ressalva honesta ao sign-off:** se a prioridade for **encerrar a Fase 3 rápido/sem
risco**, **D1=C (adiar pesquisa)** + **D2=B (adiar estudo web)** são legítimos — a
plataforma nativa já está completa. Se a prioridade for **paridade web total já**,
então **D2=A** exige o **PR à `the-light` (F3.11)** — ação humana. O Planner **não**
decide isso.

## 5. Consequência

  - **F3.11** (PR `ai-pure` ao core, condicional) — **só se D2=A**; spec em

  - **F3.12** (paridade web) — estudo profundo + léxico + conversa + export no browser
    **conforme D2** (via re-pin F3.11, ou só a parte web-ready se B/C).
  - **F3.13** (Marco 3, `gate: true`) — plataforma de estudo profundo completa.
- Se a pesquisa web (D1=A/B/D) for habilitada, a fronteira nativa passará a **buscar**
  as `WebSource` (`build_research_provider(...).search(...)`) e passá-las ao
  `deep_study` (hoje `web_sources: vec![]`) — com prova por **MOCK**
  (`MockResearchProvider::canned()`, sem rede) no CI; a rede real é a F3.10.

## Restrições (não negociáveis — valem em QUALQUER opção)

- **Offline-first:** a pesquisa web e o estudo web são **opt-in e ADITIVOS** — a
  Fase 1 e o **estudo nativo** seguem 100% offline. A pesquisa web nasce **desligada**;
  a **única** rede nova em runtime é a consulta de pesquisa (opt-in, explícita) e a

- **Anti-alucinação é LEI:** o texto do versículo, as **glosas/léxico** e as citações
