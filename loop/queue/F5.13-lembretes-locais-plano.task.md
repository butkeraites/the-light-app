## Objetivo

Add optional daily LOCAL reminders for the active reading plan using on-device scheduling — strictly offline, opt-in, no server/account/push token. Native uses expo-notifications local scheduling; web degrades gracefully (Notification API best-effort or documented no-op). Reminder preference is stored app-side (NOT in the core active.json).

> **Herda F5.7 (UI de planos) + F5.2 (KV de prefs offline + i18n).** Lembrete = notificação
> LOCAL do device (opt-in, OFF por padrão) do plano ativo. Preferência (on/off + horário) no KV
> da F5.2. `expo-notifications` = dep nova; agenda SÓ local (`scheduleNotificationAsync`), sem
> servidor/push token/conta. Corpo = cromo i18n + nome/dia do plano (core). Web: notifications é
> nativo → degrada (toggle escondido/desabilitado no web, sem quebrar o build).

## Passos sugeridos

1. Add `expo-notifications` to `app/package.json` (compatible with the installed Expo SDK) and, if needed, its config plugin to `app.json`; keep it OPT-IN (no permission requested until the user enables reminders).
2. Create `app/lib/planReminders.ts` (native + `.web.ts` pair): `enableReminder(timeHHMM)` requests LOCAL notification permission then schedules a DAILY repeating local notification (`Notifications.scheduleNotificationAsync` with a daily calendar trigger) titled after the active plan / today's reading; `disableReminder()` cancels it; `getReminder()` reads the stored pref. NO push tokens, NO network, NO account.
3. Store the reminder pref (on/off + HH:MM) app-side ONLY — e.g. a small `reading-plans/reminder.json` written via expo-file-system (native) / OPFS (web) or AsyncStorage — SEPARATE from the core's active.json (do not extend the core PlanProgress struct).
4. Add a reminder toggle + time picker to the Plans screen (`app/app/plans/index.tsx`), visible only when a plan is active; cancel the schedule on `clearReadingPlan`.
5. Web `.web.ts`: best-effort `Notification` API (if permission granted) or a clearly-documented no-op fallback that never throws and never requires network — plans stay fully usable without reminders.
6. Add a deterministic unit/headless check that MOCKS expo-notifications and asserts enable→one daily schedule call with the chosen time, disable→cancel call, and that NO network/push-token API is invoked (no real notification fired in CI).

## Restrições

OFFLINE-FIRST is the hard constraint: local scheduling only, no server/account/push. expo-notifications LOCAL notifications work fully offline on-device. Can run in PARALLEL with rp-4 (both depend only on rp-3). Adds a runtime dependency (expo-notifications) — an app-side change, NOT a the-light change. Web reminders are inherently limited; keep them best-effort so nothing essential requires them (offline-first preserved). Possible ADR (lembretes locais opt-in: expo-notifications local, pref app-side, web best-effort).

- **Offline-first (crítico p/ lembretes):** APENAS notificações LOCAIS agendadas no device (`expo-notifications` `scheduleNotificationAsync`) — SEM servidor, SEM conta, SEM push token remoto. Opt-in (OFF por padrão). A preferência (on/off + horário) persiste no KV offline da F5.2.
- **BYOK/privacidade:** nenhuma chave; nenhum dado sai do device; nada logado.
- **Anti-alucinação:** o corpo da notificação é CROMO i18n (`t()`) + nome/dia do plano do core; nunca texto bíblico inventado.
- **the-light só via PR+ADR:** `the-light-core` pinado (`2fc2dab`); app-side. `expo-notifications` é dep nova (adicionar ao package.json).
- **Qualidade:** `tsc --noEmit`; expo export web 0 (web degrada — notifications é nativo); prova (self-test/mock do agendamento, sem disparar de fato); sem regressão; permissão pedida só no opt-in.
