## Objetivo

Expor na **fronteira UniFFI** (`core/src/lib.rs`) uma função de **referências
cruzadas** que **delega** ao `the-light-core` (rev pinado `8f66004`): dado um
`db_path` e um versículo (`book`/`chapter`/`verse`), retorna as referências-alvo
relacionadas como `Vec<CrossRef>` — cada uma com a **referência** de destino
(book/chapter/verse, podendo ser intervalo) e o **número de votos** da comunidade.
O filtro por **`min_votes`** (ocultar referências disputadas, votos negativos) e o
limite vêm **de graça do core**, não são reimplementados aqui.

Nada de SQL/lógica de xref é reimplementado na fronteira: ela apenas **adapta
tipos e erros** do core para um Record UniFFI (`CrossRef`), seguindo a regra "uma
fonte da verdade". Mesmo molde da **F0.9/F1.2/F1.5/ADR-0010**: a **exportação**
existe em **todos** os alvos (forma da fronteira uniforme p/ o `ubrn` extrair
metadata no host), mas **só o corpo que toca store/xref/`rusqlite`** é
`#[cfg(not(target_arch = "wasm32"))]` (no web, stub que retorna `CoreError`). Assim
`rusqlite`/`the-light-core` com `embedded` **não entram no grafo wasm** e o build
web (F0.6b/F0.10) continua puro.

Esta é uma tarefa de **NÚCLEO** (fronteira Rust + teste de host), análoga à
F1.2/F1.5. A **UI de xref** + a **atribuição CC-BY visível** são a **F1.9** (não
fazem parte desta tarefa): a fronteira apenas **entrega os dados**; a atribuição
`Cross references courtesy of OpenBible.info (CC-BY)` (registrada em ADR-0016) é
exibida pela UI da F1.9.

    escopo primário** desta tarefa (precisa de `Lang`/agregação por capítulo; é

    é requisito de aceite**.
- `model.rs` (**PURO**, presente em **todos** os alvos): `Reference`,

**DIVERGÊNCIA CRÍTICA vs. F1.5 (ler com atenção):**
1. **Delegação por função livre + `&Connection`, não por método de `BibleSource`.**
   O `EmbeddedSource`/trait `BibleSource` **NÃO** têm método de xref (têm
   `translations`/`passage`/`search`/`chapter_count`, mas **não** `cross_refs`). A
   xref é a **função livre** `the_light_core::xref::for_verse(conn, …)`, que recebe
   uma **`&Connection`** crua. Logo o caminho de delegação é: `Store::open(db_path)`
   → `xref::for_verse(store.conn(), book, chapter, verse, min_votes, limit)` (o
   método `Store::conn()` é **público** — o próprio core o usa em `xref.rs::tests` e
   em `EmbeddedSource::new`). Isso **ainda é delegação ao core** (o SQL/ranking vive
   em `xref.rs`); a fronteira **não** escreve SQL. **NÃO** reimplementar a query
   `cross_references` no app.
2. **O `From<the_light_core::xref::CrossRef>` PRECISA ser gated `cfg(not(wasm32))`.**
   Diferente de `SearchHit`/`Verse`/`Passage` (cujo tipo-fonte está em `model`,
   **puro**, todos os alvos), o tipo-fonte `the_light_core::xref::CrossRef` vive no
   módulo `xref`, que é **`#[cfg(feature = "embedded")]`** → **só existe no nativo**.
   Portanto:
   - O **Record** `CrossRef { reference: Reference, votes: i64 }` da fronteira pode
     (e deve) ser definido em **todos** os alvos — ele só referencia `Reference`
     (puro) e `i64`, nenhum tipo `embedded`.
   - Mas o `impl From<the_light_core::xref::CrossRef> for CrossRef` **deve** ficar
     sob `#[cfg(not(target_arch = "wasm32"))]` (senão o build web quebra: o tipo-
     fonte não existe no wasm). Alternativa equivalente: fazer o mapeamento
     **inline** dentro do corpo gated da função (`CrossRef { reference:
     c.reference.into(), votes: c.votes }`), sem `impl From`. **Recomendado:** o
     `From` gated, para manter o padrão "Records via From". O Record em si **não**
     pode arrastar `xref` p/ o wasm.

**Gating de módulos do core (confirmado em `the-light-core/src/lib.rs`):**
`model`/`reference` são **puros**; `xref`/`search`/`source`/`store` são
**`#[cfg(feature = "embedded")]`** (só no nativo). Portanto a construção/chamada a
`Store::open`, `store.conn()`, `xref::for_verse`, `xref::DEFAULT_MIN_VOTES` e
`xref::DEFAULT_LIMIT` só pode aparecer dentro do bloco
`#[cfg(not(target_arch = "wasm32"))]`. O **stub web** não pode tocar nenhum desses
símbolos (consome os args com `let _ = (...)`).

**Função a expor (assinatura recomendada — molde de `search` da F1.5):**
- `cross_refs(db_path: String, book: u8, chapter: u16, verse: u16,
  min_votes: Option<i64>, limit: Option<u32>) -> Result<Vec<CrossRef>, CoreError>`
  - corpo gated `cfg(not(wasm32))`: `Store::open(db_path)` (mapeia erro p/
    `CoreError`) → `xref::for_verse(store.conn(), book, chapter, verse,
    min_votes.unwrap_or(the_light_core::xref::DEFAULT_MIN_VOTES),
    limit.map(|l| l as usize).unwrap_or(the_light_core::xref::DEFAULT_LIMIT))`
    (mapeia o `rusqlite::Error` p/ `CoreError`) → mapear cada
    `xref::CrossRef` p/ o Record `CrossRef`.
  - **Defaults documentados:** `min_votes: None` → `DEFAULT_MIN_VOTES` (= **1**,
    **oculta disputadas/negativas** por padrão); `limit: None` → `DEFAULT_LIMIT`
    (= **20**). Documente isso no doc-comment (a UI/F1.9 pode pedir um threshold
    menor para ver as disputadas).
  - **Sem parâmetro `translation`:** a xref é **independente de tradução** (chaveada
    por book/chapter/verse). Não há caminho `UnknownTranslation`; um versículo sem
    xref → `Vec` **vazio** (não erro, não panic). Erros só de I/O (`Store::open` com
    path inválido) ou `rusqlite`.
  - stub web: `let _ = (db_path, book, chapter, verse, min_votes, limit);` →
    `Err(CoreError::Generic { message: "store local indisponível no alvo web
    (F0.10: wa-sqlite+OPFS)".to_string() })` (mesma mensagem-padrão das outras).

**Novo Record UniFFI (definir em TODOS os alvos; só tipos puros):**
- `CrossRef { reference: Reference, votes: i64 }` — `reference` via o `From` já
  existente de `model::Reference`; `votes: i64` (UniFFI suporta `i64`). O `From`
  **a partir de** `the_light_core::xref::CrossRef` é **gated** `cfg(not(wasm32))`
  (ver Divergência #2). Documente no doc-comment: o que `votes` significa (votos da
  comunidade OpenBible; maior = mais relevante; negativos = disputados) e que a
  **atribuição CC-BY** é exibida na **F1.9** (a fronteira só entrega os dados).

**Schema `cross_references` (uma fonte da verdade — migração `v1_initial.sql`,
aplicada por `Store::open`):**
`cross_references(from_book, from_chapter, from_verse, to_book, to_chapter,
to_verse_start, to_verse_end, votes)` + `idx_xref_from`. **Sem coluna de
licença/atribuição** (a atribuição CC-BY vive em ADR-0016 / UI F1.9). `votes` podem
ser **negativos** (refs disputadas). A tabela **já existe (vazia)** após
`Store::open` (mesmo princípio do `verses_fts` na F1.5) → o fixture só precisa
**popular** as linhas à mão.

**Banco de teste (determinismo vs. corpus completo) — mesmo princípio da F1.5:**

- **Conteúdo recomendado do fixture** (xrefs de domínio/estrutura **de** João 3:16 =

  `from_b,from_c,from_v,to_b,to_c,to_vs,to_ve,votes`):
  - `(43,3,16, 45,5,8,8, 50)` — **Romanos 5:8** (book 45), muito votada (alvo
    principal da asserção).
  - `(43,3,16, 43,3,15,15, 30)` — **João 3:15** (xref de estrutura/contexto).

## Restrições

- **NÃO modificar, forkar, copiar ou vendorizar `the-light`.** Consumo só pela git
  dep pinada `8f66004`. A xref é **delegada** a `the_light_core::xref::for_verse`
  (com `store.conn()` de `Store::open`). Se a API real do core **não** permitir
  expor a xref sem mudar o core → **não invente**: marque a tarefa `blocked` com o
  **erro/limitação EXATA** (ex.: `Store::conn()` não público, `xref` não acessível)
  e a decisão necessária (PR+ADR ao core).
- **NÃO reimplementar SQL/lógica de xref** na fronteira. Proibido escrever
  `SELECT … FROM cross_references`, `ORDER BY votes`, montar o filtro `votes >=
  min_votes`, etc. — isso vive no core (`xref.rs`). (INSERTs em `cross_references` no
  **setup do teste** são fixture, **não** produto — como o próprio core faz em
  `xref.rs::seeded`.)
- **Gating obrigatório (atenção à divergência #2):** o corpo que toca
  `xref`/`store`/`rusqlite` **deve** ficar sob `#[cfg(not(target_arch="wasm32"))]`
  com stub web; e o `From<the_light_core::xref::CrossRef>` (cujo tipo-fonte é
  `embedded`-only) **também** deve ser gated `cfg(not(wasm32))` (ou mapeado inline no
  corpo gated). O **Record** `CrossRef` só pode referenciar tipos **puros**
  (`Reference`/`i64`). **Não regredir** o build web (sem `rusqlite` no grafo wasm). A
  exportação UniFFI permanece em todos os alvos.
- **Licenciamento/atribuição:** os dados de xref são **CC-BY** (OpenBible.info, ver
  ADR-0016). A **fronteira só entrega os dados** (referência + votos); a **string de
  atribuição** `Cross references courtesy of OpenBible.info (CC-BY)` é exibida pela
  **UI da F1.9** (não nesta tarefa). Registre no doc-comment/result que a fronteira
  entrega os dados e a atribuição visível é responsabilidade da F1.9 — **não** omita
  a obrigação CC-BY do produto.
- **Offline-first:** nenhuma rede em runtime nem no teste; o teste **primário** usa
  fixture local determinístico (não depende de `bible.sqlite` presente nem de rede).
  Lembrete: a tabela `cross_references` é criada (vazia) por `Store::open` — o
  fixture **tem** de inserir as linhas à mão (não há trigger/seed automático).
- **Anti-alucinação:** xref é só **referência** (book/chapter/verse + votos) — nenhum
  texto bíblico é gerado nem necessário aqui. Os dados vêm do store local (populado
  pelo importador canônico CC-BY na F1.7).
- **Sem segredos.** Não tocar chaves/credenciais.
- Em conflito com qualquer regra acima → **`blocked`/HALT** com o motivo exato, sem
  improviso.
