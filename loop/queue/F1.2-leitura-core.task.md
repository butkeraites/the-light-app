## Objetivo

Expor na **fronteira UniFFI** (`core/src/lib.rs`) funções de **leitura/navegação**
que **delegam** ao `the-light-core` (rev pinado `8f66004`), permitindo navegação
programática completa: **listar versões**, **listar livros**, **abrir um capítulo
inteiro numerado por versículo** e **contar capítulos de um livro**. Nada de SQL,
parsing ou texto bíblico é reimplementado aqui — a fronteira apenas **adapta tipos
e erros** do core para Records/Enums UniFFI (regra "uma fonte da verdade").

Segue o padrão da **F0.9/ADR-0010**: a **exportação** das funções existe em
**todos os alvos** (forma da fronteira uniforme p/ o `ubrn` extrair metadata no
host), mas **apenas o corpo que toca o store/`rusqlite`** é
`#[cfg(not(target_arch = "wasm32"))]` (no web, stub que retorna `CoreError`). Assim
o `bible.sqlite`/`rusqlite` **não entram no grafo wasm** e o build web (F0.6b/F0.10)
continua puro.

## Contexto / arquivos

**Fronteira (este repo) — onde implementar:**
- `core/src/lib.rs` — adicionar as funções e os novos Records/Enums. Reusar o
  padrão já presente em `get_passage` (export em todos os alvos + corpo gated +
  stub web). Reusar `CoreError`, `Reference`, `VerseRange`, `Verse`, `Passage` e
  seus `From<the_light_core::model::…>`.
- `core/Cargo.toml` — **não precisa mudar**: a matriz por alvo já liga `embedded`
  só no não-wasm (`[target.'cfg(not(target_arch="wasm32"))'.dependencies]`).
- `core/examples/gen_sample_db.rs` — referência do padrão "schema via migrações do
  core + DML de dado público KJV verbatim" (útil p/ o fixture de teste).

**API REAL do core a delegar (já descoberta na fonte — confira antes de codar em
`/Users/butkeraites/Documents/the-light/crates/the-light-core/src/`):**
- `source/mod.rs` → trait `BibleSource` (traga em escopo p/ chamar os métodos):
  - `fn translations(&self) -> Result<Vec<model::Translation>>` — **lista versões
    do banco** (lê tabela `translations`). DB-backed → corpo gated.
  - `fn passage(&self, r: &Reference, t: &TranslationId) -> Result<Passage>` — já
    usado por `get_passage`; aceita `VerseRange::WholeChapter` (capítulo inteiro,
    versículos ordenados). DB-backed → corpo gated.
- `source/embedded.rs` → `EmbeddedSource`:
  - `EmbeddedSource::new(&store)`; `store.rs` → `Store::open(path)` migra o schema.
  - `EmbeddedSource::chapter_count(&self, book: u8, t: &TranslationId) -> Result<u16>`
    — **nº de capítulos presentes no banco** (`SELECT max(chapter) … WHERE
    translation_id=? AND book_number=?`; `0` se ausente). DB-backed → corpo gated.
- `reference.rs` (módulo **PURO**, disponível em **todos** os alvos, inclusive
  wasm — não exige `embedded`):
  - `reference::BOOKS: [BookInfo; 66]` — tabela canônica dos 66 livros. `BookInfo`
    tem: `number: u8`, `name_en: &str`, `name_pt: &str`, `abbrev_en: &str`,
    `abbrev_pt: &str`, `testament: model::Testament`, `aliases`.
  - `reference::book_info(n) -> Option<&BookInfo>`, `reference::chapters_in_book(n)
    -> u16` (capítulos **canônicos**, version-independent; João=21), `CHAPTERS`.
- `model.rs` (PURO): `Translation { id: TranslationId, abbrev, name, language:
  Lang, license: License, embeddable: bool }`; `Lang { Pt, En }` (`Lang::code()` →
  `"pt"`/`"en"`); `License` (`License::as_str()`); `Testament { Old, New }`;
  `Reference::whole_chapter(book, chapter)`.

**Funções a expor (assinaturas recomendadas):**
- `list_translations(db_path: String) -> Result<Vec<Translation>, CoreError>`
  → corpo gated: `Store::open` + `EmbeddedSource::new` + `BibleSource::translations`.
- `list_books() -> Vec<Book>` → **PURO** (delegado a `reference::BOOKS` +
  `reference::chapters_in_book`). **Sem** `db_path`/`translation`, **sem** gating
  (não toca store) → funciona inclusive no wasm.
- `get_chapter(db_path: String, translation: String, book: u8, chapter: u16) ->
  Result<Passage, CoreError>` → corpo gated: `Reference::whole_chapter(book,
  chapter)` + `BibleSource::passage`; reusa o Record `Passage`.
- `chapter_count(db_path: String, translation: String, book: u8) ->
  Result<u16, CoreError>` → corpo gated: `EmbeddedSource::chapter_count`.

**Novos Records/Enums UniFFI (definir em TODOS os alvos, como `Verse`/`Passage`;
os `From` referenciam só `model`/`reference`, que são puros e existem no wasm):**
- `Translation { id, abbrev, name, language, license, embeddable }` com
  `From<the_light_core::model::Translation>` (mapeie `language` via
  `Lang::code()` → `String`; `license` via `License::as_str()` → `String`).
- `Book { number, name_en, name_pt, abbrev_en, abbrev_pt, testament, chapter_count }`
  com `From<&the_light_core::reference::BookInfo>` (use
  `reference::chapters_in_book(number)` p/ `chapter_count`).
- `Testament { Old, New }` (`uniffi::Enum`) com
  `From<the_light_core::model::Testament>`.

**DIVERGÊNCIA REGISTRADA vs. rascunho do backlog (`list_books(translation)`):** o
core **não expõe** método p/ listar a tabela `books` por tradução **sem
reimplementar SQL** (`EmbeddedSource`/`BibleSource` não têm `books()`). A única
delegação sem-SQL é a tabela canônica **pura** `reference::BOOKS` (mesma p/ KJV e
Almeida — o cânon protestante de 66 é version-independent; só o **nome de exibição**
muda por idioma, já disponível via `Book.name_en`/`name_pt` + a `language` da
`Translation`). Portanto `list_books()` é **sem parâmetro** e **puro**. Adicionar
um filtro por tradução exigiria reimplementar SQL (proibido) **ou** mudar o core
(PR+ADR, fora de escopo) → **não fazer**; manter a lista canônica. Documente esta
escolha no doc-comment da função e/ou em ADR.

**Banco de teste (determinismo vs. corpus completo):**
- `assets/data/sample.sqlite` (versionado) só tem **João 3:16** → insuficiente p/
  `get_chapter`/`chapter_count` plausíveis.
- `assets/data/bible.sqlite` (KJV 31.102 + ALM1911 31.101) **está presente agora**
  (gerado pela F1.1) mas é **gerar-ignorado / não versionado** (ADR-0013) e ~47 MB
  → **não** pode ser a única dependência do teste (ausente em clone limpo/CI, e
  regenerar exige rede — pode estar offline).
- **RECOMENDAÇÃO (use o mais determinístico e offline):** o teste de host constrói
  um **fixture pequeno-porém-multi-capítulo** num diretório temporário, no padrão
  do `gen_sample_db.rs` — `Store::open(tmp)` (schema = migrações do core) + DML de
  **domínio público KJV verbatim**, inserindo o mínimo p/ as asserções:
  `translations(kjv,…)`; João (43) cap. 3 **v16 e v17**; João (43) **cap. 21 v1**
  (p/ que `max(chapter)` do banco = **21**). Inserir via `INSERT` no **setup do
  teste** é fixture, **não** lógica de produto (o próprio core faz isso em
  `embedded.rs::seeded_store()`); as funções de produto só **delegam**. As
  asserções leem **de volta** pela fronteira (round-trip), igual à F0.9.
  - **OPCIONAL (bônus, não-determinístico):** se `assets/data/bible.sqlite`
    existir, rodar asserções extras contra ele (ex.: João 3 completo = 36
    versículos) **atrás de `if Path::new(BIBLE_DB).exists()`** — nunca como
    requisito do teste. As asserções **primárias** devem passar **só** com o
    fixture, **offline**.

**ADRs/decisões pertinentes:** ADR-0005 (matriz de features por alvo; `embedded`
só no nativo), ADR-0010 (store nativo + **gating de corpo** + anti-alucinação),
ADR-0013 (`bible.sqlite` gerado-ignorado). Estado do loop: `loop/STATUS.md`.

## Restrições

- **NÃO modificar, forkar, copiar ou vendorizar `the-light`.** Consumo só pela git
  dep pinada `8f66004`. Se a API real do core **não** permitir alguma leitura sem
  mudar o core (ex.: listar `books` por tradução sem SQL próprio) → **não invente**:
  use a delegação pura disponível (`reference::BOOKS`) **ou** marque a tarefa
  `blocked` com o **erro/limitação EXATA** e a decisão necessária (PR+ADR ao core).
- **NÃO reimplementar SQL/lógica de domínio** na fronteira. As funções de produto
  só **delegam** ao store/source/reference do core. (INSERTs de **fixture de
  teste** são permitidos — é setup, não produto, como em `embedded.rs`.)
- **Anti-alucinação:** todo texto de versículo vem **do store local**; nenhum texto
  é gerado. Só **domínio público** (KJV) no fixture/sample.
- **Gating obrigatório:** o corpo que toca `store`/`rusqlite` **deve** ficar sob
  `#[cfg(not(target_arch="wasm32"))]` com stub web; **não regredir** o build web
  (sem `rusqlite` no grafo wasm). A exportação UniFFI permanece em todos os alvos.
- **Offline-first:** nenhuma rede em runtime nem no teste; o teste **primário** usa
  fixture local determinístico (não depende de `bible.sqlite` presente nem de rede).
- **Sem segredos.** Não tocar chaves/credenciais.
- Em conflito com qualquer regra acima → **`blocked`/HALT** com o motivo exato, sem
  improviso.
