## Objetivo

Criar um **script reproduzível** (`scripts/gen-bindings.sh`) que **gera os bindings
TypeScript** a partir da fronteira UniFFI do crate `core/` (`the-light-app-core`),
populando o diretório `bindings/`. A ferramenta de geração é a escolhida na ADR-0001:
**`uniffi-bindgen-react-native`** (CLI **`ubrn`**, em transição para
`uniffi-bindgen-javascript`), que a F0.0 confirmou **NÃO instalada** neste ambiente.

Esta tarefa **instala e fixa** a versão do `ubrn`, cria a configuração da ferramenta
apontando para a fronteira do core, e entrega um script que **limpa e regenera**
`bindings/` sem erro. O escopo é **só a geração dos `.ts`** — ainda **não** é ligar o
core num app rodando (web/iOS/Android), o que pertence à F0.6/F0.7/F0.8.

## Contexto / arquivos

**Estado do core (F0.3 aceita, commit `9881c72`):**
- `core/Cargo.toml`: crate `the-light-app-core`, `crate-type = ["lib","cdylib","staticlib"]`,
  `uniffi = "0.31.2"` (modo *library* / proc-macros, sem UDL e sem `build.rs` — ADR-0003),
  `thiserror = "2.0.18"`, e git dep pinada `the-light-core` (`rev = "0888ac0..."`).
- `core/src/lib.rs` (modo library, `uniffi::setup_scaffolding!()`) exporta:
  - `ping() -> String`
  - `ping_checked(ok: bool) -> Result<String, CoreError>`
  - `parse_reference(input: String) -> Result<Reference, CoreError>`
  - tipos UniFFI: `#[uniffi::Record] Reference { book: u8, chapter: u16, verses: VerseRange }`,
    `#[uniffi::Enum] VerseRange { Single, Range, WholeChapter }`, `#[uniffi::Error] CoreError`.
  - Observação: a geração TS costuma converter `snake_case` → `camelCase` (ex.: a função
    pode sair como `parseReference`); o tipo/record `Reference` aparece como tal.

**Ferramenta (F0.0 — `loop/archive/F0.0.result.md`):** `which ubrn` e
`which uniffi-bindgen` **falham** — nada instalado. Esta tarefa **instala e fixa**.
A ADR-0001 fixou o gerador como `uniffi-bindgen-react-native` (CLI `ubrn`). A VISION §9
registra **risco #1 = maturidade do `ubrn`** (release inicial): mitigar provando na Fase 0,
exatamente aqui. A versão exata foi deliberadamente deixada para ser fixada na instalação
(ADR-0001/ADR-0003): **fixe-a agora** e registre no ADR-0004.

**Fluxo esperado do `ubrn` (genérico — confirme com `--help` da versão instalada):**
1. Instalar o `ubrn` (via `npm i -D uniffi-bindgen-react-native@<versão>` e/ou
   `cargo install uniffi-bindgen-react-native --version <X> --locked`, conforme o fluxo
   documentado da ferramenta). Rede em **dev/build é permitida** (offline-first é regra de
   **runtime do PRODUTO**, não do desenvolvimento).
2. Criar um arquivo de configuração do `ubrn` (ex.: `ubrn.config.yaml`) descrevendo o crate
   `the-light-app-core` (`core/`), o nome do módulo e o **diretório de saída dos `.ts` = `bindings/`**.
3. Gerar os bindings TS a partir da fronteira UniFFI. No **modo library**, o `uniffi-bindgen`
   extrai a metadata da **biblioteca compilada** (a `cdylib`/`staticlib` do crate); por isso
   pode ser necessário um `cargo build` **mínimo no host** antes de gerar os `.ts`. **NÃO** é
   preciso compilar `wasm32`/iOS/Android completos aqui (isso é F0.6/7/8) — faça apenas o
   **mínimo necessário** para a geração dos `.ts` ter sucesso e **documente** os comandos.

**O que é GERADO e IGNORADO vs. o que é VERSIONADO (leia com atenção — Executor e Reviewer):**
- A guarda do `.gitignore` da F0.1 ignora o **conteúdo gerado** de `bindings/`
  (`/bindings/*` + `!/bindings/.gitkeep`). Portanto **os `.ts` gerados NÃO são versionados,
  por design** — e o "working tree limpo" continua valendo justamente porque eles são
  ignorados. **Não** versione os `.ts`; **não** remova a guarda do `.gitignore`.
- A verificação de "`bindings/` populado" checa os arquivos **no FILESYSTEM** (existem e são
  não-vazios), **não** exige que estejam no git. Não se confunda: `find ... -name '*.ts'`,
  não `git ls-files`.
- O que **É versionado/committado por esta tarefa:** `scripts/gen-bindings.sh`, o arquivo de
  config do `ubrn` (ex.: `ubrn.config.yaml`), `package.json`/`package-lock.json` (se a instalação
  for via npm — para **fixar** a versão), um `tsconfig.json` (apenas **se** você optar pelo
  passo opcional do `tsc`), a atualização do `DECISIONS.md` (ADR-0004) e, se preciso, uma
  extensão do `.gitignore` (se a ferramenta emitir artefatos fora de `bindings/`).
  `node_modules/` já é ignorado (F0.1).

## Passos sugeridos

1. **Instalar + FIXAR o `ubrn`.** Siga o fluxo documentado da ferramenta (npm e/ou cargo).
   Fixe a **versão EXATA** (sem `^`/`~`/`*`): se npm, no `package.json` + `package-lock.json`;
   se cargo, registre a versão exata no ADR-0004. Confirme a instalação com
   `ubrn --help` (ou `npx uniffi-bindgen-react-native --help`) e **anote a versão exata**.
2. **Criar a config do `ubrn`** (ex.: `ubrn.config.yaml`) apontando para o crate
   `the-light-app-core` (`core/`), definindo o nome do módulo e a saída dos `.ts` em
   `bindings/`. Consulte `ubrn --help`/docs da versão instalada para as chaves exatas.
3. **Criar `scripts/gen-bindings.sh`** (executável: `chmod +x`), com `set -euo pipefail`, que:
   - resolve a raiz via `git rev-parse --show-toplevel`;
   - **LIMPA** `bindings/` (remove os `.ts` gerados, **preservando `bindings/.gitkeep`**);
   - faz o **mínimo necessário** para a geração (ex.: `cargo build -p the-light-app-core` no
     host, produzindo a `cdylib`/`staticlib` de onde a metadata UniFFI é lida);
   - invoca o `ubrn` para **gerar os bindings TS** dentro de `bindings/`;
   - sai com código **0** em sucesso e **≠0** em erro.
   - No topo do script, **documente** (comentário) os comandos exatos usados.
4. **Rodar `./scripts/gen-bindings.sh`** e validar que `bindings/` ficou populado com `.ts`
   **não-vazios** que **referenciam a API do core** (`parse_reference`/`parseReference`,
   `Reference`, `ping`). Itere config/comandos até a geração sair limpa. Se a ferramenta emitir
   artefatos **fora** de `bindings/`, **estenda o `.gitignore`** para ignorá-los (mantendo a
   guarda existente) — isso é uma mudança versionada legítima.
5. **(OPCIONAL, robusto)** Se quiser validar tipos: inclua um `tsconfig.json` mínimo +
   `typescript` como devDependency e adicione um passo `tsc --noEmit` **apenas como validação
   extra** — **nunca** como ponto de falha frágil. Se **não** incluir toolchain TS, **não**
   exija `tsc` (ele nasce plenamente com o app na F0.5). Registre a escolha no ADR-0004.
6. **Registrar ADR-0004** em `DECISIONS.md`: ferramenta (`uniffi-bindgen-react-native`/`ubrn`),
   **versão exata fixada**, mecanismo de instalação (npm/cargo + lockfile), forma de invocação
   (config + comando), decisão sobre `tsc` (incluído/adiado) e nota de **compatibilidade com
   `uniffi 0.31.2`** (ADR-0003). Atualize a linha do gerador na tabela de ambiente
   (de "NÃO INSTALADO" → versão fixada).
7. **Commit com prefixo do ID**, ex.:
   `build(F0.4): script reproduzível de geração de bindings TS via ubrn (ADR-0004)`.
   Versione **apenas**: `scripts/gen-bindings.sh`, `ubrn.config.yaml`,
   `package.json`/`package-lock.json` (se npm), `tsconfig.json` (se criado), `.gitignore` (se
   estendido) e `DECISIONS.md`. **NÃO** versione os `.ts` gerados nem `node_modules/`.

- [ ] Ao rodar `./scripts/gen-bindings.sh`, ele **LIMPA e REGENERA** `bindings/` e sai com

- [ ] `bindings/` fica **populado no FILESYSTEM** com ≥1 arquivo `.ts` **não-vazio**, e os `.ts`

  (Os `.ts` são **ignorados pelo git por design** — não precisam estar versionados.)
- [ ] **ADR-0004** registrado em `DECISIONS.md` com a versão exata do `ubrn` e o mecanismo de
  instalação; a linha do gerador na tabela de ambiente atualizada.
- [ ] **Working tree limpo** após o commit, **considerando o `.gitignore`**: os `.ts` gerados e
  `node_modules/` são ignorados; apenas os arquivos **VERSIONADOS** (script, config, manifestos,
  `DECISIONS.md`) foram committados. `git status --porcelain` vazio.
- [ ] **(Se aplicável)** `tsc --noEmit` OPCIONAL passou — **só** se `tsconfig`+`typescript`

## Comando(s) de verificação

```sh
set -e
ROOT="$(git rev-parse --show-toplevel)"

# 1) Script existe e é executável
test -x "$ROOT/scripts/gen-bindings.sh" \
  && echo "OK  scripts/gen-bindings.sh existe e é executável" \
  || { echo "FALTA scripts/gen-bindings.sh executável"; exit 1; }

# 2) Limpar e regenerar via o PRÓPRIO script (deve sair 0)
"$ROOT/scripts/gen-bindings.sh" \
  && echo "OK  gen-bindings.sh rodou limpo (exit 0)" \
  || { echo "FALHA gen-bindings.sh retornou erro"; exit 1; }

# 3) bindings/ populado com .ts NÃO-VAZIOS — checado no FILESYSTEM
#    (os .ts são ignorados pelo git por design; NÃO usar git ls-files aqui)
TS_COUNT="$(find "$ROOT/bindings" -name '*.ts' -size +0c | wc -l | tr -d ' ')"
test "$TS_COUNT" -ge 1 \
  && echo "OK  bindings/ tem $TS_COUNT arquivo(s) .ts não-vazio(s)" \
  || { echo "FALHA bindings/ sem .ts não-vazio (geração não populou)"; exit 1; }

# 4) Os bindings referenciam a API do core (snake_case OU camelCase)
find "$ROOT/bindings" -name '*.ts' -size +0c -exec grep -liE 'parse_?reference|reference|ping' {} + >/dev/null \
  && echo "OK  bindings referenciam a API do core" \
  || { echo "FALHA bindings não referenciam a API do core (parse_reference/Reference/ping)"; exit 1; }

# 5) ADR-0004 registrado em DECISIONS.md
grep -q 'ADR-0004' "$ROOT/DECISIONS.md" \
  && echo "OK  ADR-0004 registrado em DECISIONS.md" \
  || { echo "FALTA ADR-0004 em DECISIONS.md"; exit 1; }

# 6) Working tree limpo: .ts gerados e node_modules são IGNORADOS por design;
#    só os arquivos VERSIONADOS (script, config, manifestos, DECISIONS) são committados.
cd "$ROOT"
test -z "$(git status --porcelain)" \
  && echo "OK  working tree limpo (gerados ignorados; versionados committados)" \
  || { echo "FALHA working tree sujo:"; git status --porcelain; exit 1; }

echo "F0.4 verificada."
```

## Restrições

- **NÃO modificar / forkar / copiar / vendorizar o `the-light`.** A geração lê a fronteira do
  crate **`the-light-app-core`** (`core/`), não o `the-light`. O core externo segue consumido
  só via git dep pinada (ADR-0002).
- **NÃO alterar a fronteira UniFFI do core nem a versão pinada `uniffi = "0.31.2"`** (ADR-0003)
  para "fazer o `ubrn` funcionar". Construir a `cdylib`/`staticlib` do host para extrair metadata
  **não** é modificação; mas se a geração **exigir** mudar a versão do `uniffi`, mudar a forma da
  fronteira (`core/src/lib.rs`), ou qualquer escolha estratégica, isso é **DECISÃO** → reporte
  **`blocked`** com o **erro EXATO**, **NÃO** improvise.
- **Rede só em dev/build.** Instalar o `ubrn` via npm/cargo com rede é **permitido**;
  offline-first é regra de **runtime do produto**. Nenhum binding gerado pode exigir rede em
  runtime, e nenhuma chamada de rede pode ser introduzida no código.
- **NÃO escrever bindings TS à mão nem fingir geração.** Se o `ubrn` **não instalar, não rodar
  ou não gerar** (imaturidade da ferramenta — VISION §9 risco #1; incompatibilidade com
  `uniffi 0.31.2`; rename para `uniffi-bindgen-javascript`; etc.), reporte **`blocked`** com o
  **erro EXATO** do `ubrn`/`cargo`/`npm` — sem bindings escritos à mão, sem cópia, sem fingimento.
- **Fixar a versão exata do `ubrn`** (sem `^`/`~`/`*`) e registrá-la no ADR-0004; lockfile
  versionado quando via npm.
- **`tsc --noEmit` é OPCIONAL** (só se `tsconfig`+`typescript` forem incluídos) e **não pode ser
  ponto de falha frágil** — a porta dura é o script rodar limpo e popular `bindings/`.
- **Working tree limpo respeita o `.gitignore` da F0.1:** os `.ts` gerados são **ignorados por
  design**; não versione-os e não remova a guarda `/bindings/* + !/bindings/.gitkeep`.
- **Segredos nunca em git/log** (nada de tokens/chaves/`.env`).
