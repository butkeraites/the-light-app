## Objetivo

Provar a ponte Rust→Expo no alvo **Android nativo**: digitar `"Jo 3.16"` (PT) ou
`"John 3:16"` (EN) no app rodando no **emulador Android** resolve a referência
**pelo Rust nativo** (`the-light-core::reference` via UniFFI → **Turbo Module
JSI** gerado pelo `ubrn`), **não** pelo caminho web/wasm, **não** por wasm,
**não** por parsing em TS. Ambos os idiomas devem resolver para
`book=43, chapter=3, verses=Single{verse:16}`. Exige **run real no emulador**
(não apenas build), com **prova headless/determinística** (sem clique manual na
UI), capturada via `adb logcat`.

Esta tarefa é o **ESPELHO da F0.7 (iOS, já aceita — `a6f6797`, ADR-0008)** no
alvo Android. O grosso da integração — o **Turbo Module** (spec
`NativeTheLightAppCore`, C++/JSI em `cpp/`, TS em `src/`, bindings em `bindings/`,
codegen New Arch, autolinking via `app/react-native.config.js`, glue
`app/web/reference.ts` → JSI → Rust, self-test `app/web/selftest.ts`) — **JÁ
existe e é compartilhado iOS/Android**. A F0.8 **NÃO refaz isso**: só adiciona o
**caminho de BUILD Android** (`ubrn build android` via `cargo-ndk` → `.so` por
ABI em `jniLibs` + o módulo Gradle da library) e a prova no emulador.

**O caminho nativo Android do `ubrn` (a usar — já descoberto pelo Planner,
confirmar empiricamente):**
- Subcomando: **`ubrn build android`** (≠ `build ios` da F0.7; ≠ `build web` da
  F0.6b; ≠ `generate jsi bindings` host-only da F0.4). Opções relevantes
  (`node_modules/.bin/ubrn build android --help`, já inspecionado):
  `--config <ubrn.config.yaml>`, `-t/--targets <abi,...>`
  (ex.: `aarch64-linux-android` = `arm64-v8a`), `-g/--and-generate` (gera os
  bindings TS + a **glue do Turbo Module** junto), `-r/--release`, `--no-cargo`,
  `--no-jniLibs`, `--native-bindings`.
- Fluxo (lido em
  `node_modules/uniffi-bindgen-react-native/crates/ubrn_cli/src/jsi/android/commands.rs`):
  1) p/ cada target, `cargo ndk --manifest-path core/Cargo.toml --target <abi>
     --platform <apiLevel> -- build` (define `…_RUSTFLAGS=-C
     link-arg=-Wl,-z,max-page-size=16384` p/ alinhamento de página 16KB);
  2) **copia** o artefato Rust p/ `jniLibs/<abi>/` (default
     `android/src/main/jniLibs`, sob `android.directory`);
  3) com `-g`, gera os bindings TS/C++ + a glue do Turbo Module (inclui o módulo
     **Android/Gradle**: `CMakeLists.txt`, `android/src/main/java` Kotlin/JNI, e
     reusa o `cpp/` compartilhado com iOS).
- Schema de config Android (`AndroidConfig`, camelCase) em
  `node_modules/uniffi-bindgen-react-native/crates/ubrn_cli/src/jsi/android/config.rs`:
  bloco top-level **`android:`** com `directory` (default `android`), `jniLibs`
  (default `src/main/jniLibs`), `targets` (default os 4 ABIs), `apiLevel`/alias
  `platform` (default **23** — RN 0.85 pede ≥24; ajustar se o build exigir),
  `packageName` (default derivado do `package.json` da raiz — **defina explícito**
  para evitar `-` inválido em pacote Java, ex.: `com.thelight.core`),
  `codegenOutputDir` (default `android/generated`), `useSharedLibrary`
  (default `false` → staticlib; manter o default, igual ao iOS/Turbo Module).
- Guia oficial RN/Android:
  `node_modules/uniffi-bindgen-react-native/docs/src/guides/rn/getting-started.md`
  (padrão `"ubrn:android": "ubrn build android --and-generate"`). ATENÇÃO: o guia
  assume layout `create-react-native-library` (a library já tem
  `android/build.gradle` de template). **Nosso app é Expo** e a **raiz** é a
  "RN turbo-module library" (ADR-0008): se o `ubrn build android` **não** gerar
  um `android/build.gradle` buildável (só `CMakeLists.txt`/Kotlin/jniLibs), pode
  ser necessário **versionar um `android/build.gradle` hand-written** (análogo ao
  `app/react-native.config.js` da F0.7) — é glue **do app**, NÃO toca
  `the-light`/fronteira. Decidir empiricamente e registrar no **ADR-0009**.

**Feature-matrix do core no nativo (F0.6a / ADR-0005, ADR-0007):** a fronteira
consome `the-light-core` com `default-features = false` + um bloco
`[target.'cfg(not(target_arch="wasm32"))']` que liga `embedded`. Logo, os alvos
Android (não-wasm32) compilam o core **completo** (`embedded` → rusqlite
SQLite-C + reqwest) — **já provado** pelo smoke test. Aqui só `parse_reference`
(puro `reference`/`model`) é exercitado, mas o build arrasta `embedded` — risco
baixo, observar.

**Integração no Expo (SDK 56, New Arch já `newArchEnabled: true` em
`app/app.json`):**
- O app em `app/` **não tem** projeto `android/` nativo versionado. Gerá-lo exige
  **`npx expo prebuild -p android`** (de `app/`), que produz `app/android/` (Gradle
  + autolinking). Depois `gradlew assembleDebug`/`installDebug` (ou
  `expo run:android`) build+install+launch no emulador.
- O autolinking RN no Android lê o **mesmo** `app/react-native.config.js`
  (override `dependencies['the-light-app'].root = '..'` → raiz) que já existe da
  F0.7; ele deve descobrir o módulo Gradle da library na raiz. O **codegen New
  Arch** lê o `codegenConfig` da raiz (`package.json`,
  `name: RNTheLightAppCoreSpec`, `jsSrcsDir: src`) — **compartilhado iOS/Android**.
  Confirmar que o autolink Android encontra o módulo; esta é a parte de **maior
  risco residual** (ver Restrições / blocked).

**A tela e o glue de plataforma (Metro resolve por extensão) — JÁ PRONTOS da F0.7:**
- `app/app/index.tsx` importa `parseReference` de `../web/reference`. No **web**, o
  Metro resolve `app/web/reference.web.ts` (wasm, F0.6b — **manter intacto**). No
  **nativo** (iOS **e** Android), resolve `app/web/reference.ts`, que **delega** ao
  barrel gerado (`./native-generated/src/index` → `installRustCrate()` → JSI →
  Rust). **Esse glue nativo deve funcionar igual no Android** — confirmar, sem
  reescrever (sem parser TS, sem eco).
- O self-test `app/web/selftest.ts` (disparado por `useEffect` na `HomeScreen`
  **só** sob `EXPO_PUBLIC_TLA_SELFTEST=1`) já compõe o marcador a partir do
  **retorno** real do `parseReference`. **Reutilizar como está**: no Android,
  `console.log`/`console.error` caem no `logcat` (tag `ReactNativeJS`).

**Gerado (IGNORAR no git) vs. versionado:**

  **`android/`** da library na raiz (Gradle module + `CMakeLists.txt` + Kotlin/JNI

- VERSIONAR: o bloco **`android:`** em `ubrn.config.yaml`; os scripts novos

## Restrições

- **NÃO modificar nem forkar `the-light`** (ADR-0002) nem alterar a **forma da
  fronteira** `core/` (modo library / `setup_scaffolding!` / `CoreError` /
  `parse_reference`). Se o caminho Android exigir mudar o core ou a fronteira →
  **HALT/blocked**, não improviso.
- **NÃO fingir, stubar nem reimplementar o parsing**: `parse_reference` resolve no
  **Rust nativo** via Turbo Module. Eco/lógica TS/JSON hardcoded = reprovação. A
  prova (logcat) deve evidenciar a origem Rust (Turbo Module/JNI/JSI), não um
  atalho.
- **Anti-alucinação / uma fonte da verdade**: a referência vem do
  `the-light-core` via UniFFI; nada de parser paralelo.
- **NÃO refazer o Turbo Module nem regredir iOS/web**: o spec
  `NativeTheLightAppCore`, `cpp/`/`src/`/`bindings/`, `app/react-native.config.js`,
  `app/web/reference.ts`/`selftest.ts`, `app/app/index.tsx` e os blocos
  `ios:`/`turboModule:`/`web:` do `ubrn.config.yaml` são **compartilhados/da F0.7**
  — só ESTENDER (bloco `android:` + scripts + gitignore), não reescrever.
- **Offline-first é regra de runtime**: rede só em **dev/build** (cargo-ndk, Gradle,
  npm, Metro). Nenhuma rede no runtime do produto; **nenhum segredo** em git/log
  (keystore/`.env`/`local.properties` jamais commitados).
- **Gerados IGNORADOS**: `app/android/` (prebuild), `/android/` (jniLibs, `.so`,
  CMake, Kotlin, `build/`, `.gradle/`, `android/generated`),
  `app/web/native-generated/`, `bindings/`, `node_modules`, `target/`. Working tree
  limpo é critério rígido. (Se um `android/build.gradle` hand-written for
  necessário, versioná-lo com exceção dirigida no `.gitignore`.)
- **Qualidade por tarefa**: se algo Rust mudar (não deveria), `cargo fmt --check`
  + `cargo clippy -D warnings` + `cargo test` verdes; TS: `tsc --noEmit` no `app/`.
- **BLOQUEIO LEGÍTIMO (`blocked`) — decisão estratégica:** se o **caminho Android
  do `ubrn 0.31.0-3`** for imaturo/inoperante a ponto de **só** funcionar com um
  **hack proibido** (forkar/alterar `the-light` ou a fronteira; reimplementar/
  stubar o parsing; integrar o Turbo Module ao Expo de forma que quebre o
  autolinking/New Arch sem caminho sancionado; build Android do core `embedded`
  falhar de modo irreparável sem tocar o core), **reportar `blocked` com o erro
  EXATO** (saída literal do comando que falhou) e a causa-raiz. Isso é o **gate
  estratégico** sobre o caminho Android do `ubrn` (análogo ao ADR-0007 do web e ao
  ADR-0008 do iOS) — o Driver escreve `loop/HALT` para decisão humana. **Não** é
  bloqueio do core (já resolvido na F0.6a e provado pelo smoke test) nem do
  toolchain (NDK/SDK/JDK/emulador já prontos). Se a decisão for tomada e o caminho
  viabilizado, registrar em **ADR-0009**.
