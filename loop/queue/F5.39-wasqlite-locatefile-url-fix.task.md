## Bug (causa raiz REAL, com stack trace)

Mateus 1 (e QUALQUER capítulo) não abria no WEB. Diagnosticado com **Chrome headless real**
(puppeteer-core sobre o Chrome do sistema, dirigido pelo Driver): a lista de livros funciona
(66 livros), mas abrir um capítulo lança **"Failed to construct 'URL': Invalid base URL"**.
Stack: `openReadingDbWeb` → `SQLiteESMFactory({ wasmBinary })` → o glue Emscripten do
`app/web/vendor/wa-sqlite-fts5/wa-sqlite.mjs` avalia EAGERMENTE
`new URL("wa-sqlite.wasm", import.meta.url)` e, sob o bundler **DEV do Metro**, `import.meta.url`
é a string **"null"** → `new URL("wa-sqlite.wasm", "null")` lança. **Não tem relação com o
tamanho do DB nem com OPFS** (F5.36/F5.38) — a **leitura web SEMPRE esteve quebrada**; só passava
nos testes headless (Node, onde `import.meta.url` é um `file://` válido). O usuário foi o primeiro
a tentar LER no browser.

## Fix

Passar `locateFile` ao factory nos DOIS pontos de produção (`sqlite-reading-opfs.web.ts:104`,
`sqlite-lexicon-opfs.web.ts:99`): `SQLiteESMFactory({ wasmBinary, locateFile: (path) => path })`.
O glue: `Ca ||= f.locateFile ? (...) : (new URL("wa-sqlite.wasm", import.meta.url)).href` — com
`locateFile` presente, NÃO toma o branch do `new URL` inválido. Os bytes do wasm já vêm por
`wasmBinary` (o valor de `locateFile` não é buscado). Os arquivos de TESTE têm seus próprios
`SQLiteESMFactory({ wasmBinary })` — INALTERADOS (em Node funcionam).
