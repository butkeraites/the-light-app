## Bug (achado pelo smoke da F6.2)

Importar um backup numa **OPFS vazia** (instalação limpa, sem userdata ainda escrito) FALHA com
"A requested file or directory could not be found". Causa: `app/web/userdata-opfs.web.ts` (~L37-38)
chama `getDirectoryHandle(OPFS_ROOT_DIR/OPFS_USERDATA_DIR, {create:false})` SEM guarda → numa OPFS
vazia lança `NotFoundError` em vez do caminho gracioso "ausente→null/empty" que `readFile`/`deleteFile`
já têm via `resolveParent(...,false)`. O round-trip export/import foi provado (F6.2) SÓ no contexto
principal (que já tinha userdata) — o caminho de OPFS-vazia quebra "importar backup num aparelho novo".

## Escopo

1. **`app/web/userdata-opfs.web.ts`:** guardar a leitura antes-de-qualquer-escrita — quando o dir
   raiz/userdata NÃO existe (`{create:false}` lança `NotFoundError`), retornar `null`/vazio
   (dir/arquivo ausente) em vez de propagar o erro. Alinhar com o comportamento gracioso de
   `readFile`/`deleteFile` (via `resolveParent(...,false)`). A ESCRITA (`writeFile`/import) segue
   criando os dirs com `{create:true}` normalmente. NÃO alterar o comportamento quando o dir existe.
2. **Preservar dado do usuário:** só o caminho de LEITURA numa OPFS vazia muda (ausente→vazio); nada
   de apagar/criar destrutivo.
3. **Asserção no smoke (`app/web/__browser__/`):** adicionar/estender o fluxo de import p/ exercitar
   **import em OPFS VAZIA** — usar um `BrowserContext` FRESCO (sem userdata prévio) e importar um
   snapshot; assert que aplica sem erro e as notas aparecem. Marcador `TLA_WEB_import-fresh ok` (ou
   estender `export-import`). Deve reproduzir o bug ANTES do fix (se reverter o guard) e passar depois.
