## Objetivo

Expor na **fronteira UniFFI** (`core/src/lib.rs`) uma função de **busca full-text
(FTS5)** que **delega** ao `the-light-core` (rev pinado `8f66004`): dado um
`db_path`, uma `query`, uma `translation` e filtros opcionais, retorna as
ocorrências como `Vec<SearchHit>` — cada hit com a **referência** (book/chapter/
verse) e o **texto verbatim do store** (mais o trecho destacado e o score BM25).
A busca é **acento-insensível em PT** (o índice `verses_fts` usa
`remove_diacritics 2`, então `ceus` casa `céus`) — isso vem **de graça do core**,
não é reimplementado aqui.

Nada de SQL/FTS é reimplementado na fronteira: ela apenas **adapta tipos e erros**
do core para um Record UniFFI (`SearchHit`), seguindo a regra "uma fonte da
verdade". Mesmo molde da **F0.9/F1.2/ADR-0010**: a **exportação** existe em
**todos** os alvos (forma da fronteira uniforme p/ o `ubrn` extrair metadata no
host), mas **só o corpo que toca store/`rusqlite`** é
`#[cfg(not(target_arch = "wasm32"))]` (no web, stub que retorna `CoreError`). Assim
`rusqlite`/`the-light-core` com `embedded` **não entram no grafo wasm** e o build
web (F0.6b/F0.10) continua puro. Anti-alucinação: **todo texto vem do store local**.

Esta é uma tarefa de **NÚCLEO** (fronteira Rust + teste de host), análoga à F1.2.
A **UI de busca** é a F1.6 (não faz parte desta tarefa).

## Contexto / arquivos

**Fronteira (este repo) — onde implementar:**
- `core/src/lib.rs` — adicionar a função `search` e o novo Record `SearchHit`.
  Reusar o padrão de `get_passage`/`get_chapter` (F0.9/F1.2): export em todos os
  alvos + corpo gated + stub web. Reusar `CoreError`, `Reference` (já tem
  `From<model::Reference>`), e o módulo de teste de host (`read_tests`).
- `core/Cargo.toml` — **não precisa mudar**: a matriz por alvo já liga `embedded`
  só no não-wasm; `the_light_core::search`/`source`/`store` existem só no nativo.

**API REAL do core a delegar (já descoberta na fonte — confira antes de codar em
`/Users/butkeraites/Documents/the-light/crates/the-light-core/src/`):**
- `source/mod.rs` (trait `BibleSource`, traga em escopo p/ chamar o método) e
  `source/embedded.rs::150` →
  `fn search(&self, query: &str, opts: &SearchOptions) -> Result<Vec<SearchHit>>`.
  **Ponto de delegação recomendado** (mais alto nível, igual à F1.2 que usa
  `EmbeddedSource::passage`/`translations`/`chapter_count`): `EmbeddedSource::search`
  **já valida** `has_translation` e retorna `SourceError::UnknownTranslation` se a
  tradução não existe no banco — ótimo caminho de erro → mapeia p/ `CoreError`.
  Internamente ele chama `the_light_core::search::search(conn, query, opts)`.
- `search.rs` (módulo **`#[cfg(feature = "embedded")]`** — só no nativo):
  - `pub struct SearchOptions { translation: TranslationId, book: Option<u8>, limit: usize }`;
    `SearchOptions::new(translation)` → `book: None`, `limit: DEFAULT_LIMIT` (= **20**).
  - `pub fn search(conn, query, opts) -> rusqlite::Result<Vec<SearchHit>>`.
  - `pub const HL_START: &str = "\u{2}"` / `HL_END: &str = "\u{3}"` — marcadores que
    envolvem o termo casado em `SearchHit.highlighted`.
  - `build_match_query` cita os termos (anti-injeção FTS5); **query sem termo
    utilizável → `Ok(Vec::new())`** (vazio, **sem panic**) — não é erro.
- `model.rs::298` (**PURO**, presente em **todos** os alvos):
  `pub struct SearchHit { reference: Reference, translation: TranslationId, text:
  String, highlighted: String, score: f64 }`.
- `migrations/v1_initial.sql::41` — `CREATE VIRTUAL TABLE verses_fts USING fts5(text,
  translation_id UNINDEXED, verse_id UNINDEXED, tokenize = 'unicode61
  remove_diacritics 2')`. **ATENÇÃO CRÍTICA: NÃO há trigger** que popule `verses_fts`
  a partir de `verses` — o próprio core, nos seus testes (`search.rs::seeded`,
  `store.rs`), **insere explicitamente** em `verses_fts(text, translation_id,
  verse_id)` após inserir em `verses`. **Logo o fixture da F1.2
  (`build_kjv_fixture`, que só insere em `verses`) NÃO popula o FTS** — o fixture
  desta tarefa **deve** povoar `verses_fts` explicitamente, com `verse_id` igual ao
  `verses.id` (a busca faz `JOIN verses v ON v.id = verses_fts.verse_id`).

**Gating de módulos do core (confirmado em `the-light-core/src/lib.rs`):**
`model` e `reference` são **puros** (sem `cfg`); `search`/`source`/`store` são
**`#[cfg(feature = "embedded")]`** (só no nativo). Portanto:
- O Record `SearchHit` e seu `From<the_light_core::model::SearchHit>` referenciam
  **só `model`** → podem ser definidos em **todos** os alvos (como `Verse`/`Passage`).
- A construção de `SearchOptions` e a chamada a `EmbeddedSource::search`/`Store`
  referenciam módulos `embedded`-gated → **só** podem aparecer dentro do bloco
  `#[cfg(not(target_arch = "wasm32"))]`. O **stub web** não pode tocar nenhum
  desses tipos (consome os args com `let _ = (...)`).

**Função a expor (assinatura recomendada — mapeia fielmente `SearchOptions`):**
- `search(db_path: String, query: String, translation: String, book: Option<u8>,
  limit: Option<u32>) -> Result<Vec<SearchHit>, CoreError>`
  - corpo gated: `Store::open(db_path)` → `EmbeddedSource::new(&store)` →
    montar `SearchOptions::new(TranslationId::new(translation))`, setar `opts.book =
    book`, setar `opts.limit = limit.map(|l| l as usize)` quando `Some` (senão manter
    o default 20) → `source.search(&query, &opts)` → mapear cada `model::SearchHit`
    p/ o Record `SearchHit` e o erro p/ `CoreError`.
  - stub web: `let _ = (db_path, query, translation, book, limit);` →
    `Err(CoreError::Generic { message: "store local indisponível no alvo web
    (F0.10: wa-sqlite+OPFS)".to_string() })` (mesma mensagem-padrão das outras).

**Novo Record UniFFI (definir em TODOS os alvos, como `Verse`/`Passage`):**
- `SearchHit { reference: Reference, translation: String, text: String, highlighted:
  String, score: f64 }` com `From<the_light_core::model::SearchHit>` (mapeie
  `reference` via o `From` já existente; `translation` via `.to_string()`). `f64` é
  suportado pelo UniFFI; `highlighted` carrega os marcadores `HL_START/HL_END` do
  core (a UI da F1.6 decide como renderizar — fora de escopo aqui).

**Banco de teste (determinismo vs. corpus completo) — mesmo princípio da F1.2:**
- O teste de host **PRIMÁRIO** deve ser **offline e determinístico**, construindo um
  **fixture pequeno** num diretório temporário (schema = migrações do core via
  `Store::open`; DML de **domínio público**), **incluindo a população do
  `verses_fts`** (sem isso a busca retorna vazio). Estenda o padrão de
  `build_kjv_fixture` (F1.2) — mas use **`id` explícito** em `verses` e o mesmo `id`
  em `verses_fts.verse_id`, como o core faz em `search.rs::seeded`.
- **Conteúdo recomendado do fixture** (mínimo p/ as asserções):
  - `translations`: `kjv` (`en`, `public-domain`) **e** `alm` (`pt`,
    `public-domain`).
  - `verses` + `verses_fts` (id, trad, book, ch, v, texto):
    - id 1 — `kjv` 43/3/16 → `JOHN_3_16_KJV` (contém "God"/"world").
    - id 2 — `kjv` 43/3/17 → `JOHN_3_17_KJV` (contém "world" — p/ testar `limit`).
    - id 3 — `alm` 1/1/1 → `"No princípio criou Deus os céus e a terra"` (Gênesis
      1:1, Almeida, **acentuado** — p/ provar acento-insensível: `ceus` casa `céus`).
      Texto de domínio público; é o mesmo verso usado nos testes do próprio core.
- **OPCIONAL (bônus, guardado por `if Path::new(BIBLE_DB).exists()`):** se
  `assets/data/bible.sqlite` existir (F1.1, gerar-ignorado, ~47 MB, **com FTS5
  povoado pelo importador**), rodar asserções extras (ex.: `search(bible, "God",
  "kjv", …)` retorna muitos hits incluindo João 3:16). **Nunca** requisito — as
  asserções primárias passam **só** com o fixture, offline.

**ADRs/decisões pertinentes:** ADR-0005 (matriz de features por alvo; `embedded` só
no nativo), ADR-0010 (store nativo + **gating de corpo** + anti-alucinação),

## Restrições

- **NÃO modificar, forkar, copiar ou vendorizar `the-light`.** Consumo só pela git
  dep pinada `8f66004`. A busca é **delegada** a `the_light_core::search` (via
  `EmbeddedSource::search`). Se a API real do core **não** permitir a busca sem
  mudar o core → **não invente**: marque a tarefa `blocked` com o **erro/limitação
  EXATA** e a decisão necessária (PR+ADR ao core).
- **NÃO reimplementar SQL/FTS/lógica de domínio** na fronteira. Proibido escrever
  `verses_fts MATCH`, `bm25(...)`, `highlight(...)`, montar a expressão FTS, etc. —
  isso vive no core. (INSERTs em `verses`/`verses_fts` no **setup do teste** são
  fixture, **não** produto — como o próprio core faz em `search.rs::seeded`.)
- **Anti-alucinação:** todo texto de hit vem **do store local** (verbatim); nenhum
  texto é gerado. Só **domínio público** (KJV/Almeida) no fixture.
- **Gating obrigatório:** o corpo que toca `search`/`source`/`store`/`rusqlite`
  **deve** ficar sob `#[cfg(not(target_arch="wasm32"))]` com stub web; o Record
  `SearchHit`/`From` só pode referenciar `model` (puro). **Não regredir** o build
  web (sem `rusqlite` no grafo wasm). A exportação UniFFI permanece em todos os alvos.
- **Offline-first:** nenhuma rede em runtime nem no teste; o teste **primário** usa
  fixture local determinístico (não depende de `bible.sqlite` presente nem de rede).
  Lembrete: **o fixture precisa popular `verses_fts`** (não há trigger no schema do
  core) — sem isso a busca retorna vazio e o teste falharia.
- **Sem segredos.** Não tocar chaves/credenciais.
- Em conflito com qualquer regra acima → **`blocked`/HALT** com o motivo exato, sem
  improviso.
