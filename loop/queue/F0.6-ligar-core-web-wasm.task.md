## Objetivo

Provar a ponte Rust → app **no alvo web (WASM)**, ponta a ponta no navegador:
digitar `Jo 3.16` (ou `John 3:16`) na tela do app web deve mostrar a referência
**resolvida pelo Rust** (livro 43, capítulo 3, versículo 16) — vinda da fronteira
`the-light-app-core` compilada para `wasm32-unknown-unknown` e chamada via
bindings web, **não** de lógica reimplementada em TypeScript.

Esta é uma das três tarefas de ligação da ponte na Fase 0 (web=F0.6, iOS=F0.7,
Android=F0.8). É também o **ponto onde a fricção #1 da VISION (§4) — "SQLite no
WASM" — é exercida pela primeira vez**: a fronteira arrasta `the-light-core`, que
hoje depende de `rusqlite` (SQLite-C *bundled*). A pergunta central da tarefa é:
**conseguir compilar a fronteira para wasm SEM puxar `rusqlite`**, configurando
apenas o *consumo* do core (sem tocar o core).

## Contexto / arquivos

Estado provado até aqui (ver `loop/archive/`):
- **F0.3** (`9881c72`): a fronteira `core/` (`the-light-app-core`) expõe
  `parse_reference(input) -> Result<Reference, CoreError>` delegando ao
  `the-light-core::reference` (git dep pinada no rev
  `0888ac0bfed15222874c5462902a61ac3a39147c`, package `the-light-core` v1.2.0).
  O módulo `reference` é **puro** (parsing sobre `String`; sem store/SQLite/rede).
- **F0.4** (`e19064a`, ADR-0004): `scripts/gen-bindings.sh` + `ubrn` `0.31.0-3`
  (devDependency npm na **raiz**) geram os bindings **JSI/nativo** (`ubrn generate
  jsi …`) em `bindings/` (ignorados pelo git). Esse é o caminho **nativo** — o
  alvo **web/wasm é um caminho distinto** do ubrn, ainda não exercido.
  `ubrn.config.yaml` é a fonte da verdade dos caminhos.
- **F0.5** (`3262f56`): app Expo SDK 56 (expo-router) em `app/` (projeto npm
  independente). Tela `app/app/index.tsx` com `TextInput` + `<Text testID="result">`
  fazendo **eco local**, **sem** chamar o core (guarda de escopo da F0.5).
  `npx expo export --platform web` (de `app/`) sai 0 e gera `app/dist/`.
- **F0.0** (`efe334f`, DECISIONS tabela de ambiente): target Rust
  `wasm32-unknown-unknown` **NÃO instalado** (pendência leve, resolvível via
  `rustup target add`). `the-light` (Xcode/Android) só importam p/ F0.7/F0.8.

Arquivos relevantes desta tarefa:
- `core/Cargo.toml` (declaração da git dep do core — onde se configura o consumo).
- `app/web/` (glue do alvo web — diretório-alvo da seção 1 do plano; criar aqui o
  carregamento do módulo wasm + wrapper TS sobre `parseReference`).
- `app/app/index.tsx` (a tela; **aqui a guarda de escopo da F0.5 deixa de valer** —
  esta tarefa LIGA a UI ao core no web).
- `bindings/` (saída gerada, ignorada pelo git) e `ubrn.config.yaml` (config).
- Fonte clonada do core (consumo de leitura permitido, NÃO é modificar o core):
  `~/.cargo/git/checkouts/the-light-9eb8809a6d68281a/0888ac0/crates/the-light-core/Cargo.toml`
  — onde se DESCOBRE se o core é configurável p/ excluir `rusqlite`.

**Fricção #1 (VISION §4 / §9; Apêndice A gate #1):** no `wasm32-unknown-unknown`,
o SQLite em C (`rusqlite` *bundled*) **não compila trivialmente**; o mesmo vale
para `reqwest` com `blocking` + TLS nativo. Como o cargo compila o crate inteiro
`the-light-core` (e todas as suas deps) mesmo que a fronteira só use o módulo
`reference`, a ligação web só fecha se o consumo do core puder ser configurado
para **não** arrastar essas deps.

## Passos sugeridos

> A tarefa é grande; siga em sub-passos. O **sub-passo 2 é o portão decisivo** —
> se ele bloquear de forma legítima, pare ali e reporte `blocked` (ver Restrições),
> sem inventar/stubar e sem tocar o core.

1. **Descobrir a configurabilidade do core (consumo, não modificação).** Leia o
   `Cargo.toml` do `the-light-core` na fonte clonada (caminho acima) e verifique:
   há `[features]`? `rusqlite`/`reqwest` são `optional`? há `[target.'cfg(...)']`
   separando deps por alvo? Registre o que encontrar. Se houver uma feature do
   tipo "reference-only"/sem-store (ou as deps forem `optional` com
   `default-features`), configure a git dep em `core/Cargo.toml` com
   `default-features = false` e/ou as features mínimas, de modo a **não** puxar
   `rusqlite`. Isso é CONFIGURAR o consumo — permitido.
2. **Compilar a fronteira para wasm (portão decisivo).**
   `rustup target add wasm32-unknown-unknown` (idempotente, permitido) e
   `cargo build -p the-light-app-core --target wasm32-unknown-unknown` (de `core/`).
   - Se compilar **sem** `rusqlite`/SQLite-C: siga.
   - Se falhar por causa de `rusqlite`/`libsqlite3-sys`/`cc`/SQLite-C (ou
     `reqwest`/TLS) e o core **não** for feature-gateável sem ser modificado:
     **PARE e reporte `blocked`** com o erro EXATO (ver Restrições). Não modifique
     o core. Não stube.
3. **Gerar os bindings WEB/WASM via o caminho web do ubrn** (NÃO o `jsi` da F0.4).
   Descubra o subcomando/flavor web do `ubrn 0.31.0-3` (ex.: geração para wasm /
   `--flavor`/comando `web`), reutilizando `ubrn.config.yaml` como fonte da
   verdade. Se o caminho web do ubrn for inexistente/imaturo/inoperante a ponto de
   não produzir um módulo wasm+bindings utilizável, isso também é `blocked`
   legítimo (registrar o erro/saída exata).
4. **Glue web em `app/web/`.** Crie o carregamento do módulo wasm (init/instanciar)
   + um wrapper TS fino (ex.: `app/web/core.ts`) que exponha
   `parseReference(input: string): Reference` sobre os bindings gerados.
5. **Ligar a tela.** Em `app/app/index.tsx`, ao submeter/alterar o input, chame o
   wrapper e renderize a referência resolvida (livro/cap./versículo) em
   `<Text testID="result">`. No web, importe o glue de `app/web/`. (Guarda de
   escopo da F0.5 deixa de valer AQUI.)
6. **Provar de forma headless.** Escreva um teste automatizável (preferência:
   teste **node** que importa o módulo wasm + bindings gerados e chama
   `parseReference("Jo 3.16")`, esperando `book=43, chapter=3, verses=Single 16`,
   e idem para `"John 3:16"` resolvendo igual). Evite depender de abrir o
   navegador. Se um teste 100% headless do módulo wasm não for viável dado o
   empacotamento dos bindings, defina a prova mais objetiva e determinística
   possível (ex.: script que executa o bundle web e verifica a saída) e documente
   por que o caminho mais headless não coube.
7. **Não regredir o export web.** Garanta que `npx expo export --platform web`
   (de `app/`) continue saindo **0** com o core ligado (com guarda para o caso de
   o wasm carregar sob demanda).
8. **Higiene de árvore.** Ignore os artefatos gerados (wasm/bindings) no
   `.gitignore`; versione apenas glue (`app/web/*`, `app/app/index.tsx`),
   config/scripts (`ubrn.config.yaml`, `scripts/*`) e, se uma decisão de
   store/transporte no web for tomada, **ADR-0005** em `DECISIONS.md`.

## Critério de aceite

Critérios do **caminho aprovado (`passed`)** — todos verificáveis de forma
**não-interativa** (sem abrir navegador):

- [ ] A fronteira `the-light-app-core` **compila para `wasm32-unknown-unknown`
  sem `rusqlite`/SQLite-C** (`cargo build --target wasm32-unknown-unknown` sai 0).
  O `core/Cargo.toml` configura o consumo do core para não arrastar `rusqlite`
  (via features do próprio core), **sem** modificar/forkar/copiar o core.
- [ ] Bindings **web/wasm** gerados pelo caminho web do `ubrn` (não o `jsi`),
  produzindo um módulo wasm + `.ts` utilizáveis (artefatos sob diretório ignorado
  pelo git).
- [ ] A tela web (`app/app/index.tsx` + glue `app/web/`) **referencia o core**:
  ao digitar `Jo 3.16` / `John 3:16`, o resultado exibido é a referência resolvida
  **pelo Rust** (livro 43, cap. 3, versículo 16), não eco local nem lógica TS.
- [ ] **Prova headless** de que `parseReference("Jo 3.16")` e
  `parseReference("John 3:16")` retornam `book=43, chapter=3, verses=Single 16`
  via o módulo wasm + bindings (teste node determinístico, ou a prova mais
  objetiva possível, documentada).
- [ ] `npx expo export --platform web` (de `app/`) **continua saindo 0** com o
  core ligado.
- [ ] **Working tree limpo**: wasm/bindings gerados são ignorados; versiona-se
  apenas glue/config/scripts (e ADR-0005 se houver decisão). `git status
  --porcelain` vazio após o commit. O `the-light`/`core` (lógica) e a fronteira
  da F0.3 não são tocados além de configurar o consumo no `core/Cargo.toml`.

> **Saída alternativa LEGÍTIMA (`blocked`, não conta como falha):** se o sub-passo
> 2 provar que o core **não** é feature-gateável para excluir `rusqlite` sem
> modificá-lo (logo o wasm não compila), OU se o caminho web do ubrn for
> inoperante, o resultado correto é `blocked` com o erro EXATO de compilação/saída
> — este é o **gate estratégico de Fase 0 (Apêndice A #1 / VISION §4 fricção #1)**.
> O Driver escreverá HALT para decisão humana/Guia (ex.: `wa-sqlite`+OPFS com store
> injetado; web como "leitura + IA" com dados pré-indexados; ou um PR ao core
> adicionando feature de exclusão de store). Ver Restrições.

## Restrições

- **NÃO modificar/forkar/copiar/vendorizar o `the-light`/`the-light-core`.**
  Configurar o *consumo* (features/`default-features=false` no `core/Cargo.toml`)
  é permitido. **Adicionar/alterar features no próprio core, gating de `rusqlite`,
  split de crate, etc. é PR no `the-light` + ADR = decisão humana ⇒ HALT** — não
  faça aqui.
- **NÃO fingir/stubar o wasm nem os bindings.** Nada de reimplementar
  `parse_reference` em TS para "passar". A referência exibida tem de vir do Rust
  compilado para wasm. Anti-alucinação/uma-fonte-da-verdade são inegociáveis.
- **Bloqueio legítimo = `blocked` com erro EXATO (não é falha que conta retry).**
  Se a fronteira não compilar para `wasm32-unknown-unknown` por causa de
  `rusqlite`/`libsqlite3-sys`/SQLite-C (ou `reqwest`/TLS) e o core **não** for
  feature-gateável sem ser modificado, OU se o caminho web do `ubrn` for
  inexistente/imaturo/inoperante: reporte `status: blocked` colando a **saída
  exata** do `cargo build --target wasm32-unknown-unknown` (e/ou do ubrn web),
  descrevendo a fricção SQLite-no-WASM. Este é o **gate estratégico de Fase 0
  (Apêndice A #1)**; o Driver escreve HALT p/ o Guia/humano decidir a estratégia
  de store/transporte no web. **Não** improvise para "progredir".
- **Rede só em dev/build** (instalar target/ubrn, resolver deps). Offline-first é
  regra de runtime do produto; nada de rede entra no código gerado/no app.
  **Segredos nunca em git/log.**
- **ADR-0005 (se houver decisão).** Se — e somente se — uma decisão de
  arquitetura de store/transporte no web for efetivamente tomada e implementada
  nesta tarefa (ex.: confirmação de `wa-sqlite`+OPFS, ou web como leitura+IA),
  registre-a como **ADR-0005** em `DECISIONS.md`. Caso contrário, a decisão fica
  para o HALT/Guia — não invente escopo.
- **Qualidade por tarefa:** Rust `fmt`/`clippy -D warnings`/`build` verdes para a
  fronteira; TS `tsc --noEmit` limpo no `app/` com o glue ligado. Verde antes do
  commit (prefixo `feat(F0.6): …`).
