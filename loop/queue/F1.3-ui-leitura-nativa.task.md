## Objetivo

Entregar a **primeira UI de leitura NATIVA** (iOS/Android): navegação
**livro → capítulo → texto** + **seletor de versão** (KJV ⇄ Almeida 1911), lendo
**do store local no dispositivo** via a fronteira nativa exposta na F1.2
(`list_translations` / `list_books` / `get_chapter` / `chapter_count`) — Turbo
Module → JSI → `the-light-core`. Texto **sempre do store** (anti-alucinação).

O **subproblema central** desta tarefa é o **bundling do banco no app nativo**:
`get_chapter(db_path, ...)` (rusqlite, via core) precisa de um **caminho de
arquivo REAL e gravável** no device. O `bible.sqlite` (~47 MB, gerado-ignorado,
ADR-0013) precisa chegar ao app — empacotado como **asset** e **copiado** para um
caminho gravável (Documents/Caches) na 1ª execução. Decida a estratégia e
registre **ADR-0014**, documentando o **trade-off de tamanho**.

Escopo: **só nativo** nesta tarefa. A paridade web é a **F1.13** (depois do gate
estratégico F1.12) — não construa o caminho web aqui (apenas um stub que mantenha
`tsc`/Metro web verdes).

A entrega visual são as telas de navegação; a **PROVA determinística** é um
**self-test de leitura no device** (molde F0.7/F0.8) que exercita a leitura REAL
pela fronteira nativa e emite marcadores estáveis capturados por
`simctl log` (iOS) / `adb logcat` (Android).

## Contexto / arquivos

**Fronteira F1.2 (já aceita — NÃO mexer no core):**
- `core/src/lib.rs` — funções de leitura, todas com **gating de corpo por alvo**
  (`cfg(not(wasm32))` real no nativo; stub no web):
  - `list_translations(db_path) -> Vec<Translation>` (lê tabela `translations`)
  - `list_books() -> Vec<Book>` — **PURA** (cânon de 66 via `reference::BOOKS`;
    nome `name_en`/`name_pt`; `chapter_count` **canônico** por livro)
  - `get_chapter(db_path, translation, book, chapter) -> Passage` (capítulo
    inteiro numerado por versículo; `text` verbatim do store)
  - `chapter_count(db_path, translation, book) -> u16` — **DB-backed**
    (`max(chapter)` real do banco; `0` se ausente)
  - Records UniFFI já expostos: `Translation`, `Book`, `Testament`, `Verse`,
    `Passage`, `Reference`, `VerseRange`. Os **bindings gerados expõem tudo**
    (não é preciso tocar a fronteira).
  - Sanidade verificável no banco: João = livro **43**; `get_chapter(kjv,43,3)`
    contém v16 KJV verbatim; `chapter_count(kjv,43) == 21`; `list_books()` = 66.

**Glue nativo (molde a seguir):**
- `app/web/reference.ts` — glue NATIVO que **delega** ao barrel gerado
  (`./native-generated/src/index`) → JSI → Rust. `reference.web.ts` (wasm) é o par
  web; o Metro escolhe por extensão (`.ts` nativo / `.web.ts` web).
- `app/web/passage.ts` (stub nativo) + `app/web/passage.web.ts` — padrão de par
  nativo/web por extensão.
- `app/web/native-generated/bindings/the_light_app_core.ts` — bindings gerados
  (já contêm `listBooks`/`getChapter`/`chapterCount`/`listTranslations` da F1.2).
- `app/app/index.tsx` — tela atual (demo de `parseReference`; `getPassage` web).
- `app/app/_layout.tsx` — layout expo-router. `app/components/` e `app/lib/`
  existem vazios (`.gitkeep`) para os componentes/serviços do Reader.

**Modelo de self-test headless nativo (F0.7/F0.8 — REPLICAR):**
- `app/web/selftest.ts` — gancho `runReferenceSelfTest()` sob
  `EXPO_PUBLIC_TLA_SELFTEST=1`, dispara no mount de `HomeScreen`, compõe o marcador
  a partir do **retorno real** do Rust (sem eco/hardcode) e loga em
  `console.log`/`console.error`.
- `scripts/run-ios-selftest.sh` — boota iPhone 17 (iOS 26.5), Metro com o env,
  install+launch, captura `simctl spawn booted log stream --predicate
  'eventMessage CONTAINS "TLA_SELFTEST"'`, **asserta marcadores** e sai 0.
- `scripts/run-android-selftest.sh` — boota `thelight_avd` headless,
  `gradlew :app:installDebug`, Metro + `adb reverse`, captura
  `adb logcat -s ReactNativeJS:V`, **asserta marcadores** e sai 0.

**Bundling do banco (o subproblema):**
- `assets/data/bible.sqlite` (~47 MB) está presente localmente (gerado-ignorado,
  ADR-0013, `.gitignore` linha `/assets/data/bible.sqlite`). `scripts/gen-bible-db.sh`
  o (re)gera pelo `xtask import` do rev pinado `8f66004` **sem tocar `the-light`**.
- `assets/data/sample.sqlite` (~128 KB, versionado) é o subset da Fase 0 — NÃO é
  suficiente p/ a navegação completa, mas serve de molde de schema.
- O app deve **empacotar** o banco como asset (Metro `assetExts` já inclui `wasm`;
  acrescentar `sqlite`/`db`) e **copiar p/ caminho gravável** no 1º boot
  (`expo-asset` + `expo-file-system` — **não estão** em `app/package.json` ainda;
  adicionar). **Por que copiar (crítico):** no Android os assets vivem dentro do
  APK (não há path de arquivo real p/ o rusqlite); e `Store::open` roda **migrações
  idempotentes** (precisa de **write**) — logo o banco tem de estar num caminho
  **gravável** (`FileSystem.documentDirectory`), não no bundle read-only.
- `DECISIONS.md`: ADR-0010 (store nativo), ADR-0013 (bible.sqlite gerado-ignorado),
  ADR-0008/0009 (caminho iOS/Android). **Próximo número livre: ADR-0014.**

## Passos sugeridos

1. **Garantir o banco p/ o bundle.** Se `assets/data/bible.sqlite` ausente, rodar
   `./scripts/gen-bible-db.sh` (rede só em build; idempotente). Empacotar como
   asset: acrescentar `sqlite`/`db` ao `assetExts` do `app/metro.config.js`
   (padrão do `wasm`) e referenciar via `require(...)`/`expo-asset`. Garantir a
   resolução do asset a partir de `app/` (symlink/cópia em `app/assets/data/`,
   como no padrão do `sample.sqlite` ADR-0012). **bundled DB segue ignorado.**
2. **Bundling → caminho gravável (ADR-0014).** Adicionar `expo-asset` +
   `expo-file-system` ao `app/package.json`. Serviço (ex.: `app/lib/db.ts`) que no
   1º boot resolve o asset (`Asset.fromModule(...).downloadAsync()`), **copia** p/
   `FileSystem.documentDirectory + 'bible.sqlite'` (idempotente: só copia se
   ausente/tamanho divergente) e retorna o **path real** p/ passar a `get_chapter`.
   **Decidir e documentar no ADR-0014** a estratégia e o **trade-off de tamanho**:
   (a) empacotar o `bible.sqlite` completo (~47 MB); (b) gerar/empacotar um
   **subset de leitura** (que **DEVE conter João KJV completo — 21 capítulos** p/
   as asserções) deixando o banco completo p/ otimização posterior; ou (c)
   gzip + descompactar no 1º boot. Recomende a **opção mais simples que prove a
   leitura real no device**; registre o alvo escolhido p/ a prova (iOS OU Android).
3. **Glue nativo de leitura.** Criar `app/web/reading.ts` (nativo) que **delega**
   ao barrel gerado (`./native-generated/...`): `listBooks()`, `listTranslations(db)`,
   `getChapter(db, translation, book, chapter)`, `chapterCount(db, translation, book)`
   — **sem reimplementar SQL/leitura em TS**. Par `app/web/reading.web.ts` = stub
   que lança erro explícito ("leitura web = F1.13"), mantendo `tsc`/Metro web verdes.
4. **Telas de navegação (expo-router).** Em `app/app/` (grupo `(read)/...` ou
   rotas equivalentes) + componentes em `app/components/Reader*`: lista de
   **livros** (66, de `listBooks()` — puro), lista de **capítulos** do livro
   (`chapterCount(db, translation, book)`), **texto do capítulo**
   (`getChapter(db, translation, book, chapter)` → versículos numerados, texto
   verbatim do store) e um **seletor de versão** (`listTranslations(db)`; trocar
   recarrega o mesmo capítulo na outra tradução). Não quebrar a tela atual
   (`index.tsx` pode permanecer ou virar entrada da navegação).
5. **Self-test de leitura no device.** Estender `app/web/selftest.ts` (ou novo
   gancho sob `EXPO_PUBLIC_TLA_SELFTEST=1`): no boot, **copiar/abrir** o banco
   bundled (passo 2) e chamar pela fronteira nativa `list_books()`,
   `get_chapter(db,"kjv",43,3)`, `chapter_count(db,"kjv",43)`, emitindo um marcador
   estável **composto do retorno real** (sem hardcode do texto):
   `TLA_READ books=66 john3_v16="For God so loved..." john_chapters=21`. Manter o
   self-test de parse (PT/EN) intacto (sem regressão F0.x).
6. **Provar e capturar.** Estender `scripts/run-ios-selftest.sh` **OU**
   `scripts/run-android-selftest.sh` (pelo menos um) p/ **assertar também** os
   marcadores de leitura (`books=66`, substring verbatim `For God so loved the
   world`, `john_chapters=21`), além dos PT/EN existentes. Rodar e capturar o log.
   O outro alvo pode ser adicional. Registrar a escolha no ADR-0014.
7. **Qualidade + ADR.** `cd app && npx tsc --noEmit` verde; `DECISIONS.md` ganha
   **ADR-0014** (bundling do banco no app nativo: asset + copy p/ path gravável,
   deps `expo-asset`/`expo-file-system`, `assetExts`, estratégia de tamanho +
   trade-off, alvo da prova). Working tree limpo (gerados/bundled ignorados).

## Restrições

- **NÃO modificar/forkar `the-light`** nem a fronteira `core/src/lib.rs`. Se a
  leitura exigisse mudar o core/fronteira → **HALT** (PR + ADR é ação humana).
- **NÃO reimplementar SQL/leitura no app** — toda leitura passa pela **fronteira
  nativa** (`get_chapter`/`list_books`/`list_translations`/`chapter_count`). O glue
  TS só embrulha o retorno (sem parser/SELECT/eco em TS).
- **Anti-alucinação:** o texto do versículo vem **sempre do store** (KJV/Almeida,
  **domínio público**); **nunca** gerar/hardcodear texto bíblico em produção. No
  self-test, o `john3_v16` provém do **retorno de `get_chapter`**; a asserção do
  script usa um substring esperado, não a fonte.
- **Bundling sem inchar absurdamente:** documentar o **trade-off de tamanho** no
  **ADR-0014**. Se optar por subset de leitura, ele **DEVE conter João KJV
  completo (21 capítulos)** p/ as asserções valerem.
- **Gerados/bundled ignorados:** `bible.sqlite`, asset copiado em runtime,
  `app/ios`, `app/android`, `app/web/native-generated`, jniLibs/`.so` permanecem no
  `.gitignore`; working tree limpo.
- **Offline-first:** **zero rede em runtime** (a leitura é I/O local no device);
  rede só em dev/build (gen-bible-db, Metro, Gradle, pods, npm).
- **Segredos nunca em git/log.**
- **BLOQUEIO (`blocked` + HALT) com erro EXATO** se o bundling de DB no Expo nativo
  for **inviável sem hack proibido** (ex.: exigir mudar o core/`the-light`, depender
  de COOP/COEP/SharedArrayBuffer, ou empacotar de forma que o rusqlite não consiga
  abrir um path real gravável) → reportar a saída exata e propor a decisão/ADR;
  não improvisar.
