## Objetivo

Segunda etapa da trilha de sync (ADR-0036): permitir que o usuário **linke a própria
conta Google no WEB** para que, na F5.25, o snapshot da F5.23 possa ir/vir da pasta
**app-data privada** do Drive DELE. Esta tarefa entrega **apenas o fluxo de autorização**
(link/unlink + gestão de token) — push/pull do snapshot é F5.25.

Offline-first permanece a **base**: o app é 100% funcional com ZERO conta/rede. Isto é
estritamente **opt-in** e **aditivo**. Nada essencial passa a exigir Google.

## Contexto / arquitetura (ADR-0036, reconciliação com ADR-0023)

- **OAuth 2.0 com PKCE client-side, SEM servidor do app.** Client-id **público** + PKCE
  (S256), **sem client-secret** (não há backend p/ guardá-lo). Isto reconcilia com o
  ADR-0023 ("OAuth banido"): banido era OAuth **essencial/com servidor**; aqui é um
  transporte **opcional** de dados DO usuário, na conta DO usuário, sem infra do app.
- **Escopo mínimo: `https://www.googleapis.com/auth/drive.appdata`** — acesso somente à
  pasta oculta app-private (não ao Drive inteiro do usuário; não lê os arquivos dele).
- **Token nunca em git/log.** Access token em memória de sessão; refresh (se houver) em
  secure storage do alvo. O client-id público pode ficar em config (`app/lib/driveAuth`),
  mas **nada de segredo** entra no repo.
- **Web-only** (esta trilha do Drive). Nativo (iOS/Android) fica no export/import manual
  da F5.23; wiring nativo do motor de snapshot é F5.26.

## Escopo desta tarefa

1. **`app/lib/driveAuth.ts` (motor PURO, injeção de dependências).** Sem efeitos colaterais
   de rede embutidos — recebe `fetch`, `crypto`, `redirectUri`, `clientId`, e um
   `TokenStore` injetável (get/set/clear). Expor:
   - `buildAuthUrl({clientId, redirectUri, scope, codeChallenge, state})` — monta a URL de
     autorização Google (`accounts.google.com/o/oauth2/v2/auth`), `response_type=code`,
     `code_challenge_method=S256`, `access_type=offline`, `prompt=consent`.
   - `generatePkce(crypto)` — `code_verifier` (43–128 chars) + `code_challenge` = BASE64URL(SHA-256(verifier)).
   - `exchangeCode({code, codeVerifier, clientId, redirectUri, fetch})` — troca code→token
     em `oauth2.googleapis.com/token` (PKCE, sem secret); retorna `{accessToken, expiresAt, refreshToken?}`.
   - `LinkState` = `linked{email?,expiresAt} | unlinked`; `link()`/`unlink()` que gravam/limpam
     o `TokenStore`; `isLinked()`; `currentToken()` (com checagem de expiração).
   - **Nunca** logar token/verifier/challenge.
2. **Anti-alucinação/privacidade:** esta tarefa NÃO toca dados do usuário nem o snapshot;
   só autorização. Escopo `drive.appdata` (não lê o Drive do usuário).
3. **Prova headless** (`app/web/__tests__/driveAuth.web.test.mjs`, mock `fetch`+`crypto`):
   - `generatePkce` determinístico p/ um verifier fixo → challenge conhecido (vetor de teste).
   - `buildAuthUrl` inclui scope `drive.appdata`, `code_challenge_method=S256`, `response_type=code`, o state e o redirect.
   - `exchangeCode` faz POST ao token endpoint com `grant_type=authorization_code`+`code_verifier` (mock retorna token) → `link()` grava no TokenStore → `isLinked()==true` → `unlink()` limpa → `isLinked()==false`.
   - **Nenhum token/verifier aparece** no output da prova (asserção explícita).
   - Marcador final `DRIVE_AUTH pkce=ok url=ok exchange=ok link=ok unlink=ok notoken=ok`.
   - Script `test:web:driveauth`.
4. **MOCK apenas.** NENHUMA chamada real ao Google. Validação com conta/Drive REAIS é a
   **F5.27** (gate — humano; conta/token nunca transitam pelo loop).
5. **ADR-0052** — decisão do fluxo de autorização (PKCE público, escopo appdata,
   armazenamento de token, reconciliação com ADR-0023).

## Fora de escopo (tarefas seguintes)

- Push/pull do snapshot na pasta app-data + merge no pull → **F5.25**.
- UI de opt-in (link/unlink, aviso de privacidade, "funciona offline sem isto") + wiring
  nativo do motor de snapshot → **F5.26**.
- Validação com conta Google REAL → **F5.27 (⛔ gate humano)**.
