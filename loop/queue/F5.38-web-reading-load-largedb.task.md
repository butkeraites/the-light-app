## Bug (diagnóstico confirmado)

Após a F5.36 (Bíblia completa, `reading-lite.sqlite` 4,3→38,4MB), **Mateus 1 não abre no WEB**
mesmo com o Metro servindo o asset correto (verificado: `Content-Length=40308736`). Causa raiz
isolada por repro em Node: o DB + `MemoryVFS` + wa-sqlite **funcionam** (Node abre o 38MB e lê
Mateus 1 "The book of the generation of Jesus Christ..." + 66 livros). O que FALHA é o passo
**exclusivo do browser** que o Node não exercita: `openReadingDbWeb` faz
`fetch(asset 38MB) → grava em OPFS (createWritable().write(38MB)) → lê de volta → MemoryVFS`
(`app/web/sqlite-reading-opfs.web.ts:80-99,123`). A gravação de 38MB em OPFS é frágil (e o
anônimo/incognito tem quota de storage RESTRITA → a escrita falha silenciosamente). O caminho
de prova Node (`readFile → MemoryVFS`) é o análogo do fetch-direto e funciona.

## Objetivo

Fazer o store web de leitura carregar o DB de forma robusta para qualquer tamanho, eliminando o
ponto de falha do round-trip OPFS. **Leitura funciona em janela normal E anônima.**

## Escopo

1. **`app/web/sqlite-reading-opfs.web.ts`:** em `openReadingDbWeb`, carregar os bytes do DB
   **direto do asset empacotado para o `MemoryVFS`** — `fetch(readingDbUri).arrayBuffer()` →
   `MemoryVFS` — **sem** gravar/ler em OPFS. Remover (ou desligar) `readSubsetBytesFromOpfs`
   para o caminho de leitura. Justificativa: o `reading-lite.sqlite` é asset LOCAL empacotado
   (mesma origem, offline-first já garantido pelo próprio asset) — o OPFS não adicionava
   offline-first à leitura (dado read-only), só um cache que agora quebra em 38MB. Bônus: some o
   problema de invalidação de cache/auto-reseed (sempre lê o asset atual).
   - Manter a instância do wa-sqlite (build FTS5 vendored) e as queries de produção
     (`queryChapter`/`queryChapterCount`/`queryTranslations`) INALTERADAS.
   - Se houver preocupação de memória: o repro Node provou que 38MB no `MemoryVFS` abre e
     consulta OK (mesmo wasm do browser). Aceitável. NÃO reintroduzir OPFS.
   - Verificar que NENHUM outro consumidor dependia de `readSubsetBytesFromOpfs`/OPFS de leitura
     (grep); o léxico (`sqlite-lexicon-opfs.web.ts`) e o userdata OPFS são SEPARADOS — não tocar.
2. **Teste que exercita um livro ALÉM de João:** o `reading.web.test.mjs` só consulta João(43).
   Adicionar asserção de **Mateus (book_number 40, cap 1, KJV)** via as funções de produção
   (queryChapter) sobre o `reading-lite.sqlite` completo — texto VERBATIM ("The book of the
   generation of Jesus Christ...") — para que uma regressão de cobertura de livros seja pega.
   (Isto roda pelo caminho Node/readFile→MemoryVFS; a falha era só do OPFS do browser, que Node
   não tem — mas garante que a query de um livro do NT funciona ponta a ponta.)
3. Confirmar `tsc`, `expo export --platform web`, `test:web:reading`/`search`/`xref`/`coverage`,
   `test:web:perf-budget` verdes. Anti-alucinação intacta (texto do store).

## Aceite

- [ ] `openReadingDbWeb` carrega o DB via `fetch → MemoryVFS` direto (sem round-trip OPFS de
      escrita/leitura de 38MB); leitura funciona em janela normal e anônima.
- [ ] `reading.web.test.mjs` (ou coverage) passa a asserir Mateus 1 (KJV) VERBATIM além de João.
- [ ] `tsc` 0; `test:web:reading`/`search`/`xref`/`coverage`/`perf-budget` verdes; `expo export --platform web` 0.
- [ ] Nenhum outro caminho quebrado (léxico/userdata OPFS intocados); the-light `225b8c9`; sem segredo.
- [ ] (Opcional) ADR-0057 registrando "leitura web = fetch-direto do asset p/ MemoryVFS; OPFS só p/ userdata".
