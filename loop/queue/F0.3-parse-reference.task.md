## Objetivo

Expor `parse_reference(input: String) -> Result<Reference, CoreError>` na fronteira
UniFFI do crate `core/` (`the-light-app-core`), **delegando a lógica de parsing ao
`the-light-core::reference`** — sem reimplementar parsing no app (regra "uma fonte
da verdade"). O retorno deve ser um tipo **serializável via UniFFI** (record/struct
com os campos da referência: livro, capítulo, versículo etc., conforme a API real do
core descoberta na fonte clonada).

Esta tarefa **exerce a resolução real do `the-light-core`** que a F0.2 transferiu
para cá (ver ADR-0002 / ADR-0003): adicionar a git dependency pinada por commit,
fazer `cargo build` resolver/baixar o core, descobrir a API real e delegar a ela.

## Contexto / arquivos

**Estado atual (F0.2 aceita, commit `7b922eb`):** `core/Cargo.toml` declara
`the-light-app-core` com `crate-type = ["lib","cdylib","staticlib"]`, depende de
`uniffi = "0.31.2"` (fixado) + `thiserror = "2.0.18"`, e **não** depende do
`the-light-core`. `core/src/lib.rs` usa modo *library* (`uniffi::setup_scaffolding!()`),
exporta `ping()`/`ping_checked()` e já define o enum de erro
`#[derive(Debug, thiserror::Error, uniffi::Error)] pub enum CoreError { Generic { message: String } }`.
Reutilize esse `CoreError` para mapear o erro do core.

**Repositório do core (apurado pelo Driver — não re-apure):**
- O `the-light` é **PÚBLICO e alcançável**. URL:
  `https://github.com/butkeraites/the-light`.
- Commit HEAD **pinável** (apurado via `git ls-remote` pelo Driver):
  **`0888ac0bfed15222874c5462902a61ac3a39147c`**. Use **exatamente** esse `rev`
  para pinar a git dependency (ADR-0002 escolheu git dep pinada por commit).
  Você **NÃO** precisa rodar `git ls-remote` (está fora da allow-list dos
  subagentes); apenas escreva o `rev` no `Cargo.toml` e o `cargo` faz o fetch
  (`cargo` está na allow-list).

**Arquivos a tocar:**
- `core/Cargo.toml` — adicionar a git dep pinada:
  ```toml
  the-light-core = { git = "https://github.com/butkeraites/the-light", rev = "0888ac0bfed15222874c5462902a61ac3a39147c", package = "the-light-core" }
  ```
  Ajuste o campo `package` ao **nome real do crate** descoberto na fonte clonada
  (ver mecanismo de descoberta abaixo). Se o crate estiver dentro de um workspace,
  o cargo resolve por **nome de package** — basta o `package = "<nome real>"`
  bater com o `name` no `Cargo.toml` do crate dentro do repo.
- `core/src/lib.rs` — a função `parse_reference` + o(s) tipo(s) UniFFI de retorno
  (record/struct) + o mapeamento do erro do core para o `CoreError` já existente.
- `core/Cargo.lock` — versionado; será atualizado pelo `cargo build`.

**Mecanismo de descoberta da API (importante):** você **NÃO pode** ler o sibling
local `../the-light` (negado por política). **Porém**, ao adicionar a git dep e
rodar `cargo build`, o cargo **clona** o repo para
`~/.cargo/git/checkouts/the-light-*/<rev-curto>/` — esse caminho **NÃO** é negado.
Você **PODE e DEVE** ler a fonte clonada lá para descobrir a API exata
(nome do crate/package, módulo `reference`, assinatura da função de parse, nome e
campos do tipo de retorno, tipo de erro do core). Isso é **CONSUMIR** a
dependência, **não** modificá-la — totalmente dentro das regras.

## Passos sugeridos

1. **Adicionar a git dep pinada** em `core/Cargo.toml` exatamente com a URL e o
   `rev = "0888ac0bfed15222874c5462902a61ac3a39147c"` acima (campo `package`
   provisório `"the-light-core"`, a confirmar no passo 3).
2. **`cargo build`** (de `core/`) para o cargo fazer o **fetch/clone** do repo. Se
   o `package` não bater com o nome real, o build falha com o nome esperado — use
   isso e o passo 3 para corrigir.
3. **Ler a fonte clonada** em `~/.cargo/git/checkouts/the-light-*/<rev>/` para
   descobrir o **nome real do crate/package** (campo `name` no `Cargo.toml` do
   crate), o **módulo `reference`** e a **assinatura real** da função de parse de
   referência: nome da função, **tipo de retorno** (struct/enum `Reference` e seus
   campos), e o **tipo de erro** do core. Ajuste `package = "<nome real>"` no
   `core/Cargo.toml` se necessário.
4. **Implementar a fronteira UniFFI** em `core/src/lib.rs`:
   - Definir um tipo UniFFI serializável de retorno (ex.:
     `#[derive(uniffi::Record)] pub struct Reference { ... }`) **espelhando os
     campos reais** do tipo do core (livro, capítulo, versículo etc.). Se o tipo do
     core já for diretamente exportável, ainda assim defina o record da fronteira
     para manter a API estável e desacoplada.
   - `#[uniffi::export] pub fn parse_reference(input: String) -> Result<Reference, CoreError>`
     que **chama** `the_light_core::reference::<fn real>(...)`, converte o sucesso
     no record da fronteira e **mapeia o erro do core** para `CoreError` (ex.:
     `CoreError::Generic { message: e.to_string() }` ou uma variante mais específica
     se fizer sentido — sem reimplementar lógica).
   - **Não** escrever lógica de parsing própria: apenas adaptar tipos e erro.
5. **Escrever testes** em `core/src/lib.rs` (`mod tests`):
   - `"Jo 3.16"` (PT) e `"John 3:16"` (EN) resolvem para a **mesma** referência
     (mesmos campos livro/capítulo/versículo) — assert de igualdade dos campos
     relevantes.
   - Uma entrada inválida (ex.: `"xyz não é referência"`) retorna `Err` mapeado
     para `CoreError`.
6. Rodar de `core/`: `cargo fmt`, `cargo clippy -- -D warnings`, `cargo build`,
   `cargo test` — **tudo verde** antes do commit.
7. Commitar com prefixo do ID, ex.:
   `feat(F0.3): parse_reference na fronteira UniFFI delegando ao the-light-core`.

## Critério de aceite

- [ ] `core/Cargo.toml` declara a git dep do `the-light-core` pinada **pelo rev

- [ ] `core/src/lib.rs` exporta `parse_reference(input: String)` retornando um tipo

- [ ] O erro do core é **mapeado** para o `CoreError` já definido na F0.2.
- [ ] `cargo build` e `cargo test` (de `core/`) ficam **verdes**.

- [ ] `git status --porcelain` vazio após o commit (working tree limpo;

## Comando(s) de verificação

```sh
set -e
ROOT="$(git rev-parse --show-toplevel)"

# 1) Qualidade Rust do crate core (rodar de dentro de core/)
cd "$ROOT/core"
cargo fmt --check               && echo "OK  cargo fmt --check limpo"
cargo clippy -- -D warnings     && echo "OK  cargo clippy sem warnings"
cargo build                     && echo "OK  cargo build verde (git dep resolvida)"
cargo test                      && echo "OK  cargo test verde"

# 2) Pino EXATO do rev na git dep do the-light-core
grep -q 'rev = "0888ac0bfed15222874c5462902a61ac3a39147c"' "$ROOT/core/Cargo.toml" \
  && echo "OK  git dep pinada pelo rev exato 0888ac0..." \
  || { echo "FALTA git dep do the-light-core pinada pelo rev exato"; exit 1; }

grep -q 'github.com/butkeraites/the-light' "$ROOT/core/Cargo.toml" \
  && echo "OK  URL do the-light presente na git dep" \
  || { echo "FALTA URL https://github.com/butkeraites/the-light em core/Cargo.toml"; exit 1; }

# 3) Delegação real: lib.rs USA o the_light_core (não reimplementa parsing)
grep -q 'the_light_core' "$ROOT/core/src/lib.rs" \
  && echo "OK  parse_reference delega ao the_light_core (uma fonte da verdade)" \
  || { echo "FALHA lib.rs não referencia the_light_core (parsing não delegado)"; exit 1; }

# 4) A função existe na fronteira
grep -q 'fn parse_reference' "$ROOT/core/src/lib.rs" \
  && echo "OK  parse_reference exportado na fronteira" \
  || { echo "FALTA fn parse_reference em core/src/lib.rs"; exit 1; }

# 5) Working tree limpo após o commit
cd "$ROOT"
test -z "$(git status --porcelain)" \
  && echo "OK  working tree limpo" \
  || { echo "FALHA working tree sujo:"; git status --porcelain; exit 1; }

echo "F0.3 verificada."
```

## Restrições

- **NÃO modificar / forkar / copiar / vendorizar o `the-light`.** Consuma-o
  **apenas** via git dep pinada por `rev`. Qualquer mudança no core seria
  **PR + ADR = HALT** — nunca fork divergente silencioso. Ler a fonte clonada em
  `~/.cargo/git/checkouts/` é **consumo** (permitido), não modificação.
- **NÃO reimplementar `parse_reference` no app.** O parsing vem do
  `the-light-core::reference`; a fronteira só adapta tipos e erro. A verificação
  exige a presença de `the_light_core` em `lib.rs` (delegação).
- **Offline-first é regra de RUNTIME do produto**, não do dev: o **fetch/clone do
  cargo** em build/dev é permitido; o parsing em si é **puro** (sem rede). Nenhuma
  chamada de rede pode ser introduzida no código do crate.
- **Bloqueio legítimo, não improviso:** se o `the-light-core` **não resolver/buildar**
  (repo inacessível, crate com nome diferente do esperado e **não descobrível** na
  fonte clonada, falha de build por feature/toolchain, ou API incompatível com
  `parse_reference`), reporte **`blocked`** com o **erro EXATO** do cargo/compilador
  — **NÃO** improvise fork, cópia, vendorização ou reimplementação do parsing.
- **Qualidade obrigatória:** `cargo fmt` / `cargo clippy -- -D warnings` /
  `cargo build` / `cargo test` verdes antes do commit.
- **Segredos nunca em git** (nada de tokens/chaves/`.env`; o repo do core é público
  e o acesso é anônimo).
