## Objetivo

Gerar o **banco SQLite completo embarcado** com **duas versões de domínio
público** — **KJV** (EN, 31.102 versículos) e **Almeida 1911** (PT, 31.101
versículos) — **reaproveitando o importador `xtask` do `the-light`** (rev pinado
`8f66004`, ADR-0002), e empacotá-lo como asset do app em
`assets/data/bible.sqlite`. O processo deve ser **reprodutível** (um script),
**idempotente** (reimportar não duplica) e **sem modificar/forkar `the-light`**.

Esta tarefa também **decide e registra (ADR-0013)** a **estratégia de
armazenamento** do asset gerado (banco grande): **versionar** vs **gerar por
script (ignorado)** vs **Git LFS** — e atualiza o `.gitignore` de forma coerente.

Anti-alucinação e licenciamento: o texto bíblico vem **sempre** do importador
canônico sobre fontes **de domínio público** registradas no `xtask` — **nunca**
texto inventado nem versão protegida.

## Contexto / arquivos

**Importador canônico (a EXECUTAR, não reimplementar) — `xtask` do `the-light`:**
O `the-light` (pinado `rev 8f66004`) tem um workspace member `xtask` com o
subcomando `import`. A lógica de download/parse/insert e o registro de versões
livres (`SPECS`) vivem **só** no `xtask` (não no `the-light-core` que consumimos
como lib) — por isso o plano manda **rodar o xtask**, não duplicar a lógica.

- Fonte do importador (leitura, **consumo pinado** já presente no checkout do
  cargo): `~/.cargo/git/checkouts/the-light-9eb8809a6d68281a/8f66004/xtask/src/import.rs`
  (cópia read-only do `rev 8f66004` que já compilamos). O repo canônico em
  `/Users/butkeraites/Documents/the-light` é **leitura-apenas / não tocar**.
- **Subcomando e flags** (de `xtask/src/main.rs` / `import.rs`):
  `cargo run -p xtask -- import --version kjv,alm1911 --db <PATH> --seed-dir <DIR> [--offline] [--force]`
  - `--version kjv,alm1911` — as duas versões livres registradas em `SPECS`.
  - `--db <PATH>` — destino do banco (use `assets/data/bible.sqlite`).
  - `--seed-dir <DIR>` — onde os datasets brutos são **baixados/cacheados**
    (padrão `data/seed`). Aponte para um diretório **ignorado** do app.
  - `--offline` — **falha** em vez de baixar (usa só cache do seed-dir).
- **Fontes (domínio público, URLs fixadas em `SPECS`):**
  - `kjv` — King James Version, `public-domain`, scrollmapper
    (`.../formats/json/KJV.json`, ~8,4 MB), **31.102** versículos.
  - `alm1911` — Almeida 1911, `public-domain`, damarals release v1.0.0
    (`.../ALM1911.json`, ~4 MB, segue redirects), **31.101** versículos.
  - O `xtask` **valida a contagem** (`expected_verses`) e **falha** se < 30.000;
    **avisa** se ≠ exato (drift do dataset). Idempotência: `import_translation`
    apaga as linhas da versão (`translations`/`books`/`verses`/`verses_fts`)
    antes de reinserir → reimportar é estável.
- **Schema** (uma fonte da verdade — migrações do core, aplicadas por
  `Store::open`): `translations(id,abbrev,name,language,license,embeddable)`,
  `books(...)`, `verses(id,translation_id,book_number,chapter,verse,text)`,
  `verses_fts` (FTS5).

**Como rodar o xtask SEM modificar/forkar `the-light` (decisão → ADR-0013):**
escolha um caminho que **não escreva no source** do `the-light`:
- **(preferido) checkout pinado do cargo:** rodar o member `xtask` do checkout
  `~/.cargo/git/checkouts/the-light-9eb8809a6d68281a/8f66004` via
  `cargo run --manifest-path <checkout>/xtask/Cargo.toml -- import ...`,
  **com `CARGO_TARGET_DIR` apontando para FORA do checkout** (ex.: um `target/`
  do app ou um scratch ignorado) — assim nenhum artefato é escrito dentro do
  `the-light`. (O checkout do cargo é o **mesmo rev** que consumimos: `8f66004`.)
- **(alternativa) clone efêmero pinado:** `git clone` do `the-light` no rev
  `8f66004` para um diretório **de build/scratch ignorado**, rodar o xtask lá, e
  **não commitar** nada. Mais isolado, porém re-baixa o source.
Registrar a escolha no ADR-0013. **Não** rodar o xtask dentro de
`/Users/butkeraites/Documents/the-light` (escreveria no working tree do repo
protegido).

**Tamanho do asset e armazenamento (decisão → ADR-0013):** o banco completo (2
versões + FTS5) é **grande** (provável dezenas de MB). Diferente do
`sample.sqlite` (~128 KB, **versionado**, ADR-0010), versionar um binário grande é
indesejável. Opções:
- **(provável) gerar por script + IGNORAR** `assets/data/bible.sqlite` no git —
  o `scripts/gen-bible-db.sh` é a fonte reprodutível; o banco é um artefato de
  build (como `bindings/`, `rust_modules/`). O **bundling** no app nativo/web é
  fase posterior (F1.13+ / empacotamento).
- **Git LFS** — se o time preferir um asset versionado grande; exige `git lfs`
  configurado. Registrar prós/contras.
O `seed-dir` (datasets brutos JSON) é **sempre ignorado** (fora do git, como já
manda o `import.rs`).

**Reaproveitar o padrão F0.9 (sample):** `scripts/gen-sample-db.sh` +
`core/examples/gen_sample_db.rs` já mostram como gerar um banco com o schema do
core. A F1.1 é o **upgrade**: do subset (1 versículo) para o **corpus completo**
via o **xtask canônico** (não um exemplo local), porque só o xtask conhece as
fontes e os formatos (scrollmapper / thiagobodruk).

**Documentos:** `DECISIONS.md` (ADR-0002 git dep pinada `rev 8f66004`; ADR-0005
matriz por alvo / feature `embedded`; ADR-0010 schema/store nativo + sample);
`IMPLEMENTATION_PLAN.md` (§FASE 1 / F1.1; Apêndice B `store`/`model`);
`VISION_AND_ARCHITECTURE.md` (§6 modelo de dados: KJV/Almeida embarcadas, geradas
pelo `xtask`; §3 offline-first); `loop/backlog/PHASE-1.md` (F1.1).

## Passos sugeridos

1. **`scripts/gen-bible-db.sh`** (versionado, reprodutível, `set -e`):
   - resolve o `xtask` do **rev pinado** `8f66004` (checkout do cargo ou clone
     efêmero — ver Contexto), com `CARGO_TARGET_DIR` **fora** do `the-light`;
   - cria `assets/data/` e um `seed-dir` ignorado (ex.: `assets/data/seed/` ou

     --version kjv,alm1911 --db assets/data/bible.sqlite --seed-dir <seed-dir>`
     (rede em build permitida); aceita `--offline`/`--force` repassados.
2. **Validar contagens** (guarda de aceite, **lendo do banco**, não hardcode):
   `kjv` = 31.102 e `alm1911` = 31.101 (e 66 livros por versão; FTS5 povoada).
3. **Idempotência:** rodar o script 2× → contagens **estáveis** (sem duplicar).
4. **Decidir armazenamento (ADR-0013):** medir o tamanho real do
   `bible.sqlite`; escolher versionar / gerar-ignorado / LFS; ajustar
   `.gitignore` (seed-dir sempre ignorado; `bible.sqlite` conforme decisão).
5. **Atualizar `PROGRESS.md`** (ID, data, resumo, commit) e **`DECISIONS.md`**
   (ADR-0013). Commit com prefixo `F1.1`. Working tree **limpo** (artefatos
   ignorados conforme a decisão).

## Restrições

- **NÃO modificar nem forkar `the-light`** (`/Users/butkeraites/Documents/the-light`
  é leitura-apenas; consumo é git dep pinada `rev 8f66004`, ADR-0002). Rodar o
  `xtask` **sem** escrever no source dele (`CARGO_TARGET_DIR` fora do checkout, ou
  clone efêmero ignorado). Se a **única** forma de invocar o importador canônico
  exigir tocar/alterar o `the-light` → `blocked` com o erro EXATO (decisão humana:
  como expor/rodar o xtask) — **não** improvisar uma cópia paralela da lógica.
- **Só versões de DOMÍNIO PÚBLICO** (KJV, Almeida 1911), exatamente as registradas
  no `SPECS` do xtask. **Jamais** apontar para versão protegida nem adicionar
  outra fonte. **Anti-alucinação:** nenhum texto bíblico inventado/hardcoded.
- **Offline-first é regra de RUNTIME:** rede só em **dev/build** (download dos
  datasets de domínio público). O app em runtime **não** faz rede. **Segredos
  nunca** em git/log.
- **Blocked legítimo (rede/seed):** os datasets **não estão em cache**
  (`data/seed/` do core está vazio) → o xtask **baixa por rede**. Se o ambiente
  do loop estiver **offline** e não houver seed local disponível, rodar com
  `--offline` falhará → escrever `done/F1.1.result.md` com `status: blocked`, a
  **saída literal** do comando e a decisão pendente (origem/seed dos dados, ou
  **re-escopo** para um subset **maior** verificável — registrar em ADR). **Não**
  fabricar dados para "passar".
- **Tamanho/armazenamento:** decidir em **ADR-0013** (versionar vs gerar-ignorado
  vs Git LFS); o seed-dir é **sempre ignorado**. Não commitar binário grande sem a
  decisão registrada.
- **Escopo:** F1.1 entrega **o pipeline + o banco + a validação de contagem**. A
  leitura via fronteira (`list_books`/`get_chapter`/...) é **F1.2**; o **bundling**
  do banco no app nativo/web e a paridade web são fases posteriores (F1.13+). Não
  antecipar.
