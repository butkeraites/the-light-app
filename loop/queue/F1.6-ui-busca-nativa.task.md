## Objetivo

Entregar a **UI de busca NATIVA** (iOS/Android): um **campo de busca** + uma
**lista de resultados** onde cada acerto mostra a **referência** (ex.: "John
3:16") e o **trecho/snippet** com o **termo destacado**, e cuja **referência é
clicável** — ao tocar, navega para o **versículo** no Reader (rota `read/[book]/
[chapter]` da F1.3, idealmente ancorando/realçando o versículo).

A busca **lê SEMPRE pela fronteira** `search(db, query, translation, book?,
limit?)` exposta na **F1.5** (Turbo Module → JSI → `the-light-core::search`,
FTS5/BM25, acento-insensível). **Nada de SQL/FTS/`MATCH`/`bm25`/`highlight` é
reimplementado em TS** — a UI só embrulha o retorno (uma fonte da verdade;
anti-alucinação: o texto do hit vem **verbatim do store local**).

O **destaque do termo** vem de `SearchHit.highlighted`, que carrega marcadores de
controle `HL_START` (`\u{2}`) / `HL_END` (`\u{3}`) ao redor do termo casado (do
core, F1.5). A UI **converte esses marcadores em estilo** (negrito/realce) — e
**nunca exibe os caracteres de controle** como texto cru.

Escopo: **só nativo** nesta tarefa. A **paridade web da busca é a F1.14** (depois
do gate estratégico F1.12) — não construir o caminho web aqui (apenas manter
`tsc`/Metro web verdes: o glue web de busca continua um stub que lança erro).

A entrega visual é a tela (campo + lista + navegação ao clicar); a **PROVA
determinística** é um **self-test de busca no device** (molde F1.3/F1.4) que
exercita a busca REAL pela fronteira e emite um marcador estável composto do
**retorno real** de `search`.

## Passos sugeridos

1. **Regenerar os bindings nativos** (incluem `search`/`SearchHit` da F1.5):
   rodar `scripts/gen-bindings-ios.sh` (e/ou o build do `run-ios-selftest.sh`, que
   reconstrói o Turbo Module). Confirmar que `app/web/native-generated/bindings/
   the_light_app_core.ts` passa a expor `search(...)` e o tipo `SearchHit`.
2. **Estender o glue de busca** em `app/web/reading.ts`: adicionar
   `export async function search(dbPath, query, translation, book?, limit?):
   Promise<SearchHit[]>` delegando ao binding gerado `search` e reexportar
   `type SearchHit`. **Sem** SQL/FTS/parsing em TS. Estender `reading.web.ts` com
   um `search(...)` que lança `WEB_MSG` (busca web = F1.14) + reexportar o tipo.
3. **Povoar o FTS no subset (pipeline de dados — guardado):** estender
   `core/examples/gen_reading_sample_db.rs` p/, **após copiar `verses`**, preencher
   `verses_fts(text, translation_id, verse_id)` a partir do **próprio subset**
   (`INSERT INTO verses_fts(text, translation_id, verse_id) SELECT text,
   translation_id, id FROM verses;` — `verse_id` casando `verses.id`, como o core
   faz nos seus testes). Regerar `reading-sample.sqlite` (`scripts/
   gen-reading-sample-db.sh`; gerar-ignorado). **Isto é dado/fixture**, não busca.
   *Se* a API/schema do core não permitir povoar o FTS sem mudar o core →
   **`blocked`/HALT** com o erro EXATO (não improvisar).
4. **Tela/rota de busca** em `app/app/search/index.tsx` (registrar `search/index`
   no `Stack` de `app/app/_layout.tsx`, com `readOptions` temáticos): um
   `TextInput` (campo de busca) com **debounce** (~250–400 ms) que, a cada termo
   estável, resolve `ensureReadingDb()` e chama `search(dbPath, query,
   translation)` (tradução default `kjv`; pode reusar `listTranslations` p/ um
   seletor simples, **opcional**) e renderiza a **lista** de resultados (FlatList).
   Adicionar um **ponto de entrada** p/ a busca (ex.: botão/link no header ou em
   `app/app/index.tsx`/`read/index.tsx`) que faz `router.push('/search')`.
5. **Componente de resultado** `app/components/ReaderSearchResult*` (ex.:
   `ReaderSearchResultItem.tsx`) que renderiza, **com tokens de tema**:
   - a **referência** legível (ex.: "John 3:16") composta de
     `hit.reference.book`/`chapter`/`verses` + nome do livro (`listBooks()` →
     `nameEn`/`namePt`); e
   - o **snippet** a partir de `hit.highlighted`: **dividir** a string nos
     marcadores `HL_START`=`\u{2}`/`HL_END`=`\u{3}` e renderizar runs alternados
     `<Text>` (run casado em **negrito/realce** via token; runs normais via
     `verseText`). **Os caracteres de controle NUNCA aparecem** na tela (são
     consumidos no split). Centralizar `HL_START`/`HL_END` em const(s).
6. **Referência clicável → navegação:** o item é `Pressable`/`onPress` →
   `router.push('/read/<book>/<chapter>')` (rota da F1.3), navegando ao versículo.
   **Opcional/best-effort (não bloquear):** passar `verse` como param
   (`router.push({ pathname: '/read/[book]/[chapter]', params: { book, chapter,
   verse } })`) e fazer `[chapter].tsx`/`ReaderChapterView` **realçar/ancorar** o
   versículo alvo (ex.: destaque por token de cor; scroll se trivial). Se a
   ancoragem não for trivial, entregar a **navegação ao capítulo** (já satisfaz o
   aceite) e deixar a ancoragem registrada como follow-up.
7. **Self-test de busca** (par nativo): no gancho (`reading-selftest.ts` ou novo
   `search-selftest.ts` chamado por `selftest.ts`), após `ensureReadingDb()`,
   chamar `search(dbPath, "God", "kjv")`, **localizar o hit de João 3:16**
   (`reference.book===43 && chapter===3 && verses.tag==='Single' &&
   inner.verse===16`) e emitir um marcador **composto do retorno real** (texto
   limpo de `hit.text`, **sem** os marcadores de controle; ref composta de
   `listBooks().nameEn` + cap + verso), ex.:
   `TLA_SEARCH query="God" hits=<N> first_ref="John 3:16" first_text="For God so loved..."`
   (`hits` = `result.length`; `first_ref`/`first_text` do hit de João 3:16
   encontrado no retorno — não hardcoded). Manter `TLA_READ`/`TLA_PARALLEL`/parse
   **intactos** (sem regressão). Criar o par web (SKIP) se for arquivo novo.
8. **Provar e capturar:** estender `scripts/run-ios-selftest.sh` p/ **assertar
   também** `TLA_SEARCH` (hits≥1, ref "John 3:16", substring do texto "For God so
   loved"), além de PT/EN/`TLA_READ`/`TLA_PARALLEL`. Rodar e capturar o log.
9. **Qualidade:** `cd app && npx tsc --noEmit` verde; `expo export --platform web`
   sai 0 (a tela de busca não pode quebrar o build web — o glue web é stub que só
   lança em **runtime**). Working tree limpo (gerados/bundled ignorados). ADR
   **só** se houver decisão nova (não há prevista; reusa ADR-0014/0015).

## Restrições

- **NÃO modificar/forkar `the-light`** nem a fronteira `core/src/lib.rs`. A busca
  é **delegada** a `the_light_core::search` via a função `search` da F1.5. Se a UI
  exigisse mudar o core/fronteira → **`blocked`/HALT** (PR + ADR é ação humana),
  com o erro EXATO.
- **NÃO reimplementar busca/FTS/SQL no app** — a UI **só** chama a fronteira
  `search` e embrulha o retorno. Proibido em TS: `verses_fts`, `MATCH`, `bm25(`,
  `highlight(`, montar expressão FTS, ranquear/snippetar à mão. (Popular
  `verses_fts` no **gerador de dados** do subset é fixture/pipeline — como o
  importador do core e o fixture da F1.5 —, **não** busca; permitido e necessário.)
- **Anti-alucinação:** todo texto de hit vem **do store local** (verbatim,
  domínio público KJV/Almeida); **nunca** gerar/hardcodear texto bíblico em
  produção. No self-test, `first_text` provém do **retorno de `search`**; a
  asserção do script usa um substring esperado, não a fonte.
- **HL markers não vazam como texto cru:** `HL_START`(`\u{2}`)/`HL_END`(`\u{3}`)
  de `highlighted` viram **estilo**; nunca são exibidos nem entram no marcador do
  self-test (use `hit.text` limpo p/ o marcador).
- **Offline-first:** **zero rede em runtime** (busca é I/O local no device, via
  FTS5 do store); rede só em dev/build (gen-*-db, Metro, Gradle, pods, npm).
- **Só nativo:** a paridade web da busca é a **F1.14** (pós-gate F1.12). Manter o
  glue web de busca como **stub** (lança em runtime) e o **build web verde** (sem
  quebra de import/build). Não construir o caminho web aqui.
- **Gerados/bundled ignorados:** `bible.sqlite`, `reading-sample.sqlite`, asset
  copiado em runtime, `app/ios`, `app/android`, `app/web/native-generated`,
  jniLibs/`.so` permanecem no `.gitignore`; working tree limpo.
- **Segredos nunca em git/log.**
- **BLOQUEIO (`blocked` + HALT) com erro EXATO** se: a busca nativa exigir mudar
  o core/fronteira; OU o subset não puder ter `verses_fts` povoado sem alterar o
  schema do core; OU a UI de busca no Expo nativo exigir hack proibido. Reportar a
  saída exata e propor a decisão/ADR; não improvisar.
