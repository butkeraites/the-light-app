## Objetivo

**Gate estratégico (decisão humana/arquitetural).** Por ser `gate: true`, o loop
**NÃO** executa esta tarefa autonomamente: o Driver escreve `loop/HALT` e o
**humano (Renan)/Guia** decide antes de liberar a paridade web (F1.13–F1.16).

Esta tarefa **não tem código de produto a implementar**. É um **BRIEF DE DECISÃO**:
definir **como o WEB entrega leitura + busca (FTS5) + referências cruzadas + notas/
marcações sobre o corpus bíblico**, mantendo **offline-first**, **BYOK**,
**anti-alucinação** (texto sempre do store local) e **`the-light` intacto**
(mudança no core só via PR + ADR). O desfecho do ciclo é **HALT p/ sign-off
humano**; com a decisão registrada (novo ADR), o Driver re-escopa/semeia F1.13–F1.16.

---

## 1. Contexto (o que está provado e o que falta)

**Já provado no web (Fase 0):**
- **F0.6b** (`1cdde6c`, ADR-0007): `parse_reference` roda no browser via **wasm**
  (`ubrn build web`, build SYNC, **sem `SharedArrayBuffer`/COOP-COEP**). A resolução
  de referência (domínio) é uma **fonte da verdade no Rust**, também no web.
- **F0.10** (`d6e968d`, ADR-0011/0012): store web de **1 passagem** — `getPassage`
  em TS lê João 3:16 do **`sample.sqlite`** (~128 KB) via **`wa-sqlite@1.0.0`**
  (build SYNC) + **OPFS**. Sub-decisão real do ADR-0012: a leitura usa um **VFS de
  MEMÓRIA do `wa-sqlite` HIDRATADO com os bytes do OPFS** (o arquivo inteiro é
  carregado no heap JS), porque o **VFS OPFS "ao vivo"** exige
  `FileSystemSyncAccessHandle` (`createSyncAccessHandle`), que só existe em **Web
  Worker** — deliberadamente adiado. O texto vem **verbatim do store local**.

**O que falta (depende DESTA decisão):** a paridade web da Fase 1 está **toda em
stub** hoje. Em `app/web/reading.web.ts`, as funções `listBooks`,
`listTranslations`, `getChapter`, `chapterCount`, `search`, `crossRefs` e o CRUD de
`notes`/`highlights` **lançam erro explícito** ("= F1.13/F1.14/F1.15/F1.16"). Logo:
- **F1.13** (leitura/navegação + versões no web) · **F1.14** (busca FTS5 no web) ·
  **F1.15** (xref + atribuição CC-BY no web) · **F1.16** (notas/highlights no web)
  **só podem ser semeadas após esta decisão** — todas declaram `depends_on: [F1.12]`.

**Números reais do corpus (medidos agora, `assets/data/`):**
- `bible.sqlite` = **61.833.216 bytes (~59 MB)** — `verses` **62.203** (KJV 31.102 +
  ALM1911 31.101), `verses_fts` **62.203** (FTS5), `cross_references` **344.799**.
- `reading-sample.sqlite` = **4.567.040 bytes (~4,4 MB)** — subset de **3 livros**
  (Gênesis/Salmos/João, KJV+ALM1911): `verses` **9.746**, `verses_fts` **9.746**,
  `cross_references` **22.413**. **É o que o app NATIVO empacota hoje** (ADR-0014;
  o `bible.sqlite` completo foi **adiado** no nativo como "download/expansão sob
  demanda — otimização posterior").
- `sample.sqlite` = **131.072 bytes (~128 KB)**, versionado (1 passagem, F0.10).

**Tensão central a explicitar ao humano:** o **NATIVO ainda NÃO entrega o corpus
completo** — ele empacota o subset de 4,4 MB (ADR-0014). "Corpus completo no web"
(~59 MB) **excederia** o escopo atual do nativo. E o caminho de leitura provado na
F0.10 (**MemoryVFS, arquivo inteiro no heap**) escala bem para 4,4 MB, **mas não
para 59 MB** — o corpus completo exigiria o **VFS OPFS "ao vivo" em Web Worker**
(adiado no ADR-0012). Portanto a decisão tem **dois eixos acoplados**:
**(i) ESCOPO** (subset de paridade com o nativo × corpus completo) e
**(ii) MECANISMO** (MemoryVFS na main thread × OPFS-live-VFS em Worker), além de
**(iii) onde mora a lógica de busca/xref** (espelhada em TS × no core via PR).

## 2. Opções (prós/contras honestos)

### Opção A — Estender a **Opção A** (ADR-0011): `wa-sqlite` + OPFS, espelhar os SELECTs em TS
Reaproveitar a máquina da F0.10 e adicionar, no glue web, as queries equivalentes a
`get_chapter`/`search`/`for_verse` (FTS5 `MATCH`/`bm25`/`highlight`; xref por votos).
- **Sub-variante A1 (subset, paridade com o nativo):** empacotar o **mesmo
  `reading-sample.sqlite` (4,4 MB)** do nativo; ler via **MemoryVFS já provado**
  (sem Worker). **Prós:** menor passo técnico (a F0.10 já faz isso, só faltam mais
  queries); 4,4 MB de download é aceitável; **paridade real com o nativo hoje**;
  zero servidor; zero PR ao core; FTS5/xref nativos do SQLite (semântica idêntica).
  **Contras:** **busca/leitura web limitadas a 3 livros** (não é a Bíblia inteira);
  **duplica em TS** as queries de busca/xref (risco de **divergir do core** — mitigar
  por testes de paridade); depende de **FTS5 estar no build do `wa-sqlite@1.0.0`**
  (a verificar).
- **Sub-variante A2 (corpus completo, ~59 MB):** empacotar/baixar o `bible.sqlite`
  inteiro e persistir em OPFS. **Prós:** Bíblia completa + busca/xref globais;
  reaproveita 100% do schema/SQL do core. **Contras:** **~59 MB de 1ª carga**
  (cota/persistência OPFS, tempo de primeira carga, indicador de progresso);
  **MemoryVFS não serve** (59 MB no heap por sessão) → **exige OPFS-live-VFS em Web
  Worker** (`createSyncAccessHandle`) — passo arquitetural **novo** além da F0.10;
  perf de **FTS5 em wasm** sobre 62k versos a validar; mesma duplicação TS↔core.

### Opção B — Abstrair o **store no `the-light-core`** (Opção B da ADR-0011), via **PR + ADR**
Store injetável que rode em wasm (passage + search + xref + userdata em **Rust**),
eliminando a duplicação de SQL/FTS em TS.
- **Prós:** **uma fonte da verdade** (busca/xref/FTS no Rust, zero divergência);
  future-proof; o web consome o **mesmo** código de domínio dos nativos.
- **Contras:** **refator substancial do core** + **bridge Rust-síncrono ↔
  JS-assíncrono** (rusqlite é sync; `wa-sqlite` é async) — difícil; `rusqlite` **não
  compila p/ wasm** (motivo do ADR-0005) → exigiria um **trait de store** sobre
  `wa-sqlite` chamável do Rust, ou `sqlite-wasm-rs` (peso/bundle). **Exige PR ao
  `the-light`** (ação humana, ADR-0002) → o loop **não** pode fazer sozinho. É o
  ideal da VISION, mas o de **maior custo/risco** agora.

### Opção C — Entrega **segmentada/sob demanda** (chunking por livro/seção)
Empacotar um subset pequeno + **lazy-load** de livros/índices sob demanda (e/ou FTS5
num índice paginado).
- **Prós:** **menor 1ª carga**; alinhado ao precedente do nativo (subset);
  caminho p/ "a Bíblia inteira sem baixar 59 MB de uma vez".
- **Contras:** **complexidade de chunking/coerência** (montar capítulos/índices de
  pedaços); **busca pode ficar parcial** — um FTS5 **global** é praticamente o
  grosso do tamanho de qualquer jeito; `cross_references` global (344.799) é difícil
  de paginar com coerência. Mais infra para manter sincronizada com o pipeline.

### Opção D — Otimização de **entrega/transferência** do `.sqlite` (modificador, não substituto)
Servir o asset **comprimido** (gzip/br do `.sqlite`, ou delta) + indicador de
progresso, descomprimindo no cliente antes de persistir em OPFS.
- **Prós:** **corta o download** (SQLite comprime bem) sem mudar o data-model;
  combina com A2/C. **Contras:** descompressão no cliente (CPU/memória de pico);
  **não resolve** o footprint de runtime (MemoryVFS vs Worker) nem a cota OPFS;
  é **complemento** de A2/C, não decisão autônoma.

> **Rejeitada (registrar):** "pré-indexado/derivado **sem** SQLite" (índice invertido
> próprio como assets estáticos). Já recusada na ADR-0011 (Opção C): **diverge do
> data-model**, não escala p/ a Bíblia completa + FTS, e adiciona um pipeline
> paralelo a manter. O texto continuaria do store local (anti-alucinação ok), mas o
> custo de divergência não compensa.

**Para cada opção — resumo do impacto:**

| Eixo | A1 (subset) | A2 (completo) | B (core/PR) | C (chunking) | D (compress.) |
|---|---|---|---|---|---|
| **Download / OPFS** | ~4,4 MB, trivial | ~59 MB + cota/persistência | depende do empacote | menor inicial, incremental | reduz download |
| **Perf FTS5 wasm** | leve (9,7k) | a validar (62k) | a validar (62k) | parcial/paginada | igual ao subjacente |
| **Reuso do SQL do core (anti-divergência)** | parcial (espelho TS) | parcial (espelho TS) | **total** (Rust) | parcial | n/a |
| **Mecanismo runtime** | MemoryVFS (provado) | **Worker+OPFS-live** (novo) | bridge sync↔async | Worker provável | — |
| **Complexidade F1.13–F1.16** | **baixa** | média-alta | **alta** (+PR core) | alta | +média |
| **Toca `the-light`?** | não | não | **sim (PR+ADR)** | não | não |

## 3. Recomendação do Planner (uma, fundamentada)

**Adotar a Opção A (estender a Opção A da ADR-0011): `wa-sqlite` + OPFS, espelhando
em TS os SELECTs de `get_chapter`/`search`(FTS5)/`for_verse`(xref)** — pela
coerência com o que já está provado (F0.10), por **não exigir PR ao core** (o loop
avança sozinho) e por reaproveitar a **semântica nativa do SQLite/FTS5** (sem
reinventar busca/ranqueamento).

**Sub-recomendação de ESCOPO/MECANISMO (a confirmar pelo humano): começar pela
sub-variante A1 — subset de leitura (4,4 MB, o MESMO `reading-sample.sqlite` do
nativo, MemoryVFS já provado) — entregando F1.13–F1.16 em PARIDADE REAL com o
nativo de hoje**, e tratar o **corpus completo (~59 MB) + OPFS-live-VFS em Worker**
como **decisão transversal separada** (vale p/ nativo E web — o nativo também adiou
o completo, ADR-0014). Assim a paridade web fecha o Marco 1 com baixo risco, sem
prometer no web algo que o nativo ainda não entrega.

**Riscos residuais e mitigação:**
- **FTS5 ausente no build do `wa-sqlite@1.0.0`** → mitigar com **spike de
  verificação** (criar uma `verses_fts` e rodar `MATCH` no `wa-sqlite`); se ausente,
  o sign-off escolhe build/flag (sem COOP/COEP) **ou** fallback de busca. Não cravar
  A sem essa evidência.
- **Divergência espelho-TS ↔ core** → **testes de paridade** (mesma query/fixture,
  comparar contra a fronteira nativa) na F1.14/F1.15; documentar no ADR.
- **Escalar para o corpus completo depois** → o OPFS-live-VFS em Worker
  (`createSyncAccessHandle`, sem `SharedArrayBuffer`) é o caminho previsto; manter a
  porta aberta (a Opção B continua sendo a evolução "tudo no core" se a duplicação
  TS pesar).
- **OPFS indisponível** (browsers antigos/aba privada) → definir fallback
  (ex.: VFS de memória efêmero por sessão, ou mensagem clara).

## 4. O que o sign-off humano decide

**Pergunta objetiva:** qual **direção** para o store web — **A** (espelho TS sobre
`wa-sqlite`/OPFS, recomendada), **B** (store no core via PR+ADR), **C** (chunking)
ou **D** (só como otimização de entrega sobre A2/C)?

**Sub-decisões (responder junto):**
1. **ESCOPO:** paridade com o nativo = **subset 4,4 MB** (A1) **ou** **corpus
   completo ~59 MB** (A2)? (Se completo: aceita o passo extra do OPFS-live-VFS em
   Worker e um **limite de download aceitável** + **indicador de progresso**?)
2. **FTS5:** exigir o **spike de verificação** de FTS5 no `wa-sqlite@1.0.0` **antes**
   de cravar a direção de busca (F1.14)? Se ausente, qual fallback (build/flag sem
   COOP-COEP × busca alternativa)?
3. **Notas/highlights no web (F1.16):** o `userdata` do core é **file-based JSON**
   (ADR-0017), não SQLite. No web (sem filesystem) → **OPFS/IndexedDB** mantendo o
   **formato exportável compatível com o nativo**. Confirmar este caminho (é
   ortogonal à decisão do store SQLite).
4. **Fallback se OPFS indisponível** (browser antigo/privado): qual comportamento?
5. **Se escolher B:** autorizar abrir a **spec de PR** ao `the-light` em
   `loop/proposals/` (núcleo só via PR humano — o loop **não** toca `the-light`).

## 5. Consequência

- **Enquanto não houver decisão, o loop fica em HALT** (o Driver escreve
  `loop/HALT` com motivo "aguardando sign-off do gate F1.12").
- Decidido, registra-se um **novo ADR** (próximo livre = **ADR-0018**; o maior atual
  é ADR-0017) com a direção + sub-decisões 1–5 e a evidência (incl. resultado do
  spike de FTS5, se feito).
- O Driver então **re-escopa/semeia F1.13–F1.16** conforme a opção escolhida
  (todas `depends_on: [F1.12]`). Se a opção for **B**, abre-se a spec de PR em
  `loop/proposals/` **antes** de F1.13 (núcleo só via PR humano).
- F1.13–F1.16 alimentam o **Marco 1 (F1.17, `gate: true`)** — leitura offline
  completa nos 3 alvos.

## Como liberar

Humano/Guia revisa o estado (`STATUS.md`, `DECISIONS.md` ADR-0011/0012/0013/0014,
`app/web/sqlite-opfs.web.ts`, `app/web/reading.web.ts`, contagens de
`assets/data/`), responde §4 (direção + sub-decisões), registra o ADR-0018 e
**remove/edita `loop/HALT`** com o veredito. O loop retoma no próximo ciclo
(Planner semeia F1.13 conforme a decisão).

## Restrições (não negociáveis — valem em qualquer opção)

- **Offline-first:** nada essencial exige rede em runtime; a 1ª carga do banco é
  fetch da **própria origem** (asset local) e persiste em OPFS; rede só em dev/build.
- **Anti-alucinação:** **todo** texto de versículo vem do **store local** (SQLite/
  OPFS), verbatim de **domínio público** (KJV/Almeida 1911); o LLM nunca produz texto.
- **Licenciamento:** só domínio público (texto) + **CC-BY** (xref OpenBible.info,
  ADR-0016, atribuição visível); nada protegido.
- **`the-light` intacto:** consumo pinado (`rev 8f66004`); mudança no core **só via
  PR + ADR** (ação humana). Se a decisão exigir tocar o core (Opção B), é spec de PR.
- **BYOK / segredos:** nenhuma chave/segredo (não se aplica a esta fase, mas vale).
- **Sem código de produto nesta tarefa.** Desfecho = **HALT p/ sign-off humano**.
