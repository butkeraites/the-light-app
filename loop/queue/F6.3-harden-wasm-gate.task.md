## Objetivo

`app/web/wasm.web.ts:36-39` CAPTURA a falha de `uniffiInitAsync()` e a DESCARTA (comentário diz que
"reaparece ao chamar listBooks()"), mas `ready` fica `false` para sempre → `WasmGate` nunca resolve
→ **spinner infinito, sem erro** se o wasm da fronteira falhar ao inicializar. Foi exatamente esse
padrão de "erro engolido" que deixou a leitura web quebrada por 3 ciclos parecer um "carregando"
inofensivo. A F6.2 já PROVOU que o wasm da fronteira carrega OK em dev E dist hoje (abrir-capítulo
verde nos dois, o que exige `listBooks()` do wasm). Então aqui NÃO há bug de init a corrigir — só
**endurecer** para que uma falha futura seja VISÍVEL, e travar isso com o smoke.

## Escopo (produto + asserção no harness)

1. **`app/web/wasm.web.ts`:** `useWasmReady` passa a expor um ESTADO DE ERRO (não só `boolean`) —
   ex.: retornar `{ ready, error, retry }` (ou um enum) em vez de engolir no `.catch`. `ensureWasmReady`
   deve permitir **retry** (resetar a `initPromise` memoizada quando o retry é chamado, para uma nova
   tentativa real). Manter idempotência no caminho feliz.
2. **`app/components/WasmGate.tsx`:** quando o init ERROU, renderizar uma **mensagem de erro visível +
   botão "Tentar de novo"** (via `t()`, tokens de tema, a11y role/label) em vez do `ActivityIndicator`
   infinito. Enquanto carrega normalmente, segue o spinner. Sucesso → renderiza os filhos como hoje.
3. **i18n:** chaves novas (ex.: `wasm.loadError`, `wasm.retry`) em pt/en; adicionar ao guard de
   cobertura se aplicável.
4. **Asserção no smoke (`app/web/__browser__/`):** usar o switch `SMOKE_WASM_WRONG_MIME=1` do
   `static-server.mjs` (alvo dist) para servir o `.wasm` da fronteira com MIME/URL inválido de
   propósito e assert que a **UI de erro aparece** (texto de erro + botão retry) e que NÃO há spinner
   infinito. Novo marcador `TLA_WEB_wasm-error-ui ok`. Deve rodar só quando o flag está setado (não
   quebrar o fluxo normal). Bônus: clicar em "Tentar de novo" com o MIME corrigido → recupera.
