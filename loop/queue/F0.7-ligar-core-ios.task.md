## Objetivo

Provar a ponte Rust→Expo no alvo **iOS nativo**: digitar `"Jo 3.16"` (PT) ou
`"John 3:16"` (EN) no app rodando no **simulador iOS** resolve a referência
**pelo Rust nativo** (`the-light-core::reference` via UniFFI → **Turbo Module
JSI** gerado pelo `ubrn`), **não** pelo caminho web/wasm, **não** por wasm,
**não** por parsing em TS. Ambos os idiomas devem resolver para
`book=43, chapter=3, verses=Single{verse:16}`. Exige **run real no simulador**
(não apenas build), com **prova headless/determinística** (sem clique manual na
UI).

## Contexto / arquivos

**Ambiente PRONTO** (tabela de ambiente do `DECISIONS.md`, atualização
2026-06-30 — confirmado de novo nesta tarefa):
- Xcode completo **26.6** (Build 17F113); `xcode-select -p` →
  `/Applications/Xcode.app/Contents/Developer`.
- Runtime simulador **iOS 26.5**; devices iPhone 17 disponíveis
  (`xcrun simctl list devices available | grep -i 'iPhone 17'`).
- Targets Rust `aarch64-apple-ios` e `aarch64-apple-ios-sim` instalados.
- `uniffi-bindgen-react-native` (CLI `ubrn`) **`0.31.0-3`** (devDependency npm,
  pinada) já presente em `node_modules/.bin/ubrn`. A fronteira
  `the-light-app-core` usa `uniffi = "=0.31.0"` (ADR-0006).

**O caminho nativo iOS do `ubrn` (a usar — já descoberto pelo Planner, confirmar
empiricamente):**
- Subcomando: **`ubrn build ios`** (≠ `generate jsi` host-only da F0.4; ≠
  `build web` da F0.6b). Opções relevantes (`ubrn build ios --help`):
  `--config <ubrn.config.yaml>`, `--sim-only` (só simulador — suficiente aqui),
  `-g/--and-generate` (gera os bindings TS + a **glue do Turbo Module** junto do
  xcframework), `--release`, `--targets`. O fluxo padrão: `cargo build` para os
  targets iOS → `lipo`/`xcframework` (a menos de `--no-xcodebuild`) → com `-g`,
  gera a TS + C++/JSI do Turbo Module.
- Schema de config iOS (`IOsConfig`, camelCase) em
  `node_modules/uniffi-bindgen-react-native/crates/ubrn_cli/src/jsi/ios/config.rs`:
  bloco top-level **`ios:`** com `directory` (default `ios`), `frameworkName`
  (default derivado do `package.json`), `targets` (default
  `[aarch64-apple-ios, aarch64-apple-ios-sim]`), `codegenOutputDir`,
  `cargoExtras`, `xcodebuildExtras`. O `ubrn.config.yaml` (raiz) já tem
  `rust:`/`bindings:`/`web:`; esta tarefa adiciona `ios:` (e, se necessário,
  `turboModule:`) — **reutilizando** o mesmo arquivo (fonte da verdade de
  caminhos, ADR-0004/0007).
- Guia oficial do fluxo RN/iOS:
  `node_modules/uniffi-bindgen-react-native/docs/src/guides/rn/getting-started.md`
  (padrão `"ubrn:ios": "ubrn build ios --and-generate && (cd ios && pod install)"`).
  ATENÇÃO: o guia assume um layout `create-react-native-library` (cpp/, ios/,
  spec `Native*.ts`, codegen no `package.json`). **Nosso app é Expo** — a
  integração real difere (ver "Passos").

**Feature-matrix do core no nativo (F0.6a / ADR-0005, ADR-0007):** a fronteira
consome `the-light-core` com `default-features = false` + um bloco
`[target.'cfg(not(target_arch="wasm32"))']` que liga `embedded`. Logo, os
targets iOS (não-wasm32) compilam o core **completo** (`embedded` → rusqlite
SQLite-C + reqwest). O smoke test Android (2026-06-30) já provou o toolchain
nativo do core; iOS é análogo (mesma família de targets reais). Aqui só
`parse_reference` (puro `reference`/`model`) é exercitado, mas o build arrasta
`embedded` — risco baixo, observar.

**Integração no Expo (SDK 56, New Architecture já `newArchEnabled: true` em
`app/app.json`):**
- O app em `app/` tem `package.json` próprio (RN 0.85.3, expo-router) e **não
  tem** projeto `ios/` nativo versionado. Gerá-lo exige
  **`npx expo prebuild`** (de `app/`), depois **CocoaPods** (`pod install`) e
  então `xcodebuild`/`expo run:ios` para build+install+launch no simulador.
- `npx expo run:ios` encadeia prebuild (se preciso) + pod install + xcodebuild +
  launch no simulador; é o caminho mais direto para **run real**.
- O Turbo Module gerado pelo `ubrn` precisa ser **registrado/autolinkado** no
  projeto iOS gerado (autolinking do Expo/RN, Podspec, e codegen da New Arch).
  Esta é a parte de **maior risco de imaturidade** (ver Restrições / blocked).

**A tela e o glue de plataforma (Metro resolve por extensão):**
- `app/app/index.tsx` importa `parseReference` de `../web/reference`. No **web**,
  o Metro resolve `app/web/reference.web.ts` (glue wasm da F0.6b — **manter
  intacto**, sem regressão). No **nativo**, resolve `app/web/reference.ts`, hoje
  um **stub que lança** (`'parseReference (wasm) é WEB-only … chega em F0.7/F0.8'`).
- F0.7 deve **ligar o caminho nativo**: reescrever `app/web/reference.ts` (ou
  criar `app/web/reference.native.ts`) para **delegar ao binding nativo gerado**
  (módulo TS do Turbo Module do `ubrn`, que chama o JSI → Rust). A guarda de
  escopo "sem core no nativo" da F0.5 **não vale aqui** — é exatamente o objetivo.

**Gerado (IGNORAR no git) vs. versionado:**
- IGNORAR (gerados): o projeto **`app/ios/`** do prebuild, **`Pods/`**,
  `*.xcworkspace`/`build/`, o **xcframework** gerado pelo `ubrn`, qualquer
  `app/web/generated/` ou diretório de bindings gerados, `node_modules`,
  `target/`. Estender o `.gitignore` para cobrir `app/ios/`, `Pods/`, xcframework
  e a saída de codegen iOS — nada gerado pode sujar o working tree.
- VERSIONAR: o bloco `ios:` (e `turboModule:` se preciso) em `ubrn.config.yaml`;
  os scripts novos (`scripts/gen-bindings-ios.sh`, `scripts/run-ios-selftest.sh`
  ou nomes equivalentes); o config plugin / módulo Expo necessário p/ autolinkar
  o Turbo Module (se hand-written); o glue nativo (`app/web/reference.ts` ou
  `.native.ts`); o gancho de self-test na tela/app; ajustes em `app/package.json`
  (scripts/deps); o `.gitignore`; e o **ADR-0008** (caminho iOS do `ubrn`).

## Passos sugeridos

1. **Confirmar ambiente** (não destrutivo): `xcode-select -p`,
   `xcrun simctl list runtimes | grep -i ios`,
   `rustup target list --installed | grep apple-ios`.
2. **Descobrir/fixar o fluxo iOS do `ubrn`** empiricamente:
   `node_modules/.bin/ubrn build ios --help`,
   `node_modules/.bin/ubrn generate jsi turbo-module --help`, ler
   `crates/ubrn_cli/src/jsi/ios/{config,commands,mod}.rs` e
   `docs/src/guides/rn/getting-started.md`. Decidir como o Turbo Module gerado
   entra num app **Expo** (config plugin vs. módulo local vs. patch do projeto
   prebuilt) e registrar a escolha no **ADR-0008**.
3. **Configurar `ubrn.config.yaml`**: adicionar o bloco `ios:` (e
   `turboModule:`/`bindings:` conforme o schema), apontando `codegenOutputDir`/
   `directory` para dentro de `app/` de forma que o autolinking do Expo encontre
   o módulo. Manter `rust:`/`bindings:`/`web:` existentes intactos.
4. **Gerar/buildar o módulo nativo iOS** via `ubrn build ios --sim-only
   --and-generate --config ubrn.config.yaml` (encapsular em
   `scripts/gen-bindings-ios.sh`, reproduzível, saída 0). Produz o xcframework +
   a glue TS/C++/JSI do Turbo Module (gerados, ignorados).
5. **Prebuild + Pods**: de `app/`, `npx expo prebuild -p ios` (gera `app/ios/`)
   e `pod install` (via `expo run:ios` que encadeia, ou manual). Garantir que o
   Podspec/autolink do Turbo Module do `ubrn` seja incluído.
6. **Ligar a tela ao binding nativo**: implementar `app/web/reference.ts`
   (variante nativa) para delegar `parseReference` ao módulo TS do Turbo Module
   gerado (sem parsing em TS, sem eco). Não tocar `app/web/reference.web.ts`.
7. **Self-test determinístico no launch** (prova headless): adicionar um gancho
   que, sob um sinal de ambiente/arg (ex.: `EXPO_PUBLIC_TLA_SELFTEST=1`, lido na
   montagem do `HomeScreen` ou num módulo de entrada), chame
   `parseReference("Jo 3.16")` e `parseReference("John 3:16")` e emita ao
   console **marcadores estáveis**, ex.:
   `TLA_SELFTEST PT book=43 chapter=3 verse=16` e
   `TLA_SELFTEST EN book=43 chapter=3 verse=16` (logados via `console.log`, que
   no iOS cai no log do simulador). O gancho NÃO altera o comportamento normal
   da UI (só dispara sob o env de teste).
8. **Rodar no simulador + capturar log** (`scripts/run-ios-selftest.sh`,
   determinístico): bootar um device iPhone 17 (`xcrun simctl boot`), construir+
   instalar+lançar (via `expo run:ios` com o env de self-test, ou
   build+`simctl install`+`simctl launch --console-pty`/
   `simctl spawn booted log stream`), capturar o log e **assertar** a presença
   de AMBOS os marcadores PT e EN (exit 0 só se os dois aparecerem). Finalizar/
   limpar o simulador no fim. Se o `expo run:ios` exigir Metro (dev), preferir um
   **build release** (`--configuration Release` / `--no-bundler`) para
   self-test offline, ou subir o Metro em background dentro do script.
   **Alternativa robusta aceitável:** um **XCTest** nativo no `app/ios/` gerado
   que chama a função do Turbo Module e assere 43/3/16, rodado por
   `xcodebuild test -destination 'platform=iOS Simulator,name=iPhone 17'`.
   Escolher a abordagem mais confiável e documentá-la no ADR-0008.
9. **Higiene**: estender `.gitignore` (app/ios, Pods, xcframework, codegen iOS),
   rodar fmt/clippy se algo Rust mudou (não deve — fronteira intacta), `tsc
   --noEmit` no `app/`, garantir working tree limpo. Commit com prefixo `F0.7`.

## Restrições

- **NÃO modificar nem forkar `the-light`** (ADR-0002) nem alterar a **forma da
  fronteira** `core/` (modo library / `setup_scaffolding!` / `CoreError` /
  `parse_reference`). Se o caminho iOS exigir mudar o core ou a fronteira →
  **HALT/blocked**, não improviso.
- **NÃO fingir, stubar nem reimplementar o parsing**: `parse_reference` resolve
  no **Rust nativo** via Turbo Module. Eco/lógica TS/JSON hardcoded = reprovação.
  A prova deve evidenciar a origem Rust (Turbo Module/JSI), não um atalho.
- **Anti-alucinação / uma fonte da verdade**: a referência vem do `the-light-core`
  via UniFFI; nada de parser paralelo.
- **Offline-first é regra de runtime**: rede só em **dev/build** (CocoaPods,
  cargo, npm, `wasm-bindgen`/toolchains). Nenhuma rede no runtime do produto;
  **nenhum segredo** em git/log (Keychain/`.env` jamais commitados).
- **Gerados IGNORADOS**: `app/ios/` (prebuild), `Pods/`, xcframework, codegen
  iOS, `node_modules`, `target/`. Working tree limpo é critério rígido.
- **Sem regressão**: não tocar o caminho web da F0.6b (`reference.web.ts`,
  `app/web/generated/`, `ubrn.config.yaml:web`).
- **Qualidade por tarefa**: se algo Rust mudar (não deveria), `cargo fmt
  --check` + `cargo clippy -D warnings` + `cargo test` verdes; TS: `tsc
  --noEmit` no `app/`.
- **BLOQUEIO LEGÍTIMO (`blocked`) — decisão estratégica:** se o **caminho iOS do
  `ubrn 0.31.0-3`** for imaturo/inoperante a ponto de **só** funcionar com um
  **hack proibido** (forkar/alterar `the-light` ou a fronteira; reimplementar/
  stubar o parsing; integrar o Turbo Module ao Expo de forma que quebre o
  autolinking/New Arch sem caminho sancionado; build iOS do core `embedded`
  falhar de modo irreparável sem tocar o core), **reportar `blocked` com o erro
  EXATO** (saída literal do comando que falhou) e a causa-raiz. Isso é o **gate
  estratégico** sobre o caminho iOS do `ubrn` (análogo ao ADR-0007 do web) — o
  Driver escreve `loop/HALT` para decisão humana. **Não** é bloqueio do core (já
  resolvido na F0.6a) nem do toolchain (Xcode/sim já prontos). Se a decisão for
  tomada e o caminho viabilizado, registrar em **ADR-0008**.
